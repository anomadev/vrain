La estructura de directorios predeterminada de una aplicación Laravel proporciona un punto de partida claro y coherente, organizando el código en carpetas especializadas que cubren desde la lógica de la aplicación hasta la configuración, vistas, rutas, prueba y dependencias externas. El **directorio raíz** incluye subcarpetas como `app`, `bootstrap`, `config`, `database`, `public`, `resources`, `routes`, `storage`, `tests` y `vendor`, cada una con responsabilidades bien definidas que facilitan el mantenimiento y la escalabilidad del proyecto. Dentro de `app`, Laravel agrupa el **código principal** de la aplicación en subdirectorios como `Http`, `Models` y `Providers`, y expone la flexibilidad de `PSR-4` para `autoloading`, permitiendo ubicar clases donde mejor convenga mientras `Composer` las descubra automáticamente.
### Introducción 
Laravel impone mínimas restricciones en la ubicación de las clases, confiando en `Composer` y el estándar `PSR-4` para cargarlas automáticamente. Esto permite reorganizar la estructura de carpetas si así lo requiere el proyecto, sin perder compatibilidad con el `autoloading`.
### Estructura del Directorio Raíz
- `app`: Contiene casi todo el **código de la aplicación**: controladores, modelos, proveedores de servicio y más. Es el corazón de tu lógica de negocio.
- `bootstrap`: Incluye `app.php` para inicializar el `framework` y la subcarpeta `cache` con archivos generados (rutas y servicios) para mejorar el rendimiento.
- `config`: Guarda todos los **archivos de configuración** (`app.php`, `database.php`, etc.). Es recomendable repasarlos para familiarizarse con las opciones disponibles.
- `database`: Almacena **migraciones, factories y seeds**. También puede contener una base SQLite para entornos simples o pruebas.
- `public`: Punto de entrada `HTTP` - `index.php`. Aquí residen los activos públicos (CSS, JavaScript, imágenes) accesibles por el navegador.
- `resources`: Incluye las vistas (`.blade.php`). y los activos sin compilar (CSS, JS). Aquí edificas la capa de presentación de la aplicación.
- `routes`: Define las rutas de la aplicación. Por defecto
	- `web.php`: rutas con estado (sesiones, CSRF)
	- `console.php`: comandos `Artisan` y programación de tareas (`schedule`)
	- Opcionalmente `api.php` (*stateless*) y `channels.php` (*broadcasting*) pueden añadirse vía `Artisan`
- `storage`: Contiene **logs, plantillas compiladas, sesiones por archivo, caches y archivos generados**. Subdividido en `app`, `framework` y `logs`. Para exponer archivos públicos (e. g., avatares), se crea un enlace simbólico `public/storage` con `php artisan storage:link`.
- `tests`: Incluye pruebas unitarias y de características con `Pest` o `PHPUnit`. Cada clase termina en `Test` y puedes ejecutarlas con `/vendor/bin/pest`, `vendor/bin/phpunit` o `php artisan test`.
- `vendor`: Aquí residen las **dependencias de Composer**. No se debe versionar; gestionar paquetes externos y autoloading.
### El directorio `app` (subdirectorios)
Por convención. `app` incluye inicialmente `Http`, `Models` y `Providers`, pero se expande al usar `Artisan` (`make:`) para generar nuevas clases. Entre sus subcarpetas: 
- `Broadcasting`: canales de difusión de eventos (`make:channel`)
- `Console`: comandos personalizados (`make:command`)
- `Events`: clase de eventos (`make:event`) para desacoplar lógica
- `Exceptions`: manejo de excepciones personalizadas (`make:exception`)
- `Http`: controladores, middleware y peticiones; punto principal de entrada HTTP
- `Jobs`: trabajos en cola o sincrónicos (`make:job`) siguiendo el patrón `Command`
- `Listeners`: clases que reaccionan a eventos (`make:listener`)
- `Mail`: representaciones de correos (`make:mail`) encapsulando la lógica de envío
- `Models`: clases `Eloquent` que mapean tablas de base de datos; cada tabla suele corresponder a un modelo
- `Notifications`: notificaciones transaccionales (`make:notifications`) con múltiples canales
- `Policies`: políticas de autorización (`make:policy`) para controlar permisos
- `Providers`: `services providers` que registran servicios en el `container`; incluye `AppServiceProvider` por defecto
- `Rules`: reglas de validación personalizadas (`make:rule`)
## Notas
---
- Mantener actualizado el `README.md` o un diagrama de la estructura ayuda a la incorporación de nuevos desarrolladores.
- Aprovechar `PSR-4` para reorganizar clases sin romper el `autoloading`.
- Crear vistas parciales y componentes en `resources/views` para mejorar la mantenibilidad.
