> [!info] Learn the Basics
> Las interfaces de programación de aplicaciones (API) son una parte integral del desarrollo moderno, que permiten que las aplicaciones de software se comuniquen y utilicen funciones de otras aplicaciones o servicios de software. Por lo tanto, el diseño de API se convierte en una parte clave de cualquier proceso de desarrollo de software. Además, los conceptos básicos del diseño de API abarcan la comprensión de los principios de qué es una API, cómo funciona y los distintos tipos de API, como REST, SOAP y GraphQL. Esto también incluye comprender los estándares y las mejores prácticas en el diseño de API para garantizar el desarrollo de API potentes, fáciles de usar y seguras. La base del diseño de API reside en este conocimiento, que prepara el escenario para un diseño y desarrollo de API más complejos.

### Definición
En esencia, una API (Interfaz de Programación de Aplicaciones) es un conjunto de **reglas, protocolos y herramientas** que permiten que diferentes aplicaciones de software se comuniquen e intercambien datos entre sí. Piensa en ella como un contrato o un intermediario que define cómo un componente de software puede solicitar y obtener información o funcionalidad de otro, sin necesidad de conocer los detalles internos de su implementación. Simplifican el desarrollo al permitir que los programadores integren funcionalidades complejas de manera sencilla.

---
### Conceptos Clave
- **Interfaz**: Es el punto de contacto entre dos sistemas. No expone la lógica interna, solo las entradas y salidas esperadas.
- **Contrato**: La API defines las solicites que se pueden hacer, cómo hacerlas, los formatos de datos que se deben usar y las respuestas que se esperan. Ambas partes (el proveedor y el consumidor de la API) deben adherirse a este contrato.
- **Abstracción**: Oculta la complejidad del sistema subyacente. El consumidor de la API no necesita saber cómo funciona internamente el proveedor, solo cómo interactuar a través de la API
- **Intermediario (Mediador)**: Facilita la comunicación. El cliente realiza una solicitud a la API, y la API la traduce y la envía al servidor. Luego, la API recibe la respuesta del servidor y la entrega al cliente.
- **Reusabilidad**: Permite que la funcionalidad desarrollada por un equipo o servicio sea utilizada por muchos otros, promoviendo la modularidad y la eficiencia.
- **Desacoplamiento**: Los sistemas cliente y servidor pueden evolucionar de forma independiente, siempre y cuando el contrato de la API se mantenga estable.

### Ejemplos Prácticos
1. **Inicio de sesión con Google/Facebook (OAuth)**:
	- Cuando una aplicación te permite "Iniciar sesión con Google", está utilizando la API de Google.
	- Tu aplicación (cliente) solicita a la API de Google (servidor) que autentique al usuario.
	- Google maneja el proceso de inicio de sesión y, si tiene éxito, devuelve un token a tu aplicación, confirmando la identidad del usuario sin que tu aplicación necesite acceder directamente a las credenciales de Google del usuario.
2. **Mostrar un mapa en una aplicación (Google Maps API)**:
	- Una aplicación de entregas o de viajes compartidos necesita mostrar mapas
	- En lugar de construir un sistema de mapas desde cero, utiliza la API de Google Maps.
	- La aplicación encía una solicitud a la API (por ejemplo, "mostrar mapa en la dirección X") y la API de Google Maps devuelve los datos del mapa para ser mostrados.
3. **Procesamiento de pagos (Stripe/PayPal API)**:
	- Una tienda en línea necesita procesar pagos 
	- Utiliza la API de Stripe. Cuando un cliente realiza una compra, la tienda envía los detalles del pago de forma segura a través de la API de Stripe.
	- Stripe procesa el pago y devuelve una confirmación (o un error) a la tienda en la línea.
4. **Obtener datos meteorológicos (OpenWeatherMap API)**:
	- Una aplicación del clima obtiene la información más reciente sobre el pronóstico.
	- Hace una llamada a la API de OpenWeatherMap especificando una ubicación.
	- La API responde con los datos meteorológicos (temperatura, humedad, pronóstico, etc.) en un formato estructurado (generalmente JSON)
```json
// Example request to a weather API (conceptual)
// GET /weather?city=London&units=metric

// Example response from a weather API
{
  "city": "London",
  "temperature": 15,
  "unit": "celsius",
  "description": "Cloudy",
  "humidity": 70
}
```

### Notas
- **Entender la analogía del restaurante**: El cliente es el comensal, el mesero es la API y la cocina es el sistema/servidor. El comensal no necesita saber cómo se prepara la comida, solo le pide al mesero (API) lo que quiere del menú (funcionalidades expuestas por el API).
- **La clave es la comunicación estandarizada**: Sin APIs, la integración entre software sería un caos, requiriendo soluciones personalizadas y frágiles para cada conexión. Las APIs proporcionan un lenguaje común.
- P**ensar en bloques de Lego**: Las APIs permiten que diferentes "bloques" de software (servicios, aplicaciones) se conecten y trabajen juntos de manera eficiente para construir sistemas más grandes y complejos.
- **Seguridad como consideración temprana**: Aunque aquí solo definimos qué es una API, es crucial recordar desde el inicio que las APIs exponen funcionalidades y datos, por lo que la seguridad (autenticación, autorización, etc.) será fundamental más adelante.
- **No todas las APIs son web APIs**: Aunque las más comunes hoy en día son las Web APIs (que usan HTTP), el concepto de API es más amplio y puede aplicar a librerías de software, sistemas operativos, etc. Por ejemplo, la API de un sistema operativo permite a las aplicaciones interactuar con el hardware.
