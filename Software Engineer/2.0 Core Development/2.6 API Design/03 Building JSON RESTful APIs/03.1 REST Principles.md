## Definición
`REST (Representational State Transfer)` es un estilo arquitectónico propuesto por Roy Fielding que define cómo deben comunicarse los sistemas en la web. No es un protocolo rígido, sino un conjunto de **seis restricciones** que permiten que el cliente y el servidor evolucionen de forma independiente. Su esencia radica en que el cliente no manipula los recursos directamente, sino que interactúa con **representaciones** (como JSON o XML) de su estado.

## Conceptos Clave
Para que una `API` sea considerada `REST`, debe cumplir con las siguientes reglas (`constraints`):
- **`Client-Server` (Cliente-Servidor)**: Separación total de responsabilidades. El cliente se encarga de la interfaz y la experiencia del usuario, mientras que el servidor gestiona los datos y el almacenamiento. Esto permite que ambos se escalen y modifique por separado.
- **`Stateless` (Sin Estado)**: Cada solicitud del cliente al servidor debe contener toda la información necesaria para entenderla y procesarla. El servidor no debe recordar nada sobre solicitudes anteriores; no existe una "sesión" almacenada en el `backend`.
- **`Cacheable`**: Las respuestas deben definirse explícitamente como `cacheables` o no. Esto evita que los clientes tengan que solicitar los mismo datos una otra vez si no han cambiado, mejorando drásticamente el rendimiento.
- **`Uniform Interface` (Interfaz Uniforme)**: Es el pilar más crítico. Define una forma estándar de interactuar con el servidor.
	- **Identificación de recursos**: Uso de `URIs`.
	- **Manipulación mediante representaciones**: El cliente tiene una copia del recurso (`JSON/XML`) que puede modificar.
	- **Mensajes auto-descriptivos**: Cada mensaje contiene suficiente información para ser procesado (ej. `Content-Type`).
	- **`HATEOAS`**: Hipermedia como motor del estado de la aplicación.
- **`Layered System` (Sistemas de Capas)**: El cliente sabe (si necesita saber) si está hablando directamente con el servidor final o con un intermediario (como un balanceador de carga o un proxy).
- **`Code on Demand` (Opcional)**: El servidor puede enviar código ejecutable (como `JavaScript`) al cliente para ampliar su funcionalidad temporalmente.

## Ejemplos Prácticos
En una arquitectura `REST`, el flujo se basa en la transferencia de representaciones:
```http
// Solicitud de una representación del recurso "User"
GET /user/123 HTTP/1.1
Host: api.example.com
Accept: application/json

// Respuesta del servidor con la representación actual
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: max-age=3600

{
  "id": 123,
  "name": "Alex",
  "status": "active",
  "_links": {
    "self": "/users/123",
    "deactivate": "/users/123/status"
  }
}
```

## Analogía
Imagina un Restaurante de Comida Rápida:
1. **`Client-Server`**: Tú (`Client`) pides comida, la cocina (`Server`) la prepara. Tú no sabes cómo se cocina, y la cocina no sabe cómo vas a comer.
2. **`Stateless`**: El cajero no te reconoce. Cada vez que pides, debes decir tu orden completa. No puedes decir "quiero lo mismos de ayer" si no traes el recibo.
3. **`Uniform Interface`**: El menú es el mismo para todos. Usas los mismos términos (Combo 1, Combo 2) sin importar quién seas.
4. **`Representation`**: No te dan el inventario de carne y pan; te dan una hamburguesa armada (la representación del producto).

## Buenas Prácticas
- **Sustantivos, no verbos**: Usa `/books` en lugar de `/getAllBooks`.
- **Plurales consistentes**: Mantén todos tus `endpoints` en plural para mayor claridad.
- **Versionado temprano**: Aunque `REST` sugiere evolución sin versiones, en la práctica industrial es mejor usar `/v1/` para evitar romper clientes existentes.
- **`HATEOAS` para `APIs` públicas**: Si tu `API` será consumida por terceros, incluir enlaces de navegación hace que sea mucho más fácil de descubrir.

## Notas Personales
- A menudo confundimos "hacer una `API` con `JSON`" con "hacer una `API REST`". Si tu `API` no es *`stateless`* o no usa una interfaz uniforme, es simplemente una `Web API`, no `RESTful`.
- Aunque hace que el servidor sea fácil de escalar, obliga al cliente a enviar `tokens` de autenticación en cada solicitud, lo que puede aumentar ligeramente el tamaño de los `headers`.
