## Definición
`Problem Details` es un estándar (originalmente `RFC 7807`, ahora actualizado a `RFC 9457`) que define un formato de documento "modelo" para llevar detalles de errores legibles por máquinas en las respuestas `HTTP`. Su objetivo es evitar que cada `API` invente su propio formato de error, proporcionando una estructura predecible que permite a los clientes automatizar la lógica de recuperación o diagnóstico.

## Conceptos Clave
- **`Media Type`**: Siempre debes usar `application/problem+json` (o `+xml`). Esto indica explícitamente al cliente que el cuerpo no es el recurso esperado, sino un objeto de error estándar.
- **`type (URI)`**: El identificador único del tipo de problema. Idealmente apunta a una página de documentación. Si no tienes una, usa `about:blank`.
- **`title (String)`**: Un resumen breve y fijo del error (ej. "`Insuffient Funds`"). No debe cambiar entre ocurrencias para facilitar el agrupamiento de `logs`.
- **`status (Number)`**: Un duplicado del código de estado `HTTP` (ej. `403`). Ayuda a procesar el error si los encabezados se pierden o se filtran.
- **`detail (String)`**: Explicación humana específica de esta ocurrencia (ej. "Tu saldo es $10 y el cargo es de $50").
- **`instance (URI)`**: Un identificador único para este error específico (útil para buscar en `Sentry` o `logs` internos).
- **`Extensions`**: Puedes añadir campos extra (como `invalid_params`) para errores de validación, siempre que no expongas datos sensibles.

## Ejemplos Prácticos
##### Respuestas de error estándar para un fallo de validación en tu proyecto
```json
// Content-Type: application/problem+json
{
  "type": "https://stencil.app/errors/validation-failed",
  "title": "Your request parameters didn't validate.",
  "status": 400,
  "detail": "The 'requirement_id' field is mandatory for analysis.",
  "instance": "/logs/err_987654321",
  "invalid_params": [
    {
      "name": "requirement_id",
      "reason": "must not be null"
    }
  ]
}
```

## Analogía
Imagina que el manejo de errores tradicionales es una nota escrita a mano por un doctor con letra ilegible. Tú no sabes si es grave o qué medicina tomar. `RFC 9457` es un Reporte Clínico Digital Estandarizado: tiene un código de diagnostico universal (`type`), el nombre de la enfermedad (`title`), tu temperatura actual (`status`) y las notas específicas de tu caso (`detail`). Cualquier farmacia (`cliente API`) puede leer el código y sabes exactamente qué protocolo seguir.

## Buenas Prácticas
- **No reinventes la rueda**: Usa los tipos de problemas comunes registrados en la nueva `IANA Registry` introducida en `RFC 9457`.
- **Usa `URIs` descriptivas**: Aunque `type` no tiene que ser una `URL` real que cargue algo, es mejor práctica que lo sea, para que el desarrollador pueda hacer clic y leer la documentación.
- **Consistencia**: El `title` debe ser siempre el mismo para el mismo `type`. Si quieres dar detalles específicos, usa el campo `detail`.
- **Seguridad**: Nunca incluyas `Stack Traces` ni rutas de archivos del servidor en los campos `detail` o extensiones.

## Notas Personales
- Implementar esto en Laravel es sencillo extendiendo el `Handler` de excepciones. Te ahorra horas de soporte técnico porque el error le dice al usuario exactamente qué hizo mal.
- `RFC 9457` permite ahora reportar múltiples problemas en una sola respuesta si es necesario, aunque lo ideal sigue siendo mantener la simplicidad de un problema principal con detalles secundarios.
