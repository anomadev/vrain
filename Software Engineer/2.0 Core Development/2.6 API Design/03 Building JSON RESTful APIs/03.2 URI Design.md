## Definición
El Diseño de `URIs` en una `API REST` es el proceso de crear direcciones (`Uniform Resources Identifiers`) que sean intuitivas, predecibles y persistentes. Una buena `URI` debe actuar como una dirección postal clara: debe indicar exactamente dónde reside un recurso sin necesidad de explicar las acciones que se realizarán sobre él, apoyándose en la estructura jerárquica de la web.

## Conceptos Clave
Para dominar el diseño de `URIs`, aplica estas reglas fundamentales:
- **Los recursos son Sustantivos, no Verbos**: Las `URIs` deben representar "cosas" (entidades), no acciones. La acción la define el método `HTTP` (`GET`, `POST`, etc.).
- **Jerarquía y Relaciones**: Usa la barra diagonal (`/`) para indicar una relación de subordinación o jerarquía entre recursos.
- **`Kebab-Case vs Snake_Case`**: La convención más aceptada en `URIs` es el `kebab-case` (guiones medios), ya que son más legibles y amigables para los motores de búsqueda.
- **Colecciones y Elementos**: Una colección se representa en plural (`/users`) y un elemento específico añadiendo su `ID` (`/users/123`).
- **Minúsculas**: Las `URIs` son sensibles a mayúsculas/minúsculas en algunos servidores. La mejor práctica es usar simplemente minúsculas para evitar confusiones.
- **Sin extensiones de archivo**: Evita `.json`, `.xml` o `.php` en la `URL`. El formato se negocia mediante cabeceras (`Accept`).

## Ejemplos Prácticos
A continuación, la forma correcta e incorrecta de diseñar rutas para un sistema de comercio exterior (SaaS):
```
# Good: Resource-based and hierarchical
GET    /shipments                   # List of all shipments
GET    /shipments/SH-990            # Specific shipment details
GET    /shipments/SH-990/documents  # Collection of documents for that shipment
POST   /shipments/SH-990/documents  # Add a document to a specific shipment

# Bad: Verb-based or messy
GET    /getShipment?id=990          # Action in URL (RPC style)
POST   /shipments/update/990        # Verb "update" inside the path
GET    /Shipments/sh990.json        # CamelCase and file extension
```

## Analogía
Imagina que las `URIs` son la etiquetas de los estantes de un almacén.
- El estante principal dice `PRODUCTOS`
- Dentro, hay secciones numeradas: `PRODUCTOS/4`
- Si quieres saber qué hay en la caja de ese estante, miras `PRODUCTOS/4/DETALLES`
No pones una etiqueta que diga `TRAER_EL_PRODUCTO_4`. El estante simplemente es el lugar donde está el producto. Si quieres "traerlo", simplemente lo tomas (haces un `GET`). Si quieres "tirarlo", lo quitas del estante (haces un `DELETE`).

## Buenas Prácticas
- **Plurales siempre**: Usa `/customers` en lugar de `/customer`. Es más consistente cuando tratas con colecciones y elementos individuales.
- **Usa `IDs` opacos**: Evita usar nombres o datos sensibles en la `URI` si pueden cambiar. Los `UUIDs` o `IDs` numéricos son más persistentes.
- **Evita el anidamiento profundo**: Más de dos o tres niveles (ej. `/users/1/posts/2/comments/3`) hace que la `API` sea difícil de usar. Considera aplanar la estructura.
- **`Query strings` para filtrados**: No uses la ruta para filtrar. Mal: `/users/active`. Bien: `/users?status=active`.

## Notas Personales
- Al principio, cuesta dejar de pensar en "páginas" o "scripts" y empezar a pensar en "recursos".
- La persistencia es clave. Una `URI` bien diseñada debería ser válida durante años. Si cambias la lógica interna de tu `SaaS`, las `URIs` de tus clientes no deberían romperse.
