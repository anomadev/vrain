## Definición
El `API Versioning` es la práctica de gestionar los cambios realizados en una `API` de forma que se garantice la estabilidad para los clientes existentes. Es el proceso que permite introducir mejoras, corregir errores o cambiar el contrato de la interfaz sin romper las integraciones actuales, permitiendo que múltiples versiones del contrato coexistan simultáneamente.

## Conceptos Clave
- **`Backward Compatibility`**: La capacidad de una versión nueva de seguir funcionando con clientes diseñados para versiones anteriores (evitando los `Breaking Changes`).
- `Semantic Versioning (SemVer)`: Estándar de nomenclatura basado en tres números `MAJOR.MINOR.PATCH`.
	- `MAJOR`: Incremento cuando hay cambios incompatibles.
	- `MINOR`: Incremento cuando se añade funcionalidad compatible.
	- `PATCH`: Incremento para correcciones de errores compatibles.
- Estrategias de Implementación:
	- `URI Versioning`: Incluir la versión en la ruta (`/v1/users`). Es la más común y fácil de cachear.
	- `Header Versioning`: Pasar la versión en una cabecera `HTTP` personalizada (`X-API-Version: 2`). Mantiene las `URIs` limpias.
	- `Media Type Versioning`: Usar el encabezado `Accept` para negociar el contenido (`Accept: application/vnd.myapi.v1+json`). Es la más purista desde el punto de vista `REST`.
- `Deprecation & Sunset`: Políticas para informar a los usuarios que una versión dejará de recibir soporte o será eliminada.

## Ejemplos Prácticos
```http
// 1. URI Versioning (Standard industry practice)
GET /api/v1/products

// 2. Query Parameter Versioning
GET /api/products?version=2

// 3. Header Versioning
GET /api/products
X-API-Version: 1

// 4. Media Type Versioning (Content Negotiation)
GET /api/products
Accept: application/vnd.company.app.v1+json
```

## Analogía
Imagina que el versionado de una `API` es como el **menú de un restaurante**.
- **`v1`**: Es el menú original
- **`v2`**: El chef decide cambiar la receta de la hamburguesa y eliminar los refrescos
- Para no perder clientes que aman la receta vieja, el restaurante mantiene el Menú Clásico (`v1`) mientras ofrece el Menú Nuevo (`v2`). Eventualmente, podrán un aviso diciendo que el Menú Clásico que dejará de servir el próximo mes (`Sunset Policy`), dando tiempo a los clientes para que se acostumbran a la nueva receta.

## Buenas Prácticas
- **`Use URI Versioning for simplicity`**: Aunque existen métodos más elegantes, el versionado en la `URL` es el estándar de facto por su visibilidad y facilidad de prueba.
- **`Increment the Major version only for Breaking Changes`**: No fuerces a tus clientes a migrar si el cambio no rompe su código.
- **`Implement a Sunset Header`**: Utiliza cabeceras estándar como `Deprecation` y `Sunset` para comunicar fechas de fin de vida de forma programática.
- **`Versioning the entire API, not individual endpoints`**: Es más fácil de gestionar y entender para el consumidor.

## Notas Personales
- El versionado en la `URI` es pragmático pero técnicamente rompe el principio de que la `URI` debe identificar el recurso, no su representación. Sin embargo, en el mundo real, la `Developer Experience (DX)` suele ganar sobre la teoría pura de `REST`.
- El mayor reto no es técnico, sino comunicativo: establecer una política de `Sunset` justa que no enfurezca a tus clientes pero que te permita limpiar el código `legacy`.
