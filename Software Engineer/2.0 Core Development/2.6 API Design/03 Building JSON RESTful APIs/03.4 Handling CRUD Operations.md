## Definición
Las operaciones `CRUD` son las cuatro funciones básicas de almacenamiento persistente. En el contexto de las `APIs REST`, el manejo de `CRUD` consiste en mapear estas acciones de base de datos a los **métodos `HTTP`** estándar, permitiendo que un cliente manipule recursos (entidades) de forma predecible. Mientras que `CRUD` se enfoca en la persistencia, una `API` robusta debe ir más allá, capturando la **intención del negocio** y no solo el cambio de datos.

## Conceptos Clave
Para dominar `CRUD` en `REST`, lo más importante es el contrato semántico entre el verbo y el resultado:
1. Mapeo Estándar:
	- `Create`: `POST` (a una colección) o `PUT` (a una `URI` específica si el cliente conoce el `ID`).
	- `Read`: `GET` (recurso individual o colección).
	- `Update`: `PUT` (reemplazo total) o `PATCH` (modificación parcial).
	- `Delete`: `DELETE`
2. Códigos de Estado Críticos:
	- `201 Created`: Éxito en creación + cabecera `Location`.
	- `200 OK`: Éxito con cuerpo de respuesta.
	- `204 No Content`: Éxito sin cuerpo (común en `DELETE` o `PUT`).
	- `409 Conflict`: Intento de crear un recurso duplicado.
3. Idempotencia y Seguridad: `*GET` es seguro (no cambia nada):
	- `GET`, `PUT` y `DELETE` son idempotentes (repetirlos no cambia el resultado final). `POST` no lo es.

## Ejemplos Prácticos
Utilizando una estructura típico de `Node/Express` para tu lógica de negocio.
```javascript
// GET /shipmets - Read Collection
router.get('/shipmets', async (req, res) => {
	const data = await Shipmet.findAll({ limit: req.query.limit });
	res.status(200).json(data);
});

// POST /shipments - Create Resource
router.post('/shipments', async (req, res) => {
	const newShipment = await Shipment.create(req.body);
	res.status(201)
		.header('Location', `/shipments/${newShipment.id}`)
		.json(newShipment);
});

// PATCH /shipments/:id - Partial Update (Intent-based)
router.patch(`/shipments/:id`, async (req, res) => {
	const updated = await Shipment.update(req.body, { where: { id: req.params.id } });
	if (!updated) return res.status(404).send();
	res.status(200).json(updated);
});
```

## Analogía
Imagina que eres un **Archivista en una oficina**:
- **`POST`**: Alguien te entrega un documento nuevo sin número; tú le asignas una carpeta, le pones un número y le dices al remitente dónde lo guardaste.
- **`GET`**: Alguien te pide ver el expediente `123`; tú se lo prestas para que lo lea. No cambias nada.
- **`PUT`**: Te traen un expediente completo para reemplazar el `123`. Tiramos el viejo y ponemos el nuevo.
- **`PATCH`**: Te traen una nota para añadir o corregir solo un dato dentro del expediente `123`.
- **`DELETE`**: Te ordenan destruir el expediente `123`. Una vez hecho, si te lo piden nuevo, dirás que no existe.

## Buenas Prácticas
- **Retorna el recurso creado/actualizado**: No solo respondas "OK"; envía el objeto completo para que el `frontend` actualice su estado sin otra llamada `GET`.
- **Usa validación de esquemas**: Antes de procesar un `POST` o `PUT`, valida el `body` (ej. con `Joi` o `Zod`) para evitar datos corruptos.
- **Implementa `ETags` para actualizaciones concurrentes**: Evita que dos usuarios sobrescriban el mismo recurso al mismo tiempo (`Lost Update Problem`).
- **Diferencia `PUT` de `PATCH`**: No uses `PUT` para actualizaciones parciales; esto rompe la idempotencia si el cliente olvida campos obligatorios.

## Notas Personales
En sistemas complejos (como un SaaS), a veces el `CRUD` puro se queda corto. Pasar de un `PATCH` genérico a un endpoint con intención (ej. `POST /shipments/{id}/verify`) puede evitar muchos `bugs` de concurrencia. Para operaciones de lectura (`Read`), la paginación y el filtrado por `query params` no son opcionales, son obligatorios para mantener la `API` responsiva.
