## Definición
La paginación es la técnica de dividir un conjunto de datos extenso en fragmentos más pequeños (páginas) que se entregan al cliente de forma secuencial. Su objetivo es optimizar el tiempo de respuesta del servidor, reducir el consumo de ancho de banda y evitar el agotamiento de recursos (memoria/CPU) tanto en el `backend` como en el `frontend` al procesar listas masivas.

## Conceptos Clave
Existen tres estrategias dominantes en la industria:
- **`Offset Pagination (Limit/Offset)`**: Es el método más sencillo El cliente solicita una cantidad de elementos (`limit`) y se salta un número específico (`offset`).
	- Pros: Fácil de implementar en `SQL` (`OFFSET` y `LIMIT`). Permite saltar a cualquier página.
	- Contras: Se vuelve lento en `offsets` muy grandes y puede mostrar datos duplicados si se insertan registros mientras el usuario navega.
- **`Keyset/Cursor Pagination`**: Utiliza un identificador único (un "cursor", usualmente un `ID` o un `timestamp`) para marcar dónde terminó la última página.
	- Pros: Extremadamente eficiente para grandes `datasets` y estable ante cambios en los datos en tiempo real. Es el estándar en `APIs` de alto rendimiento (`Facebook`, `Slack`).
	- Contras: No permite saltar a una página específica (ej. "ir ala página 10").
- **`Page-based Pagination`**: Similar al `offset`, pero usa un número de página (`page`) y el tamaño de la misma (`size`). Es muy común en interfaces web tradicionales.
- **`Pagination Metadata`**: Información adicional que el servidor devuelve para que el cliente sepa dónde está. Incluye campos como `total_count`, `next_page_url`, y `has_more`.

## Ejemplos Prácticos
##### `Offset Pagination (Query Params)`
```http
// Request: Get the second page of shipments, 20 per page
GET /shipments?limit=20&offset=20 HTTP/1.1
```
##### `Cursor Pagination (JSON Response)`
```json
{
  "data": [
    { "id": "uuid-990", "reference": "EXP-2026-A" },
    { "id": "uuid-991", "reference": "EXP-2026-B" }
  ],
  "links": {
    "next": "/shipments?cursor=uuid-991&limit=2",
    "prev": "/shipments?cursor=uuid-990&limit=2"
  ,
  "meta": {
    "cursor": "uuid-991",
    "has_more": true
  }
}
```
##### `Link Header` (Estándar de Atlassian/GitHub)
En lugar de ensuciar el `JSON`, se envían los enlaces de navegación en las cabeceras `HTTP`:
```http
Link: <https://api.example.com/users?page=3>; rel="next",
	  <https://api.example.com/users?page=1>; rel="prev"
```

## Analogía
Imagina que estás leyendo una Enciclopedia de 20 tomos.
- **Sin paginación**: Intentas cargar los 20 tomos en tus brazos a la vez. Probablemente te caigas y no puedas moverte (`timeout` del servidor).
- **`Offset Pagination`**: "Tráeme el tomo que está en la posición 5". Si alguien quita un tomo antes de que llegues, podrías terminar con el tomo equivocado.
- **`Cursor Pagination`**: "Terminé de leer el tomo con el código `CIENCIAS-A`. Tráeme el que sigue después de ese". No importa si la estantería cambia, siempre sabrás cuál es el siguiente.

## Buenas Prácticas
- `Set Default Limits`: Nunca permitas un `limit` infinito. Establece un valor por defecto (ej. 20) y un máximo permitido (ej. 100) para proteger tu infraestructura.
- `Use Cursors for Infinite Scroll`: Si estás construyendo un `feed` o una lista que crece constantemente, el cursor es obligatorio para evitar duplicados.
- `Include Total Count selectively`: Calcular el `total_count` en bases de datos masivas es costoso. Si no es estrictamente necesario para la UI, omítelo o usa una estimación.
- `Consistency`: Elige una estrategia y aplícala de forma idéntica en todos tus `endpoints`.

## Notas Personales
- La `Cursor Pagination` es tu mejor aliada para los `logs` de envíos o movimientos de carga, ya que son datos que se insertan constantemente.
- Casi nadie usa el salto de página en una `API`; los clientes suelen consumir los datos de forma secuencial o mediante búsqueda. No sacrifiques rendimiento por una característica que nadie usa.
- `Caching` ([[1.09 HTTP Caching]]): Las páginas con `offset` son más fáciles de `cachear`, pero los cursores ofrecen una `API` mucho más robusta para sistemas en tiempo real.
