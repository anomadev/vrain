## Definición
El `Rate Limiting` es una estrategia de control de tráfico que limita el número de solicitudes que un cliente puede realizar a una `API` en un periodo de tiempo determinado. Su objetivo principal es garantizar la disponibilidad del servicio, prevenir ataques de denegación de servicio (`DoS`) y proteger los recursos del servidor (CPU, Memoria, Base de Datos) de ser sobrepasados por un solo consumidor.

## Conceptos Clave
Para entender el diseño de límites, debemos distinguir tres conceptos que a menudo se confunden:
- **`Rate Limit`**: Bloquea las peticiones que exceden el límite (*ej. Máximo 100 peticiones por minuto*). Responde con un error `429 Too Many Requests`.
- **`Throttling`** (Estrangulamiento): En lugar de bloquear, ralentiza el procesamiento de las peticiones para mantener el consumo dentro de los límites permitidos.
- **`Debouncing`**: Asegura que una acción solo se ejecute después de que haya pasado un tiempo de "silencio" (muy usado de `frontends` para `inputs` de búsqueda que llaman a una `API`).
**Algoritmos Principales**
1. **`Token Bucket`**: El servidor tiene un "cubo" de `tokens`. Cada petición consume uno. Los `tokens` se reponen a una tasa constante. Permite ráfagas (`bursts`) de tráfico si el cubo está lleno.
2. **`Leaky Bucket`**: Las peticiones entran al cubo y "salen" (se procesan) a una tasa constante. Si el cubo se llena, las nuevas peticiones se descartan. Suaviza el tráfico perfectamente.
3. **`Fixed Window Counter`**: Cuenta peticiones en ventanas de tiempo fijas (*ej. 12:00 a 12:01*). Es simple pero permite el doble de tráfico en los bordes de la ventana.
4. **`Sliding Window Log/Counter`**: Una versión más precisa que evita el problema de los bordes al rastrear el tiempo exacto de cada petición.

## Ejemplos Prácticos
##### Cabeceras `HTTP` Estándar
Cuando implementas `Rate Limiting`, el servidor debe informar al cliente su estado actual mediante `headers`:
```
HTTP/1.1 429 Too Many Requests
Content-Type: application/json
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1718234500  // Unix timestamp de cuándo se reinicia el límite
Retry-After: 30                // Segundos sugeridos para reintentar

{
  "error": "Rate limit exceeded. Please try again in 30 seconds."
}
```
##### Implementación Conceptual (Pseudo-código)
```javascript
// Simple Fixed Window logic with Redis
const currentRequest = await redis.incr(userKey);
if (currentRequests === 1) {
	await redis.expire(userKey, 60); // Window of 60 seconds
}

if (currentRequests > 100) {
	return res.status(429).send("Too Many Requests");
}
```

## Analogía
Imagina un **Dispensador de Agua** en una oficina:
- `Rate Limit`: La máquina solo permite llenar 5 vasos por hora por empleado. Si intentas el sexto, la palanca se bloquea.
- `Throttling`: Si muchos intentan llenar agua a la vez, el chorro sale más delgado para que el tanque no se vacíe de golpe, obligando a todos a esperar un poco más.
- `Debouncing`: El dispensador espera a que dejes de presionar el botón por 2 segundos antes de registrar cuánta agua serviste en total.

## Buenas Prácticas
- Diferencia por `Tier`: No limites igual a un usuario anónimo que a uno de pago en tu SaaS.
- Usa `Redis` o `Memcached`: El conteo de peticiones debe ser ultra rápido y compartido entre múltiples instancias de tu servidor.
- Información Clara: Siempre devuelve las cabeceras `X-RateLimit-*` para que los desarrolladores clientes puedan programar reintentos de forma inteligente.
- `Whitelist`: Permite que tus propios servicios internos o socios críticos tengan límites mucho más altos o inexistentes.

## Notas Personales
- El `Rate Limiting` es crucial no solo por seguridad, sino por costos. Si usas servicios de terceros que te cobran por petición (como tracking de carga), un bucle infinito en el cliente podría vaciar tu cuenta de banco en minutos.
- Considera el "`Graceful Degradation`": Si el sistema está bajo mucho estrés, puedes reducir los límites globalmente de forma temporal.
- Los límites suelen aplicarse por `API Key` o `User ID`, no solo por `IP` (que puede ser compartida por muchos usuarios).
