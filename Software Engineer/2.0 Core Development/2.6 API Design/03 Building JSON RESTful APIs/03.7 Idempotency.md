## Definición
En el diseño de `APIs`, la **idempotencia** es la propiedad de ciertas operaciones que permite que estas se realicen múltiples veces sin cambiar el resultado más allá de la aplicación inicial. En términos de `HTTP`, un método es idempotente si una solicitud idéntica puede hacerse repetidamente y el estado del servidor permanece igual que después de la primera llamada exitosa. Es una garantía crítica para la **tolerancia a fallos** y la fiabilidad en sistemas distribuidos.

## Conceptos Clave
Para un arquitecto, entender la idempotencia implica clasificar los métodos HTTP según su comportamiento.
- `Safe Methods (Métodos Seguros)`: Son aquellos que no modifican el estado del recurso (solo lectura). Todos los métodos seguros son intrínsecamente idempotentes `GET`, `HEAD`, `OPTIONS`.
- `Idempotent Methods (Métodos Idempotentes)`: No son necesariamente "seguros" (pueden modificar datos), pero el efecto final es el mismo tras $N$ ejecuciones.
	- `PUT`: Reemplaza el recurso. Si lo reemplazas 10 veces con el mismo objeto, el resultado es el mismo.
	- `DELETE`: Borra el recurso. La primera vez se borra; las siguientes veces el recurso sigue sin existir.
- `Non-Idempotent Methods (No Idempotentes)`: Cada ejecución puede producir un cambio adicional en el estado o un nuevo recurso.
	- `POST`: Generalmente se usa para crear. Si envías 5 veces el mismo `POST`, podrías terminar con 5 registros duplicados.
	- `PATCH`: Depende de la implementación. Si incrementas un valor (`counter += 1`), no es idempotente. Si asignas un valor fijo, si lo es.

## Ejemplos Prácticos
##### `PUT` (Idempotente)
Incluso si hay un reintento por fallo de red, el recurso queda exactamente igual.
```http
PUT /api/v1/shipments/SH-123 HTTP/1.1
Content-Type: application/json

{
    "status": "DELIVERED",
    "delivery_date": "2026-03-14"
}
```
##### `DELETE` (Idempotente)
El primer llamado devuelve `204 No Content`. El segundo llamado (reintento) devuelve `404 Not Found`, pero el estado del servidor es el mismo: el recurso ya no está.
```http
DELETE /api/v1/shipments/SH-123 HTTP/1.1
```
##### `POST` (No Idempotente)
Sin controles adicionales, esto genera duplicidad.
```http
POST /api/v1/payments HTTP/1.1
{
    "amount": 100.00,
    "currency": "USD"
}

// Response 1: 201 Created (ID: 1)
// Response 2 (Retry): 201 Created (ID: 2) -> Duplicate payment!
```

## Analogía
Imagina un Interruptor de Luz vs. un Botón de Ascensor:
- **Idempotente (Interruptor)**: Si mueves el interruptor a la posición de "`ENCENDIDO`", la luz se prende. Si lo mueves a "`ENCENDIDO`" 10 veces más, la luz simplemente sigue prendida. Es estado final es predecible.
- **No idempotente (contador de personas)**: Cada vez que alguien cruza una puerta y presiona un botón en "`AÑADIR`", el contador aumenta. Si presionas el botón 3 veces por error, el contador marcará 3 personas extra. El estado final depende de cuántas veces se pulsó el botón.

## Buenas Prácticas
- **`Network Retries`**: Los clientes (como móviles o browsers) deben reintentar automáticamente métodos idempotentes (`GET`, `PUT`, `DELETE`) si hay un error de conexión, pero nunca un `POST` sin una `Idempotency Key`.
- **`Idempotency Keys`**: Para hacer que un `POST` sea seguro, requiere que el cliente envíe un `header` único (ej. `Idempotency-Key: uuid`). El servidor guarda el resultado de ese `UUID` y, si recibe el mismo, devuelve la respuesta cacheada en lugar de procesar la acción de nuevo.
- **`Statelessness`**: La idempotencia facilita el escalado horizontal. Si una petición `PUT` falla y el cliente reintenta contra otro servidor del `cluster`, el resultado final será consistente.
- **`Status Codes`**: No te obsesiones con devolver siempre el mismo código. En `DELETE`, el primero puede ser `204` y el segundo `404`; lo que importa es que el estado del recurso no cambió entre el primer y el segundo intento.

## Notas Personales
- Siempre diseñar pensando que la red va a fallar. Si la `API` no es idempotente en sus mutaciones, el sistema no es resiliente.
- Conecta este tema con `Caching` y `Error Handling`. Un `PUT` bien diseñado permite que el cliente sea mucho más agresivo con sus reintentos sin miedo a corromper los datos.
