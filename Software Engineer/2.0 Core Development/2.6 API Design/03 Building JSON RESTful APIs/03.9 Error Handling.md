## Definiciﾃｳn
El Manejo de Errores en una `API` es el proceso de comunicar fallos de manera clara, predecible y segura. Consiste en utilizar los cﾃｳdigos de estado `HTTP` adecuados junto con un cuerpo de respuesta estandarizado que proporcione contexto humano y procesable por mﾃ｡quinas. Un buen sistema de errores transforma un fallo frustrante en una guﾃｭa de resoluciﾃｳn para el desarrollador.

## Conceptos Clave
1. **Uso Semﾃ｡ntico de Status Codes**:
	- **`4xx` (Errores del Cliente)**: El cliente enviﾃｳ algo mal (`400 Bad Request`, `401 Unauthorized`, `422 Unprocessable Entity`).
	- **`5xx` (Errores del Servidor)**: El servidor fallﾃｳ al procesar una peticiﾃｳn vﾃ｡lida (`500 Internal Server Error`, `503 Service Unavailable`).
2. **Estandarizaciﾃｳn (RFC 7807 / 9457)**: Conocido como `Problem Details for HTTP APIs`. Define un esquema estﾃ｡ndar con campos como `type` (`URI` del error), `title` (resumen corto), `status` (cﾃｳdigo `HTTP`), `detail` (explicaciﾃｳn humana) e `instance` (`URI` de la ocurrencia).
3. **`Application-Specific Error Codes`**: Cﾃｳdigos internos (ej. `ERR-CONF-001`) que permitan al cliente manejar errores de forma programﾃ｡tica y mﾃ｡s granular que un simple cﾃｳdigo `HTTP`.
4. **`Actionable Messages`**: Los mensajes deben ser especﾃｭficos. En lugar de "`Invalid Input`", usa "`Email formar is incorrect in user_email field`".
5. **Sanitizaciﾃｳn y Seguridad**: Es vital nunca exponer `Stack Traces`, nombres de tablas de la `DB` o versiones de software en las respuestas de producciﾃｳn.
6. **Centralizaciﾃｳn (`Middleware`)**: Implementar un manejador global (ej. `Global Exception Handler` en `Spring` o `Laravel`) para asegurar que todos los errores tengan el mismo formato.

## Ejemplos Prﾃ｡cticos
##### `Standard Error Object (RFC 7807 style)`
```json
{
  "type": "https://example.com/probs/out-of-credit",
  "title": "You do not have enough credit.",
```

##### `Laravel Error Handling (Simplified)`
```php
// app/Exceptions/Handler.php
public function render($request, Throwable $exception) {
	if ($exception instanceof ValidationException) {
		return response()->json([
			'error_code' => 'VAL-422',
			'message' => 'The given data was invalid.',
			'errors' => $exception->errors(),
		], 422);
	}
	
	return response()->json([
		'error_code' => 'SYS-500',
		'message' => 'An unexpected error occurred.',
		'request_id' => $request->header('X-Request-ID')
	], 500);
}
```

## Analogﾃｭa
Imagina que vas a un **Cajero Automﾃ｡tico (API)**:
- **Mal manejo de errores**: El cajero se reinicia o muestra una pantalla azul con cﾃｳdigo binario. No sabes si el banco se cayﾃｳ, si tu tarjeta no sirve o si no tienes dinero.
- **Buen manejo de errores**: El cajero muestra: "Saldo insuficiente (`COD: 403-BAL`)". Su saldo es $50, el retiro mﾃｭnimo es $100". Aquﾃｭ el sistema te da un cﾃｳdigo para el sistema, un mensaje para ti y una acciﾃｳn clara para resolver el problema.

## Buenas Prﾃ｡cticas
- `Never return 200 OK for errors`: Es un antipatrﾃｳn comﾃｺn que rompe el contrato `HTTP`.
- `Provide a Request ID`: Crucial para que el usuario pueda reportar el error y tﾃｺ puedas buscarlo en tus `logs` con exactitud.
- `Document all error codes`: Una `API` bien documentada incluye una tabla con cada `error_code`, su significado y cﾃｳmo solucionarlo.
- `Use Retry-After`: Si implementas `Rate Limiting` ([[03.6 Rate Limiting]]) o el servidor estﾃ｡ en mantenimiento, indica al cliente cuﾃ｡ndo es seguro volver a intentarlo.

## Notas Personales
- A veces olvidamos que el usuario de nuestra `API` es otro desarrollador. Facilitarle la vida con errores claros reduce drﾃ｡sticamente los `tickets` de soporte.
- En Laravel, centralizar los errores en el `Handler` permite que tus controladores se enfoquen ﾃｺnicamente en el "`Happy Path`".
- `Loguear` el error completo con `Stack Trace` en tu sistema de monitoreo (ej. `Sentry` o `Bugsnag`), pero entregar al cliente solo una respuesta sanitizada.
