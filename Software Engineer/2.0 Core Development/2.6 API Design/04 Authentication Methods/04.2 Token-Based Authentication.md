## Definición
La Autenticación Basada en Tokens es un protocolo de seguridad `stateless` (sin estado) donde el servidor genera un `token` firmado digitalmente tras validar las credenciales del usuario. Este `token` se entrega al cliente, quien lo almacena y lo envía en cada solicitud posterior para demostrar su identidad. A diferencia de las sesiones tradicionales, el servidor no necesita almacenar el estado del usuario en una base de datos o memoria para validar cada petición, ya que toda la información necesaria está contenida y protegida dentro del propio token.

## Conceptos Clave
- **`Statelessness`**: El servidor no guarda "sesiones". La validez se comprueba verificando la firma del token, lo que facilita el **escalado horizontal**.
- `Access Token`: El token de corta duración que se usa para acceder a recursos protegidos.
- `Refresh Token`: Un token de larga duración utilizado para obtener nuevos `access tokens` sin que el usuario tenga que reingresar sus credenciales.
- `JWT (Json Web Token)`: Es estándar más común. Se divide en:
	- `Header`: Algoritmo y tipo de token
	- `Payload`: Contiene los *`claims`* (datos del usuario, expiración, roles)
	- `Signature`: Criptografía que asegura que el token no ha sido alterado
- `Bearer Token`: Esquema donde quien posee el token tiene el acceso (enviado normalmente en el `header Authorization: Bearer <token>)
- `Signature Verification`: El servidor usa una `Private Key` (o `secret`) para firmar y una `Public Key` (o el mismo `secret`) para validar.

## Ejemplos Prácticos
##### `Login and Token Issuance (Server-side pseudo-code)`
```php
// After validating username and password
$payload = [
    'sub' => $user->id,
    'role' => $user->role,
    'iat' => time(),
    'exp' => time() + 3600 // 1 hour expiration
];

$jwt = JWT::encode($payload, config('app.jwt_secret'), 'HS256');
return response()->json(['access_token' => $jwt]);
```
##### `Accessing a Protected Resource (Client-side)`
```http
GET /api/v1/user/profile HTTP/1.1
Host: api.myapp.com
Authorization: Bearer <YOUR_JWT_TOKEN>
```

## Analogía
Imagina que vas a un Parque de Diversiones (`API`):
- `Session Auth`: Es como si el parque tuviera una lista con tu nombre en cada atracción. Cada vez que quieres subir a un juego, el operador debe buscarte en un libro gigante para ver si pagaste. Si hay mucha gente, buscar en el libro se vuelve lentísimo.
- `Token Auth`: Al entrar, te ponen una pulsera con un código QR firmado (`token`). El operador de la montaña rusa solo escanea tu pulsera y verifica que el código sea auténtico y no haya expirado. El parque no necesita un libro gigante; la prueba de tu acceso está en tu propia muñeca.

## Buenas Prácticas
- **`Use HTTPS (TLS)`**: Los `tokens` son como llaves maestras; si se interceptan en una red no segura, el atacante tiene acceso total.
- **`Short-lived Access Token`**: Mantén los `tokens` de acceso con una vida corta (ej. 15-30 min.) para minimizar el daño si uno es robado.
- **`Secure Storage`**: Evita guardar `tokens` sensibles en `localStorage` si eres vulnerable a `XSS`. Considera `HttpOnly Cookies` para mayor seguridad en la web.
- **`Payload Minimalism`**: No metas datos sensibles (como `password`) en el `payload` del `JWT`, ya que cualquier persona puede decodificarlo (`Base64` no es un cifrado).
- **`Token Revocation (Blacklisting)`**: Implementa un mecanismo (usualmente Redis) para invalidar `tokens` antes de su expiración en caso de `logout` o brecha de seguridad.

## Notas Personales
- El mayor beneficio que es la interoperabilidad. Un `token` generado en tu `backed` de Laravel puede ser validado por un microservicio en `Go` o `Node` sin compartir bases de datos de sesión.
- El reto principal es la invalidación. A diferencia de las sesiones, no puedes simplemente borrar una fila en la `DB` para "desloguear" a alguien instantáneamente sin usar una lista negra de `tokens` (que reintroduce un poco de estado al sistema).
- En sistemas distribuidos, esta es la opción por excelencia para evitar cuellos de botella en la base de datos de sesiones.
