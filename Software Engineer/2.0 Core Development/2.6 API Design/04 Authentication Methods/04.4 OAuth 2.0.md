## Definición
`OAuth 2.0` es un estándar de la industria que permite a una aplicación obtener acceso limitado a los datos de un usuario en otro sitio (como Google, GitHub o Facebook) sin que el usuario tenga que compartir su contraseña. Se basa en la delegación de permisos mediante `Access Tokens`, actuando como una capa de autorización que separa el rol del cliente del propietario de los recursos.

## Conceptos Clave
Para entender `OAuth 2.0`, primero debes identificar los 4 roles fundamentales:
1. **`Resource Owner`**: El usuario que posee los datos (tú).
2. **`Client`**: La aplicación que quiere acceder a tus datos (ej. una app de edición de fotos).
3. **`Authorization Server`**: El servidor que valida tu identidad y emite los `tokens` (ej. `Google Auth`).
4. **`Resource Server`**: El servidor que guarda tus datos reales y acepta el `token` (ej. `Google Drive API`).
#### Componentes esenciales:
- **`Scopes`**: Permisos granulares (ej. `read:photos`, `write:profile`).
- **`Access Token`**: La "llave" temporal que el cliente usa para acceder al `Resource Server`. 
- **`Refresh Token`**: Un `token` de larga duración para obtener nuevos `Access Tokens` sin molestar al usuario.
- **`Consent`**: La pantalla donde el usuario acepta dar permisos específicos al cliente.

### `Grant Types (Flows)`
No hay un solo camino en `OAuth`; el "flujo" depende del tipo de aplicación.
- **`Authorization Code Grant`**: El más común y seguro. Usa un código temporal que se intercambia por un `token` en el servidor.
- **`PKCE (Proof Key for Code Exchage)`**: Una extensión del anterior obligatoria para apps móviles y `SPAs (Single Page Apps)`, ya que no pueden guardar secretos de forma segura.
- **`Client Credentials`**: Para comunicación máquina a máquina (`M2M`) sin intervención de un usuario (ej. un `cron job` de `backend`).
- **`Device Code`**: Para dispositivos con entrada limitada (Smart TVs, consolas).
- **`Implicit & Passwords Grand`**: `Legacy/Deprecated`. No deben usarse en proyectos nuevos por riesgos de seguridad.

## Ejemplos Prácticos
##### `Abstract Authorization Code Flow`:
```http
// 1. Client redirects user to Authorization Server
GET /authorize?response_type=code&client_id=CLIENT_ID&scope=read:photos&redirect_uri=CALLBACK_URL

// 2. User grants consent, Server redirects back with 'code'
CALLBACK_URL?code=AUTHORIZATION_CODE

// 3. Client exchanges code for token (Server-to-Server)
POST /token
{
  "grant_type": "authorization_code",
  "code": "AUTHORIZATION_CODE",
  "redirect_uri": "CALLBACK_URL",
  "client_id": "CLIENT_ID",
  "client_secret": "CLIENT_SECRET"
}

// 4. Server returns Access Token
{ "access_token": "ACCESS_TOKEN", "expires_in": 3600 }
```

## Analogía
Imagina que `OAuth 2.0` es como el Servidor de Valet Parking de un hotel:
1. Tú (`Resource Owner`) le das al valet (`Client`) no tu llave maestra de la casa, sino una llave especial de valet.
2. Esa llave solo le permite conducir tu coche (`Resource`) y no le da acceso al maletero o a tu guantera (`Scopes`).
3. El valet usa esa llave para estacionar el coche. Si el valet intenta entrar a tu habitación de hotel con esa llave, no funcionará.
4. Tú puedes recuperar tu coche y "revocar" el acceso del valet en cualquier momento.

## Buenas Prácticas 
- **Usa `PKCE` siempre**: Incluso en aplicaciones de servidor, añade una capa extra de seguridad contra ataques de interceptación de código.
- Principio de Mínimo Privilegio: Solicita solo los `scopes` que tu aplicación realmente necesita. 
- Validación de `state`: Envía un parámetro `state` aleatorio para prevenir ataques `CSRF` durante el flujo de redirección.
- `Short-lived Tokens`: Los `Access Tokens` deben expirar rápido (minutos) y los `Refresh Tokens` deben asegurarse con máxima seguridad (ej. Base de datos cifrada).

## Notas Personales
- `OAuth 2.0` es complejo debido a su flexibilidad, pero como arquitecto, mi decisión por defecto debe ser: `Authorization Code + PKCE`.
- Nunca confundas "Loguearse con Google" con puro `OAuth`. Eso es `OIDC` (`OpenID Connect`). `OAuth` solo te dejaría ver los contactos o el `Drive`, no necesariamente saber quién es el usuario.
- Herramientas como `Auth0` o `Laravel Passport/Sanctum` abstraen mucho de este trabajo, pero entender el "baile" de los `tokens` es viral para depurar errores de seguridad.
