## Definición
La Autenticación Basada en Sesiones es una estrategia de autenticación `stateful` (con estado) en la que el servidor es responsable de crear y almacenar la información de la sesión de un usuario una vez que este se ha autenticado con éxito. A diferencia de los métodos `stateless`, el servidor "recuerda" quién es el usuario a través de un identificador único que se almacena en el lado del servidor y se comparte con el cliente mediante una `cookie`.

## Conceptos Clave
- **`Stateful` (Con estado)**: El servidor mantiene un registro activo de cada usuario conectado en su memoria, sistema de archivos o base de datos. (ej. `Redis`).
- **`Session ID`**: Un identificador único y aleatorio generado por el servidor que actúa como una "llave" para encontrar los datos del usuario en el almacenamiento de sesiones.
- **`Cookies`**: El mecanismo de transporte estándar. El servidor envía el `Session ID` en una cabecera `Set-Cookie`, y el navegador lo almacena y lo envía automáticamente en cada petición posterior al mismo dominio.
- **`Session Store`**: El lugar donde reside la información de la sesión. Puede ser:
	- **`In-Memory`**: Muy rápido pero pierde si el servidor se reinicia.
	- **`Database/Redis`**: Persistente y permite escalabilidad horizontal si varios servidores comparten la misma base de datos.
- **`Sticky Sessions`**: Una técnica de balanceo de carga que asegura que todas las peticiones de un usuario vayan al mismo servidor físico que contiene su sesión en memoria.

## Ejemplos Prácticos
##### Flujo de Trabajo (`Node/Express`):
Inicio de Sesión: El usuario envía credenciales. El servidor las valida y crea la sesión.
```javascript
// Usando express-session
app.post('/login', (req, res) => {
    if (req.body.username === 'admin' && req.body.password === '123') {
        req.session.userId = 'user_id_99'; // El servidor crea la sesión aquí
        res.redirect('/dashboard');
    }
});
```
##### Respuesta del Servidor: Envía la cabecera para fijar la `cookie`.
```http
HTTP/1.1 200 OK
Set-Cookie: connect.sid=s%3A7h9...; HttpOnly; Path=/; Secure
```
##### Acceso Protegido: El cliente envía la `cookie` automáticamente
```http
GET /dashboard HTTP/1.1
Cookie: connect.sid=s%3A7h9...
```

## Analogía
Imagina que entras a un **Club Exclusivo**:
1. En la entrada (`Login`), el portero revisa tu identificación y anota tu nombre y preferencias en un libro de visitas (`Session Store`).
2. En lugar de pedirle el `ID` cada vez que pidas un trago, te da un boleto con un número (`Session ID`).
3. Cada vez que te mueves por el club, muestras ese boleto. El personal mira el número, va al libro de visitas, ve que el número 45 pertenece a "Juan" y te sirve lo que te gusta.
4. Si te portas mal, el portero simplemente borra tu nombre del libro. Aunque conserves el boleto, ya no sirve de nada porque no hay registro que lo respalde.

## Buenas Prácticas
- **`HttpOnly & Secure Flags`**: Siempre configura tus `cookies` con `HttpOnly` (para que JavaScript no pueda leerlas y evitar `XSS`) y `Secure` (para que solo se envíen sobre `HTTPS`).
- **`SameSite Attribute`**: Usa `SameSite=Lax` o `Strict` para proteger tu aplicación contra ataques de `CSRF (Cross-Site Request Forgery)`.
- **`Use Redis`**: Para aplicaciones en producción que necesitan escalar, usa `Redis` como almacén de sesiones en lugar de la memoria del servidor.
- **`Session Expiry`**: Implementa tiempos de expiración razonables y mecanismos de "limpieza" para sesiones inactivas para no saturar el almacenamiento.
- **`Immediate Revocation`**: Aprovecha la mayor ventaja de las sesiones: la capacidad de invalidar una sesión de inmediato desde el `backend` si se detecta actividad sospechosas.

## Notas
- Este método es el estándar para aplicaciones tradicionales (monolíticas) y sitios renderizados en el servidor (`SSR`) debido a su simplicidad y al soporte nativos de los navegadores para las `cookies`.
- Aunque los `tokens` (como `JWT`) son más populares para microservicios, las sesiones ofrecen un control centralizado mucho más potente: puedes desconectar a un usuario de todos sus dispositivos instantáneamente.
- Olvidar configurar la protección contra `CSRF`. Las `cookies` se envían automáticamente, lo que las hace vulnerables si no se implementan `tokens` de seguridad adicionales o atributos `SameSite`.
