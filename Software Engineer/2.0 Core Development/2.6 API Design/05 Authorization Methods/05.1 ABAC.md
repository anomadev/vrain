## Definición
`ABAC` es un modelo de control de acceso que define los permisos basándose en atributos (características) de los sujetos, los recursos, las acciones y el entorno. A diferencia de `RBAC` (basado en roles fijos), `ABAC` utiliza lógica booleana para evaluar si una solicitud es permitida en tiempo real, lo que permite un control extremadamente fino y dinámico.

## Conceptos Clave
El poder de `ABAC` reside en la combinación de cuatro tipos de atributos:
1. `Subject Attributes (Sujeto)`: Características del usuario que intenta acceder (ej. cargo, departamento, nivel de seguridad, años de antigüedad).
2. `Resources Attributes (Recurso)`: Características del objeto al que se quiere acceder (ej. tipo de archivo, creador, etiquetas de proyecto, clasificación de sensibilidad).
3. `Action Atrributes (Acción)`: Lo que el usuario intenta hacer (ej. leer, escribir borrar, exportar).
4. `Environment Attributes (Entorno)`: El contexto de la petición (ej. dirección IP, hora del día, ubicación geográfica, dispositivo utilizado).
**Lógica de Evaluación**: Las políticas en `ABAC` funcionan como sentencias lógica. Se puede expresar matemáticamente como:
$$Permit \iff f(Attr_{subject}, Attr_{resource}, Attr_{action}, Attr_{env}) = True$$

## Ejemplos Prácticos
##### Caso de Uso en `AWS IAM`
`AWS` utiliza `ABAC` mediante el uso de `etiquetas (tags)`. Puedes crear una política que diga: "*Permitir a los desarrolladores leer y escribir en instancias de `EC2` solo si la etiqueta `Project` de la instancia coincide con la etiqueta `Project` del usuario*".
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["ec2:StartInstances", "ec2:StopInstances"],
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "aws:ResourceTag/Project": "${aws:PrincipalTag/Project}"
        }
      }
    }
  ]
}
```
##### Acceso Condicional por Entorno
"Permitir acceso a documentos financieros solo si el usuario pertenece al departamento de `Contabilidad`, el documento está marcado como `Confidencial` y la conexión proviene de la `VPN corporativa` durante el `horario laboral`.

## Analogía
Imagina un Aeropuerto Inteligente:
- `RBAC (Roles)`: Tienes un pase de "Piloto". Tu Puedes entrar a cualquier cabina de cualquier avión porque ese es tu rol.
- `ABAC (Atributos)`: Tu pase dice "Piloto". Tu pase dice que eres "Juan", tienes "Licencia Tipo A", y estás "Asignado al vuelo 402".
	- La puerta del avión solo se abre si: `(Sujeto)` tiene Licencia Tipo A + `(Recurso)` el avión es el vuelo 402 + `(Entorno)` faltan menos de 30 minutos para el despegue.
Si intentas entrar al vuelo 505, la puerta no se abre aunque seas piloto, porque los **atributos no coinciden**.

## Buenas Practicas
- **Estrategia de Etiquetado**: La efectividad de `ABAC` depende totalmente de que tus recursos y usuarios estén correctamente etiquetados. Si olvidas etiquetar un recurso, la política podría fallar o ser demasiado permisiva.
- **Evitar la Explosión de Políticas**: Diseña políticas genéricas que utilicen variables (como el ejemplos de `AWS`) en lugar de crear una política específica para cada proyecto.
- **Autoridad y Logs**: Debido a su complejidad, es vital registrar qué atributos causaron que una petición fuera denegada para poder depurar problemas de acceso rápidamente.
- **Testing**: Utiliza herramientas de simulación de políticas antes de aplicarlas en producción, ya que una lógica mal escrita puede bloquear el acceso a usuarios legítimos de forma masiva.

## Notas Personales
- `ABAC` es el futuro de la seguridad `Zero Trust`. Permite un nivel de detalle que simplemente no es posible con roles tradicionales.
- La implementación inicial es mucho más costosa y compleja que `RBAC`. Requiere una gobernanza de datos impecable. Si las etiquetas de tus usuarios en el `Active Directory` están desactualizadas, tu sistema de autorización se rompe.
- Si estas diseñando una plataforma SaaS donde los permisos pueden ser muy dinámicos, `ABAC` es la solución definitiva para evitar la "explosión de roles".
