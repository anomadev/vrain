## Definición
`RBAC` es un método de gestión de acceso que autoriza a los usuarios basándose en un rol dentro de una organización en lugar de su identidad individual. Es un modelo de control de acceso estático y jerárquico donde los permisos no se asignan directamente a las personas, sino a funciones laborales (roles), y los usuarios "heredan" esos permisos al ser asignados a dichos roles.

## Conceptos Clave
- **`Role (Rol)`**: Una colección de permisos que definen un conjunto de responsabilidades (ej. `Editor`, `Admin`, `Auditor`).
- **`Permission (Permiso)`**: La aprobación para realizar una operación específica sobre un recurso (ej. `create:invoice`, `view:reports`).
- **`User-Role Mapping`**: El proceso de asignar uno o más roles a un usuario.
- **`Hierarchy (Jerarquía)`**: Los roles pueden tener una estructura de niveles donde un rol superior (ej. `SuperAdmin`) hereda automáticamente los permisos de roles inferiores (ej. `Moderator`).
- **`Additive Model`**: Los permisos son aditivos. Si tienes dos roles, tus permisos finales son la unión de todos los permisos de ambos roles.
- **`Separation of Duties (SoD)`**: Práctica de seguridad que asegura que tareas críticas requieran más de un rol para completarse, reduciendo el riesgo de fraude o error.

## Ejemplos Prácticos
##### Definición de Roles y Permisos (Estructura `JSON/Auth0`):
```json
{
  "roles": {
    "marketing_specialist": [
      "create:newsletter",
      "edit:newsletter"
    ],
    "marketing_publisher": [
      "publish:newsletter",
      "distribute:newsletter"
    ]
  }
}
```
##### Verificación de permisos en el código (`Pseudo-code`)
```php
// English: Checking if the current user's roles include the required permission
if ($user->hasPermission('publish:newsletter')) {
    $newsletter->publish();
} else {
    throw new UnauthorizedException("You don't have the required role to publish.");
}
```

## Analogía
Imagina la producción de una Película (Sistema):
- El `Director (Rol)` tiene permiso para gritar "Acción!", cambiar el guion y entrar a la sala de edición.
- El `Actor (Rol)`tiene permisos de entrar al set y leer el guion, pero no puede editar la película.
- El `Catering (Rol)` tiene permiso para entrar al área de comida, pero no al set de grabación.
Si contratas a una persona nueva, no le explicas qué puertas puede abrir una por una; simplemente le dices: "Tu eres el `Catering`". Automáticamente, esa persona sabe que puede entra a la cocina pero no a la oficina del Director.

## Buenas Prácticas
- **Principio de Menor Privilegio (`PoLP`)**: Asigna a los usuarios el mínimo de roles necesarios para realizar su trabajo.
- **Evita la "Explosión de Roles"**: No crees un rol para cada pequeña variación. Mantén los roles genéricos y reutilizables.
- **Auditorías Regulares**: Revisa periódicamente quién tiene qué roles. En empresas grandes, es común que la gente cambie de puesto pero conserve sus roles antiguos ("`Privilege Creep`").
- **Usa Grupos para Asignación**: En lugar de asignar roles a usuarios individuales, asígnalos a grupos de tu directorio (`LDAP/AD`) para facilitar la gestión masiva.

## Notas Personales
- `RBAC` es excelente para la eficiencia operativa. Si un equipo de 100 personas necesita acceso a una nueva herramienta, solo editas 1 rol en lugar de 100 perfiles.
- `RBAC` se vuelve "pesado" cuando necesitas condiciones dinámicas (ej. *"solo puede editar si es dueño del archivo"*). Ahí es donde debes considerar un enfoque híbrido con `ABAC` ([[05.1 ABAC]]).
- Recuerda que `Kubernetes` usa `RBAC` de forma nativa para gestionar el acceso a sus recursos (`Pods`, `Nodes`, `Clusters`). Es una habilidad transferibles muy valiosa.
