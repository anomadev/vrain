El `Control de Acceso Discrecional (DAC)` es un modelo de seguridad donde el propietario de un recurso (como un archivo, carpeta o base de datos) tiene la autoridad total para decidir quién puede acceder a él y qué acciones puede realizar. Se denomina "discrecional" porque el acceso queda a discreción del usuario que creó el objeto, permitiéndole transferir permisos a otros sujetos sin intervención obligatoria de un administrador central.

## Conceptos Clave
- **Propiedad (`Ownership`)**: Cada objeto en el sistema tiene un dueño (usualmente su creador). El dueño tiene privilegios máximos sobre el objeto y puede delegar esos derechos.
- **Sujeto y Objeto**: El sujeto es quien solicita el acceso (usuario o proceso) y el objeto es el recurso protegido (archivo, dispositivo, etc.).
- **`ACL (Access Control Lists)`**: Es el mecanismo técnico más común para implementar `DAC`. Es una lista adjunta a cada objeto que detalla qué usuarios o grupos tienen permisos específicos (lectura, escritura, ejecución).
- **Descentralización**: A diferencia de otros modelos, no depende de una autoridad central para cada cambio de permiso; el control está distribuido entre los usuarios.
- **Identidad del Usuario**: El acceso se concede basándose estrictamente en la identidad verificada del sujeto tras el proceso de autenticación.

## Ejemplos Prácticos
##### Sistemas de Archivos (`Linux/Unix/Windows`)
Cuando usas el comando `chmod` en `Linux` o cambias los permisos en la pestaña de "Seguridad" en Windows, estás operando un sistema `DAC`. Como dueño del archivo, decides si tus colegas pueden solo leerlo o también editarlo.
##### Plataformas de Colaboración (Google Docs / Dropbox)
Al crear un documento y hacer clic en "Compartir", tú decides qué correos electrónicos tienen acceso y si pueden "Comentar" o "Editar". Tienes el poder de revocar ese acceso en cualquier momento.
##### Redes Sociales
En un grupo de Facebook o un álbum de fotos privado, el creador actúa como el dueño que otorga permisos de visibilidad o publicación a otros miembros de forma discrecional.

## Analogía
Imagina que eres el **Dueño de una Casa**
- Tú decides quién tiene una copia de las llaves (**Sujeto**).
- Puedes darle a un amigo una llave que solo abra la puerta principal (**Permiso de Lectura**) o una que también abra la bodega (**Permiso de Escritura**).
- Incluso puede decirle a tu amigo: "Puedes invitar a alguien más si quieres" (**Delegación de permisos**).
- El sistema de seguridad no te impide dar llaves; confía en tu discreción para proteger tu propiedad.

## Buenas Prácticas
- **Principio de Menor Privilegio**: Como dueño, otorga solo el acceso mínimo necesario (ej. dar "Lectura" en lugar de "Control Total" si solo necesitan ver un reporte).
- **Auditorías de Permisos**: Revisa periódicamente quién tiene acceso a tus recursos. En `DAC` es común el "`permission sprawl`" (acumulación de permisos olvidados).
- **Uso de Grupos**: En lugar de dar permisos usuario por usuario, crea grupos para facilitar la  gestión y reducir errores humanos.
- **Combinar con otros modelos**: Para recursos críticos, no confíes solo en `DAC`; utiliza `MAC` (`Mandatory Access Control`) o `RBAC` para imponer reglas que el usuario no pueda saltarse.

## Notas Personales
- Flexibilidad vs. Seguridad: `DAC` es el rey de la Experiencia de Usuario porque es intuitivo, pero es una pesadilla para la seguridad corporativa estricta debido al riesgo de error humano.
- Casi todas las herramientas de productividad modernas (Notion, Figma, Slack) usan una forma de `DAC`. Es esencial para la colaboración rápida.
- El mayor peligro es la "transferencia de permisos". Si le doy permiso a un usuario y él tiene permiso para delegar, el control sobre mis datos se pierde rápidamente en una cadena de favores.
