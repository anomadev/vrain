## Definición
`PBAC` es un modelo de autorización que utiliza políticas de negocio explícitas y centralizadas para determinar el acceso. A diferencia de `RBAC` (roles estáticos) o `ABAC` (atributos aislados), `PBAC` actúa como un motor de orquestación que combina roles, atributos y condiciones contextuales en reglas lógicas (ej. `if-then-else`) que viven fuera del código de la aplicación. Es el enfoque de "Autorización como Servicio".

## Conceptos Clave
Para dominar `PBAC`, es fundamental entender su arquitectura desacoplada:
- **`PEP (Policy Enforcement Point)`**: El lugar en tu aplicación donde se intercepta la petición y se "ejecuta" la restricción.
- **`PDP (Policy Decision Point)`**: El cerebro central que evalúa las políticas y responde con un simple "Permitir" o "Denegar".
- **Políticas como Código (`PaC`)**: Las reglas se escriben en lenguajes declarativos (como Rego para OPA o Polar para Oso) y se versionan en Git, igual que el código fuente.
- **Externalización**: La lógica de quién puede hacer qué ya no está enterrada en los `if` de tus controladores; está centralizada en un repositorio de políticas.
- **Gobernanza Dinámica**: Permite cambiar permisos en tiempo real sin tener que desplegar de nuevo toda la aplicación.

## Ejemplos Prácticos
##### Política de Negocio Completa
Un usuario con el rol `Analista` puede ver `Reportes Financieros` solo si pertenecen a su `Departamento` Y la solicitud realiza en `Horario Laboral` Y el dispositivo está en una `Red Segura`
##### Ejemplo en Pseudocódigo de Política (Estilo `OPA/Rego`)
```
# English: Deny access if not within business hours or untrusted network default allow = false

allow {
	input.user.role == "Analyst"
	input.resource.type == "FinantialReport"
	is_business_hours
	input.request.network == "secure"
}
```

## Analogía
Imagina que estás en un **Juzgado Moderno**:
- **`PEP` (El Alguacil)**: Está en la puerta y detiene a todos. No sabe las leyes, pero sabes a quién preguntar.
- **`PDP` (El Juez)**: Esta en su oficina con un Libro de Leyes (Políticas) actualizado.
- **El Proceso**: El alguacil le grita al juez: Puede entrar esta persona? El juez consulta el libro, mira la hora, mira el expediente de la persona y grita. Si!.
Si el congreso cambia la ley (actualiza la política), el Juez simplemente lee la página nueva. El alguacil no tiene que aprender nada nuevo; solo sigue preguntando.

## Buenas Practicas
- **Centralización**: Mantén una única fuente de verdad para tus políticas. Evita tener reglas de autorización duplicadas en diferentes microservicios.
- **Políticas Legibles por Humanos**: Escribe reglas que un gerente de producto o un auditor de seguridad pueda entender sin ser experto en código.
- **Versionado (`GitOps`)**: Trata a tus políticas como código. Cada cambio debe pasar por un `Code Review` y ser testeado antes de subir al `PDP`.
- **Principio de Mínimo Privilegio**: Por defecto, deniega todo (`default allow = false`). Solo abre acceso mediante políticas explícitas.
- **`Performance`**: Dado que cada petición requiere una consulta al `PDP`, asegúrate de que el motor de políticas sea ultra rápido y esté cerca de tus servicios (ej. como un `Sidecar Container`).

## Notas Personales
- `PBAC` es la solución definitiva al "`Role Explosion`". En lugar de crear 50 roles para cada caso de borde, creas 3 roles base y dejas que las políticas manejen las excepciones dinámicamente.
- Como arquitecto, el mayor reto es la latencia. Al desacoplar la decisión, añades un salto de red o un proceso extra. Es vital usar motores ligeros y eficientes.
- Esta arquitectura es sumamente poderosa para entornos regulados (`Fintech`, `Health`), donde los auditores necesitan ver exactamente qué reglas rigen el sistema sin leer miles de líneas de código fuente.
