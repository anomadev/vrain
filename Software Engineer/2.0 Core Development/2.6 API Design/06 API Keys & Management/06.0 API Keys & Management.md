Una `API Key` (Llave `API`) es un código único que un cliente utiliza para identificarse ante una `API`. A diferencia de los `tokens` de usuario (como `JWT`), las `API Keys` suele identificar a la aplicación o proyecto que realiza la llamada, no necesariamente a un individuo específico. El `API Key Management` es la disciplina de crear, distribuir, rotar y revocar estas llaves de manera segura para prevenir accesos no autorizados y abusos de recursos.

## Conceptos Clave
Para una gestión profesional de llaves, se deben dominar estos pilares:
- **Identificación vs. Autenticación**: Las `API Keys` se usan principalmente para identificar quién está llamando (para cuotas y monitoreo). Aunque sirven como forma simple de autenticación, son menos seguras que los protocolos de identidad (como `Auth`).
- **Ciclo de Vida (`Lifecycle`)**: Proceso que incluye la generación (creación), distribución segura, monitoreo de uso, rotación periódica y eventual revocación.
- **`Secret Management`**: Las llaves nunca deben tratarse como texto plano. Deben almacenarse en "`Secret Vaults`" (ej. `Infisical`, `HashiCorp Vault`, `AWS Secret Manager`) y no en el código fuente.
- **`Scoping (Least Privilege)`**: Una llave no debería tener acceso total. Se deben crear llaves con permisos restringidos (ej. "solo lectura" o "solo acceso a módulo de envíos").
- **`Rate Limiting por Llave`**: Capacidad de limitar el número de peticiones que una llave específica puede realizar para evitar costos excesivos o ataques.

### Diferencias: `API Keys` vs. `OAuth Tokens`
| Características | `API Keys`                             | `Auth` / `JWT Tokens`                      |
| :-------------- | -------------------------------------- | ------------------------------------------ |
| Propósito       | Identificar la aplicación/proyecto     | Identificar al usuario final               |
| Expiración      | Usualmente de larga duración           | De corta duración (minutos/horas)          |
| Seguridad       | Menor (si se filtra, el daño es menor) | Mayor (requieren refresco constante)       |
| Uso común       | Comunicación `Server-to-Server`, `SDK` | Aplicaciones móviles y web (`Client Side`) |
## Ejemplos Prácticos
##### Cabecera de envío estándar
Es la forma más segura y recomendada de enviar la llave:
```http
GET /v1/data HTTP/1.1
Host: api.example.com
X-API-Key: abc123def456ghi789
```
##### Almacenamiento Seguro (`Environment Variables`)
Nunca hagas esto: `const API_KEY = "my-secret-key";`. Haz esto:
```javascript
// .env file
STRIPE_API_KEY=sk_test_51Mz...

// Code
const stripe = require('stripe')(process.env.STRIPE_API_KEY);
```

## Analogía
Imagina que una `API Key` es la llave de un coche de una empresa de alquiler.
- No identifica quién conduce (el usuario), sino que identifica al coche (la aplicación) que tiene permiso para circular.
- Si dejas la llave pegada al cristal (la subes a `GitHub`), cualquiera puede llevarse el coche.
- El "Management" es como el llavero central de la oficina: ahí se sabe quién tiene cada llave, se pueden cambiar las cerraduras si se pierde una, y se puede limitar cuántos kilómetros puede recorrer cada coche.

## Buenas Prácticas
- **`No Hardcoding`**: Nunca subas llaves a repositorios de código. Usa archivos `.env` y asegúrate de que estén en tu `.gitignore`.
- **Rotación Periódica**: Cambia tus llaves cada 90 días (o menos en entorno críticos) para reducir el impacto si una ha sido comprometida sin que lo sepas.
- **Restricciones de `IP/Referer`**: Si la `API` lo permite, restringe la llave para que solo funcione desde las `IPs` de tus servidores o desde tus dominios específicos.
- **Monitoriza el uso**: Configura alertas para picos de tráfico inusuales. Si una llave de repente hace 10,000 peticiones en un minuto, podría ser filtrada.
- **Borrado inmediato**: Si sospechas de una filtración, revoca la llave de inmediato y genera una nueva.
