## Definición
La Seguridad de `APIs` es el conjunto de prácticas, protocolos y herramientas diseñados para proteger las interfaces de programación de aplicaciones contra ataques, abusos y filtraciones de datos. A diferencia de la seguridad web general, se centra en la lógica de autorización, el consumo de recursos y la integridad de los datos estructurados (`JSON`/`XML`).

## El `OWASP API Security Top 10`
##### `API1:2023` - `Broken Object Level Authorization` (`BOLA`)
Es la vulnerabilidad más común. Ocurre cuando un usuario puede acceder o manipular un objeto que no le pertenece simplemente cambiando el `ID` en la `URL` (ej. de `api/orders/100` a `/api/orders/101`).
**Mitigación**: Validar siempre que el usuario autenticado tenga permisos específicos sobre el objeto solicitado en la base de datos.
##### `API2:2023` - `Broken Authentication`
Debilidad en el proceso de autenticación que permiten a los atacantes comprometer `tokens` o explotar fallos en el diseño de inicio de sesión.
**Mitigación**: Usar estándares como `OAuth2`/`OIDC`, implementar `MFA` (`Multi-Factor Authentication`) y asegurar la rotación de llaves.
##### `API3:2023` - `Broken Object Property Level Authorizarion` (`BOPLA`)
Sucede cuando la `API` expone más propiedades de las necesarias o permite al usuario cambiar campos que no debería (`Mass Assignment`).
**Mitigación**: Usar `DTO` (`Data Transfer Objects`) para filtrar la entrada y salida, asegurando que solo se procesen los campos permitidos.
##### `API4:2023` - `Unrestricted Resource Consumption`
Falta de límites en el uso de recurso como `CPU`, memoria o ancho de banda, lo que puede llevar a ataques de Denegación de Servicio (`DoS`).
**Mitigación**: Implementar `Rate Limiting`, límites de tamaño de carga (`payload`) y paginación obligatoria.
##### `API5:2023` - `Broken Function Level Authorization` (`BFLA`)
Cuando un usuario con privilegios bajos puede ejecutar funciones administrativas (ej. un usuario regular accediendo a `POST /api/admin/delete-user`).
**Mitigación**: Implementar una jerarquía de roles estricta y validar permisos en cada función crítica, no solo en la interfaz.
##### `API6:2023`- `Unrestricted Access to Sensitive Business Flows`
Explotación de flujos de negocio legítimos (como la reserva de boletos o compra de productos escasos) mediante automatización masiva.
**Mitigación**: Implementar detección de `bots`, análisis de comportamiento y límites de frecuencia pro flujo de negocio.
##### `API7:2023` - `Server Side Request Forgery` (`SSRF`)
Ocurre cuando la `API` puede ser inducida a realiza peticiones hacia una `URL` proporcionada por el usuario, permitiendo acceder a servicios internos protegidos.
**Mitigación**: Validar y sanitizar todas las `URLs` de entrada, usar listas blancas de dominios permitidos.
##### `API8:2023` - `Security Misconfiguration`
Configuraciones por defecto inseguras, cabeceras `HTTP` faltantes, o mensajes de error demasiado descriptivos que revelan la infraestructura.
**Mitigación**: Endurecer la configuración de servidores (`hardening`), usar `TLS 1.3` y deshabilitar métodos `HTTP` innecesarios.
##### `API9:2023` - `Improper Inventory Management`
Tener versiones antiguas de la `API` (`V1`, `V2`) o `endpoints` de "prueba" activos que no tienen los mismos parches de seguridad.
**Mitigación**: Documentar todos los `hosts` y versiones, y aplicar políticas de retiro (`sunset`) para versiones obsoletas.
##### `API10:2023` - `Unsafe Consumption of APIs`
Confiar ciegamente en los datos recibidos de otras `APIs` de terceros.
**Mitigación**: Tratar los datos de terceros como entrada no confiable, validarlos y usar capas de transporte seguras (`TLS`).

## Principios de Seguridad Defensiva
- **`Zero Trust`**: Nunca confíes, siempre verifica cada petición, incluso si viene de la red interna.
- **`Least Privilege`**: Otorga el mínimo acceso necesario para que una función o usuario cumpla su tarea.
- **`Defense in Depth`**: Implementa seguridad en múltiples capas (`WAF`, `API Gateway`, Código, Base de Datos).
