## Definición
Las mejores prácticas de seguridad en `APIs` son un conjunto de directrices técnicas y organizativas diseñadas para mitigar riesgos durante todo el ciclo de vida de la `API`. Se basan en el principio de Defensa de Capas, donde si un control falla, otros protegen el recurso.

## Seguridad en Capa de Transporte
El primer paso es asegurar que los datos no sean interceptados en tránsito.
- **`HTTPS/TLS everywhere`**: Usa `TLS 1.3` (o `1.2` como mínimo). No permitas tráfico `HTTP` plano; redirige o bloquea inmediatamente.
- **`HSTS (HTTP Strict Transport Security)`**: Obliga al navegador a comunicarse siempre por `HTTPS`.
- **`Certificate Pinning`**: En aplicaciones móviles, asegura que la app solo confíe en el certificado específico de tu servidor, evitando ataques de interposición (`MitM`).

## Autenticación y Autorización Robusta
- **Usa Estándares (`OAuth 2.0`/ `OIDC`)**: No intentes "inventar" tu propio protocolo de seguridad. `OAuth2` es el estándar para autorización y `OpenID Connect` para identidad.
- **Principio de Menor Privilegio (`Scopes`)**: No des acceso total. Usa `scopes` (permisos granulares) para que una aplicación solo pueda hacer lo estrictamente necesario.
- **Manejo de `Tokens`**
	- Usa `JWT` solo para estados efímeros.
	- Implementa `Refresh Tokens` para mantener la sesión sin pedir credenciales constantemente.
	- Almacena secretos en bóvedas (`Vaults`), nunca en variables de entorno de texto plano o código.

## Integridad y Validación de Datos
- **Validación de Esquema**: Usa `JSON Schema` para validar que la entrada tenga el formato, tipo y longitud correctos antes de procesarla.
- **Sanitización**: Limpia los datos de entrada para evitar inyecciones `SQL` o `XSS`, especialmente si los datos se mostrarán luego en una web o se usarán en consultas.
- **`Content-Type Validation`**: Asegúrate de que el cliente envíe `application/json` (o el formato que esperas) y rechaza cualquier otro para evitar ataques de ejecución de archivos.

## Protección de Recursos (Disponibilidad)
- **`Rate Limiting & Throttling`**: Implementa límites por `IP`, por usuario y por `API Keys`. Es tu defensa principal contra ataques `DoS` y fuerza bruta.
- **Limites de Carga (`Payload Size`)**: Restringe el tamaño máximo del `JSON` de entrada para evitar que el servidor se quede sin memoria intentando `parsear` archivos gigantes.
- **Paginación Obligatoria**: Nunca permitas un `GET` que devuelva "todos" los registros. Forzar límites protege la base de datos y la red.

## Prevención de Exposición de Datos
- **Uso de `DTOs (Data Transfer Objects)`**: Filtra los datos en el código. La base de datos entrega la entidad completa, pero el `DTO` solo selecciona los campos públicos para la respuesta.
- **`Error Handling` Sanitizado**: En producción, devuelve mensajes genéricos. Nunca envíes `stack traces`, nombres de funciones o versiones de la base de datos al cliente.
- **Seguridad de Cabeceras (`CORS`)**: Configura `Cross-Origin Resource Sharing` de forma estricta. No uses `Access-Control-Allow-Origin: *` a menos que la `API` sea pública por diseño.

## Monitoreo y Auditoría
- **`Logging` con Intención**: Registra quién accedió a qué y cuándo, pero nunca guardes datos sensibles (`passwords`, `tokens`, tarjetas) en los `logs`.
- **Alertas en tiempo real**: Configura alarmas para picos de errores  `401` (posible ataque de fuerza bruta) o `403` (posible escaneo o vulnerabilidades).

### Resumen de la Capa de Protección
| **Capa**       | **Herramienta / Práctica** | **Objetivo**                       |
| :------------- | -------------------------- | ---------------------------------- |
| **Red**        | `WAF / API Gateway`        | Bloquear tráfico malicioso y `DoS` |
| **Transporte** | `TLS 1.3 + HSTS`           | Cifrar la comunicación             |
| **Identidad**  | `OAuth2 / JWT`             | Saber quien es el cliente          |
| **Aplicación** | Validación + `DTOs`        | Evitar `BOLA` y `Mass Assignment`  |
| **Operación**  | `Logging` + Auditoría      | Detectar brechas `post-evento`     |

## Notas de Seguridad
Como ingeniero, recuerda que la seguridad es un **blanco móvil**. Lo que hoy es seguro (como `TLS 1.0` hace años), mañana será una vulnerabilidad. Mantener las dependencias actualizadas y realizar pruebas de penetración (`pentesting`) periódicas es tan importante como código inicial.
- **Regla de Oro**: Trata toda entrada del usuario como hostil hasta que se demuestre lo contrario mediante validación y autorización.
