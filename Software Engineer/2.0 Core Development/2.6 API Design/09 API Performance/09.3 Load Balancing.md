## Definición
El Balanceo de Carga (`Load Balancing`) es la práctica de distribuir de manera eficiente el tráfico de red entrante entre un grupo de servidores de `backend`, también conocido como *`server farm`* o *`pool`* de servidores. Su objetivo principal es asegurar la disponibilidad, escalabilidad y confiabilidad de una aplicación o `API`, evitando que un solo servidor se convierta en un punto único de falla o se vea saturado por el exceso de peticiones.

## Conceptos Clave
- **`Scalability` (Escalabilidad Horizontal)**: Permite añadir más servidores al "`pool`" de forma sencilla para manejar el crecimiento del tráfico sin necesidad de mejorar el `hardware` de una sola máquina (escalabilidad vertical).
- **`Redundancy` (Redundancia)**: Al tener múltiples servidores, si uno falla, el balanceador redirige el tráfico a los servidores restantes sanos.
- **`Health Checks` (Validaciones de Salud)**: El balanceador realiza peticiones constantes a los servidores para verificar si están operativos. Si un servidor no responde, se marca como `offline` y deja de recibir tráfico automáticamente.
- **`Layer 4 vs Layer 7 Balancing`**:
	- **`L4 (Transporte)`**: Toma decisiones basadas en protocolos de red y transporte (`IP` y puertos `TCP`/`UDP`). Es extremadamente rápido pero no puedo ver el contenido del mensaje.
	- **`L7 (Aplicación)`**: Toma decisiones basadas en el contenido de la petición `HTTP` (`URL`, `headers`, `cookies`). Permite cosas como el `routing` basado en rutas (ej. `/api/v1/users` va a un servidor y `/api/v1/billing` a otro).
### Algoritmos de Distribución
La elección del algoritmo determina cómo se "reparte el pastel" entre tus servidores:
1. **`Round Robin`**: Las peticiones se envían de forma secuencial (1,2,3,1...). Es el más simple y asume que todos los servidores tienen la misma capacidad.
2. **`Least Connections` (Menos Conexiones)**: Envía la petición al servidor que tiene menos conexiones activas en ese momento. Ideal para procesos que tardan tiempo variables.
3. **`IP Hash`**: Utiliza la dirección `IP` del cliente para determinar qué servidor recibirá la petición. Esto asegura que un usuario específico siempre (o casi siempre) llegue al mismo servidor, útil para mantener estados locales o persistencia.
4. **`Weighted` (Ponderado)**: Permite asignar "pesos" a los servidores. Si el `Server A` es el doble de potente que el `B`, recibirá el doble de tráfico.
### Diferencia Crítica: `API Gateway` vs. `Load Balancer`
Es común confundirlos en una arquitectura moderna, pero cumplen funciones distintas:
- **`Load Balancer`**: Su enfoque es puramente la **disponibilidad y el rendimiento**. Se encarga de que ninguna máquina muera bajo el paso del tráfico.
- **`API Gateway`**: Su enfoque es la gestión y seguridad. Maneja autenticación, transformación de datos, `rate limiting` y agregación de microservicios.
En arquitecturas complejas, el `Load Balancer` suele estar delante del `API Gateway`, o el `API Gateway` tiene funciones integradas del balanceo.

## Analogía
Imagina un Restaurante muy concurrido (Tu Sistema):
- **Sin Balanceador**: Los clientes hacen una sola fila larga. El único mesero disponible se estresa y tarda 20 minutos en tomar cada orden. Si el mesero se enferma, el restaurante cierra.
- **Con Balanceador (El Host/Anfitrión)**: Hay un anfitrión en la puerta. A medida que llegan los clientes, él los dirige a diferentes mesas donde hay **múltiples meseros (Servidores)**. Si un mesero está muy ocupado, el anfitrión envía al siguiente cliente a un mesero libre. Si un mesero se va a su descanso, el anfitrión simplemente deja de enviarle gente.

## Análisis Técnico
Al planificar la refactorización de una plataforma SaaS o el lanzamiento de un nuevo proyecto, el balanceo de carga debe considerarse desde el diseño:
- **`Session Persistence (Sticky Sessions)`**: Si tu aplicación guarda datos en la memoria local del servidor (mal diseño para escalar, pero común en apps `legacy`), necesitarás activar `sticky sessions`. Sin embargo, lo ideal es mover el estado a una DB o `Redis` para que cualquier servidor pueda atender a cualquier usuario sin problemas.
- **`SSL Termination`**: Configura el balanceador para que maneje el cifrado `SSL`/`TLS`. Esto libera de esa carga computacional a tus servidores de aplicación, permitiéndoles concentrarse únicamente de ejecutar el código (`PHP`, `Go`, `Pytho`, etc.).
- **`Global Load Balancing (GSLB)`**: Si tus usuarios están en diferentes continentes, considera balanceadores geográficos que dirijan al usuario al centro de datos más cercano a su ubicación física.
