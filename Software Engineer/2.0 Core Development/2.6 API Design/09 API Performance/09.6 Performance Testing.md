## Definición
El `Performance Testing` (Pruebas de Rendimiento) es una práctica de control de calidad que busca determinar cómo se comporta un sistema en términos de capacidad de respuesta, estabilidad y escalabilidad bajo una carga de trabajo particular. A diferencia del `Monitoring` (que es observación continua en producción) y el `Profiling` (que es análisis interno del código), el `Testing` es un esfuerzo deliberado y controlado para encontrar los límites del sistema antes de que los usuarios reales lo hagan.

## Tipos de Pruebas de Rendimiento
Para dominar el ciclo de vida de una `API`, debes conocer estas cuatro variantes:
1. **`Load Testing` (Pruebas de Carga)**: Evalúa el comportamiento de la `API` bajo el volumen de tráfico esperado (ej. 100 usuarios simultáneos). El objetivo es verificar si se cumplen los `SLAs` de latencia.
2. **`Stress Testing` (Pruebas de Estrés)**: Empuja la `API` más allá de su capacidad normal para ver en qué punto se rompe. Ayuda a entender cómo falla el sistema (falla con elegancia o colapsa por completo?).
3. **`Spike Testing` (Pruebas de Picos)**: Observa cómo reacciona la `API` ante aumentos repentinos y masivos de tráfico (ej. el lanzamiento de una oferta relámpago).
4. **`Soak/Endurance Testing` (Pruebas de Resistencia)**: Mantiene una carga constante durante un largo periodo (horas o días) para detectar fugas de memoria o saturación de recursos que no aparecen en pruebas cortas.

### El Ecosistema: `Testing` vs. `Monitoring` vs. `Profiling`
Para un arquitecto, estas tres disciplinas se alimentan entre sí:

| Disciplina            | Momento                         | Objetivo Principal                   | Herramienta Ejemplo                  |
| :-------------------- | ------------------------------- | ------------------------------------ | ------------------------------------ |
| `Performance Testing` | Antes de Desplegar (`Pre-prod`) | Encontrar el límite de ruptura       | `K6`, `JMeter`, `Locust`             |
| `API Monitoring`      | En Producción (Continuo)        | Detectar caídas o degradación        | `Postman Monitors`, `Datadog`        |
| `API Profiling`       | Durante el `Test` o `Debugging` | Identificar la línea de código lenta | `Xdebug`, `Go Proof`, `Flame Graphs` |
### `Insights` de las fuentes
##### `Postman Monitors` (Pruebas Sintéticas)
`Postman` permite transformar tus colecciones de pruebas funcionales en monitores de rendimiento.
- **`Continuous Validation`**: Ejecuta pruebas de latencia y veracidad cada pocos minutos.
- **`Global Perspective`**: Permite ver si el rendimiento varía según la región geográfica, lo cual es vital para `APIs` distribuidas.
##### `Pinterest Engineering` (`Profiling` bajo Carga)
`Pinterest` destaca que el `testing` por sí solo no basta. Cuando una prueba de carga falla, el `Profiling` entra en acción:
- **`Flame Graphs`**: Permiten visualizar dónde se "atasca" la `CPU` durante una prueba de estrés.
- **`Contextual Data`**: No solo miden que la `API` es lenta, sino que identifican si es debido a la serialización de datos, la lógica de negocio o la latencia de la base de datos.

## Analogía
Imagina que estás diseñando un **Puente (tu `API`)**
- `Performance Testing`: Es pasar camiones de 40 toneladas sobre el puente antes de inaugurarlo para ver cuánto peso soporta antes de agrietarse.
- `Monitoring`: Es tener sensores en el puente 24 horas para avisar si hay demasiada vibración o si el tráfico se detiene.
- `Profiling`: Es enviar a un ingeniero debajo del puente con una lupa para ver exactamente qué viga se está doblando bajo el peso de los camiones.

## Buenas Prácticas
- **`Think in Percentiles`**: No te fíes de los promedios. En `performance testing`, el `P99` (el tiempo de respuesta del 1% de los usuarios más lentos) es la métrica que define la insatisfacción del cliente.
- **`Test in a Production-like Environment`**: Probar en tu laptop no sirve. El entorno de pruebas debe tener la misma configuración de `CPU`, `RAM` y latencia de red que el entorno real.
- **`Automate in CI/CD`**: El rendimiento es una regresión. Si una nueva función hace que la `API` pase de 100ms a 500ms, el `pipeline` de integración continua debería marcar un error.
- **`Warm up the system`**: Antes de medir, realiza unas cuantas peticiones para "calentar" el caché y la conexión a la base de datos, evitando falsos positivos de lentitud inicial.
