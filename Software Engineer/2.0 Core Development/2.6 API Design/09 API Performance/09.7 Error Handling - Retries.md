## Definición
El Manejo de Errores y Reintentos (`Retries`) es una estrategia fundamental para construir sistemas resilientes que puedas recuperarse de fallos transitorios. Un fallo transitorio es un error temporal (como un parpadeo en la red, un tiempo de espera en la base de datos o un límite de tasa excedido) que probablemente se resolverá solo si se intenta de nuevo unos segundos después.

### Estrategias de Resiliencia
Para evitar que un error puntual arruine la experiencia del usuario, se utilizan tres conceptos clave:
##### `Timeouts` (Tiempos de Espera)
Establecer un límite de tiempo para cada petición. Si el servidor no responde en, por ejemplo, t segundos, la petición se cancela. Esto evita que los recursos del cliente (hijos, memoria) se queden bloqueados indefinidamente esperando una respuesta que nunca llegará.
##### `Retries` con `Exponential Backoff`
No todos los errores deben reintentarse. Solo se deben reintentar aquellos que son recuperables (ej. `503 Service Unavailable`, `504 Gateway Timeout`, `429 Too Many Requests`).
- **`Exponential Backoff`**: En lugar de intentar inmediatamente, se añada un retraso que crece exponencialmente con cada fallo (2s, 4s, 8s, 16s...). Esto da tiempo al servicio de destino para recuperarse si está saturado.
##### `Jitter` (Aleatoriedad)
Si mil clientes fallan al mismo tiempo y todos usan exactamente el mismo algoritmo de reintento, todos volverán a atacar al servidor al mismo tiempo después de 2 segundos, causando el efecto de `"Thundering Herd"` (manada estruendosa).
- **Solución**: Añadir un factor aleatorio al tiempo de espera (ej. en lugar de 2s, esperar entre 1.8s y 2.2s). Esto distribuye las peticiones en el tiempo y facilita la recuperación del servidor.
### El `Circuit Breaker` (Corta-fuegos)
Cuando un servicio está caído de forma prolongada, los reintentos constantes pueden empeorar la situación (ataque de denegación de servicio accidental).
- **Funcionamiento**: Si la tasa de errores cruza un umbral (ej. 50% de fallos), el "circuito se abre" y todas las peticiones fallan inmediatamente sin intentar llamar al servidor. Después de un tiempo, el circuito entra en "medio abierto" para probar si el servicio ya es estable.
### Idempotencia: La Regla de Oro
`Retrying` sin idempotencia es peligroso. Si reintentas un `POST /pagos` porque hubo un `timeout`, corres el riesgo de cobrarle dos veces al usuario si la primera petición llegó al servidor pero la respuesta se perdió en el camino.
- **Solución**: Usar `Idempotency Keys` (un `UUID` único enviado a los `headers`). El servidor guarda el resultado del primer intento y, si recibe la misma llave, devuelve el resultado guardado un lugar de procesar el pago otra vez.

## Ejemplo Práctico (`Node`/`Promises`)
Usar librerías como `promise-retry` permite configurar estas reglas de forma declarativas:
```javascript
// [00:05:55] Configuración de reintentos con backoff
const promiseRetry = require('promise-retry');

promiseRetry((retry, number) => {
    return fetch('https://api.openai.com/v1/images/generations')
        .then(response => {
            if (response.status === 429) { // Rate limit [00:06:29]
                retry(new Error('Rate limit exceeded'));
            }
            return response.json();
        });
}, {
    retries: 5,
    minTimeout: 2000,
    factor: 2 // Crecimiento exponencial
});
```
