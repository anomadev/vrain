Los Patrones de Integración de `APIs` son los planos arquitectónicos que dictan cómo los diferentes sistemas se comunican entre sí. No se trata solo de hacer una llamada `GET`; se trata de diseñar un flujo que sea resiliente, escalable y fácil de mantener.

## Patrones de Integración de `APIs`
### Patrón Agregador (`Aggregation`)
Es uno de los más comunes en arquitecturas de microservicios. El cliente realiza una única petición a un `endpoint`, y este se encarga de llamar a múltiples servicios internos, combinar sus respuestas y devolver un único objeto `JSON` consolidado.
- Problema: El cliente (`frontend`) tiene que hacer demasiadas llamadas para mostrar una sola pantalla.
- Solución: Un "orquestador" centraliza las llamadas.
- Beneficio: Reduce la latencia de red y el consumo de batería en dispositivos móviles.
### Patrón de Encadenamiento (`Chainig`)
En este patrón, la salida de la `API A` se utiliza como entrada para la `API B`, y así sucesivamente. Es un flujo secuencial y dependiente.
- Uso común: Procesos de `checkout` donde primero validas el `stock`, luego procesas el pago y finalmente generas la orden de envío.
- Riesgo: Si un eslabón de la cadena falla o es lento, toda la transacción se detiene. Se recomienda implementar `timeouts` agresivos.
### Patrón `Proxy` / `Gateway`
Funciona como una capa intermedia entre el cliente y los servicios. El `Gateway` no solo redirige el tráfico, sino que añade funcionalidades transversales.
- Funciones: Autenticación, `Rate Limiting`, Transformación de protocolos (ej. de `XML` a `JSON`) y `logging`.
- Ventaja: Los servicios internos no necesitan preocuparse por la seguridad o el tráfico; el `Gateway` es el escudo.
### Patrón Basado en Eventos (`Webhooks` / `Pub-Sub`)
En lugar de que el cliente pregunte constantemente "ya terminó?", el servidor notifica al cliente cuando ocurre un evento relevante.
- `Webhooks`: El servidor hace un `POST` a una `URL` del cliente cuando el evento sucede.
- `Pub/Sub`: Los sistemas se suscriben a "tópicos". Cuando hay datos nuevos, el sistema de mensajería (como `RabbitMQ` o `Kafka`) los distribuye.
### Patrón de Sondeo Asíncrono (`Async Polling`)
Ideal para tareas que tardan mucho tiempo (generación de reportes masivos, procesamiento de video).
1. El cliente envía la petición
2. El servidor responde inmediatamente con un `202 Accepted` y un `Job ID`
3. El cliente pregunta (`poll`) periódicamente a un `endpoint` de estado hasta que la tarea termina.
### Comparativa de Patrones
| Patrón         | Tipo de Comunicación  | Mejor para...                               |
| :------------- | --------------------- | ------------------------------------------- |
| Agregador      | Síncrona              | Unificar datos de múltiples fuentes         |
| Encadenamiento | Síncrona / Secuencial | Procesos de negocio con pasos lógicos       |
| `Webhooks`     | Asíncrona (`Push`)    | Notificaciones en tiempo real sin `polling` |
| `Async Pollig` | Asíncrona (`Pull`)    | Tareas pesadas que bloquean el servidor     |
### El Consejo del Arquitecto
Al elegir un patrón recuerda la Ley de Postel: "Sé conservador en lo que envías y liberal en lo que aceptas". En integraciones complejas, el Patrón de `Circuit Breaker` es tu mejor amigo; si la `API B` esta caída, no permitas que la `API A` siga intentando llamarla y agote sus propios recursos.
