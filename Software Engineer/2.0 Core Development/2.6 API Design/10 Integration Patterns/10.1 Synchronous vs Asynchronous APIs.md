### APIs Síncronas (`Sync`)
Es el modelo de "solicitud y respuesta" más tradicional. El cliente envía una petición al servidor y se queda bloqueado (esperando) hasta que el servidor procesa la solicitud y devuelve un resultado.
- **Protocolo común**: `REST`, `GraphQL`, `gRPC`
- **Estado**: Bloqueante (`Blocking`)
- **Ideal para**: Operaciones rápidas y simples (ej. consultar el saldo de una cuenta, validar un `login`)
**Ventajas y Desventajas**
- **Pro**: Simplicidad. El flujo de código es lineal y fácil de seguir/depurar.
- **Con**: Si el servidor tarda mucho, el cliente puede experimentar un `timeout` o dejar la interfaz de usuario congelada. No escala bien con procesos pesados.
### APIs Asíncronas (`Async`)
En este modelo, el cliente envía la petición y el servidor responde inmediatamente con un "recibido, estoy trabajando en ello" (usualmente un código `202 Accepted`). El cliente queda libre para seguir con otras tareas mientras el servidor procesa el trabajo en segundo plano.
- **Protocolo común**: `Webhooks`, `WebSockets`, colas de mensajes (`RabbitMQ`, `Kafka`)
- **Estado**: No bloqueante (`Non-blocking`)
- **Ideal para**: Tareas de larga duración (ej. generar un PDF masivo, procesar un video, enviar 10,000 correos).
### Patrones de Comunicación
Para recuperar el resultado de una tarea asíncrona, se suelen usar tres métodos principales:
##### `Polling` (Sondeo)
El cliente pregunta periódicamente al servidor: Ya terminaste?. El servidor responde "No" hasta que el recurso está listo.
- **Punto Crítico**: Es ineficiente para la red si se hace con demasiada frecuencia, pero es más fácil de implementar si el cliente no puede recibir conexiones entrantes.
##### `Webhooks` (`Callbacks`)
El cliente le da al servidor una `URL` de retorno. Cuando el servidor termina, él mismo realizar una petición `POST` a esa `URL` con el resultado.
- **Punto Crítico**: Requiere que el cliente tenga un `endpoint` público y gestione la seguridad para validar que la notificación realmente viene del servidor.
##### `Pub/Sub` (Publicación/Suscripción)
El servidor publica un mensaje en un "tópico" o bus de eventos. Cualquier sistema interesado que está suscrito recibirá la información instantáneamente.
- **Punto Crítico**: Excelente para arquitecturas de microservicios altamente desacopladas.
#### Comparativa Técnica

| Característica | Síncrono                      | Asíncrono                                 |
| :------------- | ----------------------------- | ----------------------------------------- |
| **Bloqueo**        | Sí (El cliente espera)        | No (El cliente sigue trabajando)          |
| **Código `HTTP`**  | `200 OK`, `201 Created`       | `202 Accepted`                            |
| **Latencia**       | Crítica (debe ser baja)       | Menos crítica (el tiempo es variable)     |
| **Complejidad**    | Baja                          | Alta (Requiere manejo de estados y colas) |
| **Escalabilidad**  | Limitada por hilos/conexiones | Alta (ideal para sistemas distribuidos)   |
## Analogía
Imagina que estás en un **Restaurante**:
- Síncrono: Pides un café en la barra y te quedas parado frente al barista hasta que te lo entrega. No puedes hacer nada más; solo esperar tu café.
- Asíncrono: Pides una pizza. Te dan un `beeper` (`Token/Job ID`) y te vas a sentar. Puedes hablar con tus amigos o leer. Cuando la pizza está lista, el `beeper` vibra (`Wehook`/Evento) y vas a recogerla.
### Cuándo elegir cual?
- **Usa Síncrono si**: La respuesta es necesaria para el siguiente paso inmediato del usuario y el proceso tarda menos de 1-2 segundos.
- **Usa Asíncrono si**: El proceso es impredecible, requiere mucha `CPU`/`IO`, o involucra a terceros (`APIs` externas) que podrían fallar o tardar.
