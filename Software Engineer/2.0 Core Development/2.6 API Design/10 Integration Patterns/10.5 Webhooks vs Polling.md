En la arquitectura de integración, la forma en que los sistemas "se enteran" de los cambios es viral para el rendimiento y la experiencia del usuario. La elección se reduce a dos filosofías opuestas: el cliente preguntando constantemente (`Polling`) o el servidor avisando cuando algo sucede (`Webhooks`).

### `Polling` (El modelo `Pull`)
El `Polling` es el mecanismo donde el cliente realiza peticiones periódicas al servidor para verificar si hay nuevos datos o si un proceso ha cambiado de estado.
- **Funcionamiento**: El cliente envía una petición `GET` cada $X$ segundos/minutos. El servidor responde con los datos actuales o un mensaje vacío si no hay novedades.
- **Estado**: Síncrono y orientado al cliente.
##### Pros y Contras
- **Ventajas**: Muy fácil de implementar; no requiere que el cliente tenga una `IP` pública o un servidor escuchando; ideal para clientes detrás de `firewalls` estrictos.
- **Desventajas**: Ineficiente (genera muchas peticiones vacías que consumen `CPU` y ancho de banda); introduce latencia (la información solo llega en la siguiente "vuelta" del reloj.
### `Webhooks` (El modelo `Push`)
Un `webhook` es una "`API` a la inversa". Es una petición `HTTP` (usualmente `POST`) enviada por un servidor a una `URL` específica proporcionada por el cliente, disparada por un evento concreto.
- **Funcionalidades**: El cliente registra una `URL` de `"callback"`. Cuando ocurre el evento (ej. pago completado), el servidor envía los datos inmediatamente a esa `URL`. 
- **Estado**: Asíncrono y orientado al evento.
##### Pros y Contras
- **Ventajas**: Tiempo real (baja latencia); extremadamente eficiente (solo hay tráfico cuando hay datos); reduce la carga en el servidor principal.
- **Desventajas**: Más complejo de asegurar; requiere que el cliente tenga un `endpoint` accesible desde internet; el servidor debe manejar reintentos si el cliente está caído.
### Comparativa Técnica

| Característica     | `Polling`                             | `Webhook`                               |
| :----------------- | ------------------------------------- | --------------------------------------- |
| Iniciador          | Cliente                               | Servidor                                |
| Carga del Servidor | Alta (muchas peticiones innecesarias) | Baja (solo una petición por evento)     |
| Latencia           | Variable (depende del intervalo)      | Mínima (tiempo real)                    |
| Complejidad        | Baja                                  | Media (requiere manejo de seguridad)    |
| Accesibilidad      | Funciona en cualquier cliente         | El cliente debe ser un servidor público |
### Cuando elegir cuál?
##### Usa `Polling` si:
- **Datos de cambio constante**: La información cambia tan seguido que una conexión abierta o notificaciones constantes saturarían el sistema.
- **Restricciones de red**: El cliente no puede recibir conexiones entrantes (está detrás de un `NAT` O `Firewall`).
- **Procesos cortos**: El tiempo de espera es predecible y muy breve (ej. menos de 5 segundos).
##### Usa `Webhook` si:
- **Eventos esporádicos**: Los cambios ocurren de forma irregular (ej. un usuario se suscribe, un pago se procesa).
- **Necesidad de Tiempo Real**: El usuario necesita la información al instante (notificaciones `push`, actualizaciones de `stock`).
- **Escalabilidad**: Quieres evitar que miles de clientes saturen tu `API` con peticiones vacías.
### El punto Crítico: Seguridad en `Webhooks`
A diferencia del `polling`, donde el cliente confía en el servidor al que llama, en los `webhooks` el cliente debe verificar que quien lo llama es realmente el servidor esperado.
- **Secretos Compartidos**: El servidor firma el `payload` con una clave secreta; el cliente verifica la firma antes de procesar.
- **Verificación de `IP`**: El cliente solo acepta peticiones de una lista blanca de `IPs` del proveedor.
- **Idempotencia**: Dedo que el servidor puede reintentar el envío si falla, el cliente debe estar listo para recibir la misma notificación dos veces sin duplicar la lógica de negocio.

## Analogía
Imagina que estás esperando un Paquete de Amazon:
- `Polling`: Vas a la puerta de tu casa cada 10 minutos para ver si el camión ya llegó. Si el camión tarda 5 horas, habrás caminado a la puerta 30 veces para nada.
- `Webhooks`: Te quedas sentado en tu sofá. El repartidor toca el timbre cuando llega. Solo te levantas cuando la acción realmente sucede.
