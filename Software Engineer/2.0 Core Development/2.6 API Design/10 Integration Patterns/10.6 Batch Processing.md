## Definición
El `Batch Processing` (Procesamiento por Lotes) es un patrón de diseño donde una `API` permite al cliente enviar múltiples registros u operaciones en una única petición `HTTP`, en lugar de realizar llamadas individuales para cada elemento. Este enfoque está diseñado para manejar grandes volúmenes de datos de manera eficiente, reduciendo drásticamente el `"overhead"` de red y optimizando el uso de recursos del servidor y la base de datos.

### `Batch` vs `Bulk` Cuál es la diferencia?
Aunque a menudo se usan como sinónimos, tienen matices técnicos:
- **`Bulk Operations`**: Se refiere al procesamiento de un gran volumen de datos del mismo tipo (ej. importar 50,000 contactos en un solo `POST /bulk/contacts`).
- **`Batch Operations`**: Se refiere a la ejecución de múltiples operaciones que pueden ser de diferente tipo o sobre diferentes recursos en una sola llamada (ej. crear un usuario, actualizar un pedido y eliminar un comentario en un solo `POST /batch`).
### Patrones de Implementación
##### `Synchronous Batch` (Síncrono)
El cliente envía el lote y espera a que el servidor procese todos los elementos antes de recibir una respuesta.
- **Ideal para**: Lotes pequeños (ej. 10-50 elementos) donde el tiempo de procesamiento es predecible.
- **Riesgos**: Si el lote es muy grande, la conexión puede expirar (`timeout`) o bloquear hilos del servidor innecesariamente.
##### `Asynchronous Batch` (Asíncrono)
El servidor recibe el lote, valida la sintaxis básica y responde inmediatamente con un código `202 Accepted` y un `Job ID`. El procesamiento real ocurre en segundo plano.
- **Ideal para**: Procesos pesados o importaciones masivas (ej. miles de registros)
- **Seguimiento**: El cliente debe consultar un `endpoint` de estado (`GET /jobs/{id}`) o esperar un `Webhook` para saber cuándo terminó el proceso.
### Consideraciones Críticas de Diseño
##### Manejo de Errores y `"Partials Success"`
Qué sucede si envías 100 registros y el número 42 falla? Hay dos filosofías:
1. **`All-or-Nothing` (Atómico)**: Si un solo elemento falla, se revierte toda la operación (`rollback`). Útil para transacciones financieras.
2. **`Partial Succes` (Éxito Parcial)**: El servidor procesa lo que puede y devuelve un reporte detallado indicando qué elementos fallaron y por qué.
	- Código `HTTP`: Se suele usar `207 Multi-Status`.
##### Límites de Tamaño (`Payload Limits`)
Es vital establecer límites máximos para evitar que un solo lote agote la memoria del servidor.
- **Límite por registros**: Ej. máximo 1,000 elementos por lote.
- **Límite por tamaño de archivo**: Ej. máximo 10MB por petición.
##### Idempotencia
Es el procesamiento por lotes, la probabilidad de reintentos por fallos de red es alta. Se deben usar llaves de idempotencia para asegurar que si un lote se envía dos veces, no se dupliquen los datos en la base de datos.
### Ventajas Técnicas
- **Reducción de Latencia**: Menos `"handshakes"` de `TCP/TLS`
- **Transacciones Optimizadas**: Permite realizar una sola apertura y cierre de transacción en la base de datos para múltiples inserciones.
- **Control de Tráfico**: Es más fácil para el servidor gestionar un solo proceso pesado que miles de pequeñas peticiones concurrentes.

## Analogía
Imagina que eres un Mensajero (Cliente) entregando paquetes en una Oficinal Postal (Servidor):
- Sin `Batch`: Vas a la oficina postal por cada carta que tiene que enviar. Caminas 100 veces.
- Con `Batch`: Pones las 100 cartas en una sola saca y haces un solo viaje. El empleado postal recibe la saca y, si una carta tiene mal la dirección, te devuelve esa específicamente pero envía las otras 99 (`Partial Success`).
