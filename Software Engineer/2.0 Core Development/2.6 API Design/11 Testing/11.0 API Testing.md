A diferencia de las pruebas de interfaz de usuario (UI), que se centran en el aspecto y la sensación de una aplicación, el `API Testing` se encarga de validar la lógica de negocio, la seguridad y la integridad de los datos en la capa de comunicación. Es probar el "motor" del coche sin importar el color de la pintura.

### Tipos de Pruebas de `API` 
Para asegurar que tu sistema sea robusto, no basta con recibir un `200 OK`. Debes cubrir diferentes ángulos:
- **Pruebas Funcionales**: Verifican que la `API` haga lo que se supone que debe hacer (ej. si pido un usuario, que me devuelva el objeto correcto).
- **Pruebas de Carga (`Load Testing`)**: Evalúan cómo se comporta la `API` cuando miles de usuarios la llaman simultáneamente.
- **Pruebas de Seguridad**: Buscan vulnerabilidades como inyecciones `SQL`, fallos en la autenticación (`JWT`) o exposición de datos sensibles.
- **Pruebas de Integración**: Aseguran que la `API` se comunique correctamente con otros servicios o bases de datos.
- **Pruebas de Negativo**: Envían datos basura o mal formateados para ver si la `API` los maneja con elegancia (ej. errores `400` o `422`) en lugar de colapsar (`500`).

### Anatomía de una Validación (`Assertion`)
Cuando pruebas un `endpoint`, tu `"checklist"` mental (y automatizado) debería verse así:

| Elemento           | Qué validar                                                          |
| :----------------- | -------------------------------------------------------------------- |
| `Status Code`      | Es correcto? (`201` para creación, `404` para no encontrado, etc.)   |
| `Response Body`    | El esquema `JSON` es correcto? Los datos son los esperados?          |
| `Response Headers` | El `Content-Type` es `application/json`? Hay `headers` de seguridad? |
| `Response Time`    | La respuesta llega en un tiempo aceptable (ej. <200ms)?              |

### El Flujo de Automatización
En un entorno profesional, las pruebas no se hacen a mano cada vez. Se utiliza el enfoque `Documentation-as-Tests`:
1. **Colecciones**: Agrupas tus peticiones en herramientas como `Postman`.
2. **`Scripts de Pre-solicitud`**: Generas `tokens` o datos dinámicos antes de la llamada.
3. **`Test/Assertions`**: Escribes `scripts` en `JavaScript` para validar la respuesta automáticamente.
4. **`Newman`**: Usas la versión `CLI` de `Postman` para ejecutar estas pruebas en tu `pipeline` de `CI/CD` (`GitHub Actions`, `GitLab`, etc.).
**Punto Crítico**: Muchos desarrolladores cometen el error de proba solo el `"Happy Path"`. La verdadera calidad de una `API` se demuestra en cómo maneja los errores. Si tu `API` devuelve un `200 OK` con un cuerpo que dice `{"error": "not found"}`, estás rompiendo el estándar `REST` y confundiendo a quien consuma tu servicio.

### Herramientas Recomendadas
- **`Postman`**: La navaja suiza para diseño, documentación y `testing`
- **`Testsigma`**: Ideal para pruebas automatizadas basadas en `IA` y sin código (`Low-code`)
- **`RestAssured`**: Para los amantas de `Java` que prefieren escribir `tests` en código puro
- **`K6 / JMetter`**: Especializadas en pruebas de carga y estrés
