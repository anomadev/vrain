## Definición
El `Unit Testing` (Pruebas Unitarias) en el contexto de una `API REST` consiste en verificar el funcionamiento de lar partes más pequeñas y aisladas del código que maneja la `API`. A diferencia de las pruebas de integración (que prueban cómo se comunican las partes), el `unit testing` se enfoca en la lógica interna de un controlador, un servicio o un modelo, asegurando que ante una entrada específica, se produzca la salida esperada sin depender de factores externos como la base de datos o servicios de terceros.

### Los Tres Pilares del `Unit Testing`
Para que una prueba sea considerada "unitaria", debe cumplir con estas tres reglas:
1. **Aislamiento Total**: Si la prueba falla, debe ser porque la lógica del código tiene un error, no porque la base de datos está caída. Para lograr esto, se utilizan `Mocks` (objetos simulados que reemplazan las dependencias reales).
2. **Velocidad**: Deben ejecutarse en milisegundos. Un proyecto con miles de `Unit Tests` debería completarse en pocos segundos.
3. **Determinismo**: Una prueba unitaria debe pasar el 100% de las veces si el código no cambia. No puede depender de la hora del sistema o de una conexión a internet.

### `Unit Testing` vs `Integration Testing`

| Característica | `Unit Testing`            | `Integration Testing`                   |
| :------------- | ------------------------- | --------------------------------------- |
| Alcance        | Una sola función o clase  | Múltiples componentes trabajando juntos |
| Dependencias   | Simuladas (`Mocks/Stubs`) | Reales (Base de datos, red, archivos)   |
| Velocidad      | Ultra rápida              | Más lenta                               |
| Dificultad     | Baja (fácil de aislar)    | Alta (difícil de configurar el entorno) |

### Qué probar en una API (`Checklist Unitario`)
Al probar un componente de tu `API` (como un controlador), debes validar:
- **Lógica de Negocio**: Si el usuario tiene `X` puntos, se le aplica el descuento `Y` correctamente?
- **Códigos de Estado**: Si el servicio devuelve una excepción, el controlador responde con un `400` o `500` específico?
- **Validación de Entrada**: El código rechaza correctamente un `JSON` mal formado antes de procesarlo?
- **Manejo de Errores**: Se capturan la excepciones y se devuelven mensajes de error controlados?

### La Técnica Maestra: `Mocking`
Dado que no queremos tocar la base de datos, usamos `Mocking`.
- **Ejemplo**: Si tu controlador llama a un `PaymetGateway` para procesar una tarjeta, en tu `Unit Testing` no llamarás a `Stripe` o `Paypal`. Crearás un "Simulacro" que simplemente diga: "Sí, el pago fue exitoso" o "No, fondos insuficientes". Eso te permite probar cómo reacciona tu código ante ambos escenarios sin gastar un centavo ni esperar a la red.

## Analogía
Imagina que eres un Inspector de Calidad en una fábrica de smartphones:
- `Unit Testing`: Sacas un solo microchip de la línea de montaje, lo conectas a una máquina de pruebas y verificas que emita la señal eléctrica correcta. No te importa si la pantalla o la batería funcionan; solo te centras en ese chip.
- `Integration Testing`: Montas el teléfono completo y verificas que al presionar el botón de encendido, la pantalla se ilumine y el sistema cargue. Aquí pruebas que todas las piezas encajen y se hablen entre sí.

## Notas Personales
Muchos desarrolladores escriben "`Integration Tests`" pensando que son "`Unit Tests`". Si tu prueba requiere un archivo `.env` configurado con una base de datos real para pasar, **no es un `unit test`**. El `unit testing` es tu primera línea de defensa; si tus unidades son sólidas, tu sistema será mucho más fácil de depurar.
