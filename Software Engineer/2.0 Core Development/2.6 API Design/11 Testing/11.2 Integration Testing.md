## Definición
El `Integration Testing` (Pruebas de Integración) de `APIs` es el proceso de verificar que diferentes módulos o servicios externos de una aplicación funcionen correctamente al interactuar entre sí. Mientras que el `Unit Testing` se asegura de que una función aislada devuelve el valor correcto, el `Integration Testing` se asegura de que el flujo de datos entre el controlador, la base de datos, el sistema de caché y las `APIs` de terceros sea coherente y exitoso.

### Por qué es Vital?
En un entorno de microservicios o aplicaciones modernas, los componentes rara vez fallan de forma aislada. Los problemas suelen surgir en las "costuras".
- **Desajustes de Esquema**: El servicio `A` espera un `string` pero el servicio `B` envía un `integer`.
- **Fallos de Autenticación**: El `token` generado por el `AuthService` no es reconocido por el `InventoryService`.
- **`Side Effects` (Efectos Secundarios)**: Una llamada a la `API` devuelve `200 OK`, pero no guardó los datos en la base de datos ni envió el correo de confirmación.

### Qué se Prueba en una Integración
Para un ingeniero de software, el enfoque debe estar en los "puntos de contacto".
1. **Persistencia de Datos**: La petición `POST` realmente creó un registro en la base de datos con los atributos correctos?
2. **Consumo de `APIs` Externas**: Nuestra `API` maneja correctamente los tiempos de espera (`timeouts`) o errores de `APIs` como `Stripe`, `Twilio` o `Maps`?
3. **Seguridad y Flujo de `Auth`**: Se puede acceder a un recurso protegido después de un `login` exitoso?
4. **Estado del Sistema**: Si la `API` usa `Redis` para caché, se actualiza la caché después de una mutación de datos?

### Enfoques de Pruebas de Integración

| Enfoque     | Descripción                                                                                                      |
| :---------- | ---------------------------------------------------------------------------------------------------------------- |
| `Big Bang`  | Se prueban todos los módulos juntos al final. Es arriesgado porque es difícil aislar el origen de los fallos.    |
| `Top-Down`  | Se aprueban los módulos de nivel superior primero (ej. Controladores) usando `stubs` para los de nivel inferior. |
| `Bottom-Up` | Se prueban primero los componentes base (ej. Repositorios/DB) y se va subiendo hacia la `API`.                   |
| `Sandwich`  | Una mezcla de `Tow-Down` y `Bottom-Up`. Ideal para sistemas complejos.                                           |
### El Rol de `Postman` en la Integración
`Postman` es ideal para este nivel de pruebas gracias a las Colecciones:
- **`Chaining Requests`**: Permite tomar un valor de una respuesta (ej. un `order_id`) y usarlo automáticamente como parámetro en la siguiente petición.
- **`Environment Variables`**: Permite cambiar fácilmente entre entornos de `Stagging`, `Testing` y `Local`.
- **`Tests Scripts`**: Valida no solo el `JSON`, sino que el tiempo de respuesta y los `headers` cumplan con el contrato definido.

### El Punto Crítico (`Critical View`)
Aunque las pruebas de integración son fundamentales, tienen un costo alto:
- **Fragilidad (`Flakiness`)**: Son más propensas a fallar por factores externos (red lenta, base de datos no limpia).
- **Mantenimiento**: Cada cambio en la infraestructura puede romper decenas de `tests` de integración.
- **Velocidad**: Son significativamente más lentas que las unitarias.
**Consejo de Arquitectura**: No intentes cubrir cada caso de borde con integración. Usa la pirámide de pruebas: muchas unitarias para la lógica, y solo las integraciones necesarias para validar los cambios críticos del negocio.
