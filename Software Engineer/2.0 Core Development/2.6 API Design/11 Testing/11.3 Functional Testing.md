## Definición
El `API Functional Testing` (Pruebas Funcionales de `API`) es un tipo de prueba de software que se centra en verificar que una `API` cumpla con sus requisitos funcionales específicos. A diferencia de las pruebas de `UI`. estas pruebas se realizan a nivel de protocolo (`HTTP/HTTPS`), validando que el `backend` responda correctamente ante entradas específicas, sin preocuparse por cómo se ve la información en pantalla.
Es el paso previo y fundamental antes de realizar pruebas de rendimiento; primero aseguras que la `API` funciona, luego pruebas qué tan rápido lo hace.

### Objetivos del `Testing` Funcional
El propósito principal es garantizar que la lógica de negocio expuesta por la `API` sea sólida. Esto incluye validar:
- **`Uptime` y Disponibilidad**: El `endpoint` está activo y responde cuando se le llama?
- **Exactitud de los Datos**: La respuesta contiene la información solicitada en el formato correcto (`JSON`/`XML` bien formado)?
- **Códigos de Estado `HTTP`**: Se devuelve `200 OK` para éxito, `404` para no encontrado o `401` para no autorizado?
- **Manejo de Errores**: La `API` responde con mensajes claros y códigos adecuados ante entradas inválidas?

### Qué se valida en una Prueba Funcional?
Al realizar una prueba funcional, se evalúan tres componentes principales de la respuesta:
1. **`Status Line`**: El código de estado y el mensaje (ej. `HTTP/1.1 201 Created`).
2. **`HTTP Headers`**: Información como `Content-Type`, `Cache-Control` o `tokens` de seguridad.
3. **`Response Body`**: El contenido real de la respuesta. Se verifica que los campos existan y que los valores coincidan con lo esperado.

## Beneficios frente al UI `Testing`
Muchos equipos comenten al error centrar toda su automatización en la interfaz gráfica (`GUI`), pero el `testing` funcional de `APIs` ofrece ventajas críticas.
- Velocidad: Las pruebas de `API` son órdenes de magnitud más rápidas que las de `UI`, permitiendo una retroalimentación casi instantánea en el ciclo de desarrollo.
- Confiabilidad: Son menos propensas a fallos por cambios estéticos en la interfaz (`flakiness`).
- Detección Temprana: Permite probar el `backend` antes de que el `frontend` esté terminado, detectando errores de lógica mucho antes en el ciclo de vida del software.

### Escenarios Comunes de Prueba

| Escenario             | Descripción                                                                                   |
| :-------------------- | --------------------------------------------------------------------------------------------- |
| `Happy Path`          | Probar con datos válidos  para asegurar que el flujo principal funciona.                      |
| `Boundary Analysis`   | Probar con valores límite (ej. el máximo de caracteres permitido en un campo).                |
| `Negative Testing`    | Enviar datos inválidos, nulos o con tipos de datos incorrectos para verificar la resiliencia. |
| `Security Validation` | Intentar acceder a un `endpoint` funcional sin el `token` de autorización correspondiente.    |
### Analogía
Imagina una Máquina Expendedora:
- `UI Testing`: Verificas que los botones sean brillantes, que el cristal esté limpio y que la ranura de monedas sea fácil de ver.
- `API Functional Testing`: Presionas el código `A1`. No te importa cómo se ve el botón; te importa que la máquina suelte las patatas fritas, que te devuelva el cambio exacto y que si pones un billete falso, te muestre un mensaje de error y no se trague el billete.
