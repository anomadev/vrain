El `API Load Testing` (Pruebas de Carga de `API`) es un subconjunto de las pruebas de rendimiento que simula la demanda del mundo real sobre una `API` para determinar cómo se comporta bajo condiciones de tráfico específicas. El objetivo no es solo ver si la `API` "se rompe", sino entender su comportamiento, latencia y estabilidad antes de que los usuarios finales encuentren problemas.

### Objetivos Principales
- **Identificar cuellos de botella**: Descubrir qué parte del sistema (base de datos, red, `CPU`, código) limita el rendimiento.
- **Determinar la capacidad máxima**: Saber cuántos usuarios o peticiones simultáneas puede manejar el sistema antes de degradarse.
- **Validar `SLAs`**: Asegurar que la `API` responda dentro de los tiempos acordados (ej. el 95% de las peticiones en menos de 200ms).
- **Verificar la escalabilidad**: Comprobar si el sistema escala correctamente al añadir más recursos.

### Tipos de Pruebas de Carga
Según el enfoque de `Grafana (k6)` y `Postman`, existen diferentes perfiles de carga para distintos objetivos:
1. `Load Test` (Prueba de Carga Estándar): Simula el tráfico esperado para verificar que el sistema cumple con los objetivos de rendimiento.
2. `Stress Test` (Prueba de Estrés): Aumenta la carga más allá de los límites normales para identificar el punto de ruptura del sistema y ver cómo se recupera.
3. `Spike Test` (Prueba de Picos): Simula un aumento repentino y masivo de tráfico (como un "Black Friday") para ver si el sistema sobrevive a ráfagas inesperadas.
4. `Soak / Endurance Test` (Prueba de Resistencia): Mantiene una carga constante durante un largo periodo (horas o días) para detectar fugas de memoria o problemas de almacenamiento que no aparecen cortas.

## Métricas Críticas (`KPIs`)
Para analizar los resultados de una prueba de carga, debes centrarte en estas métricas: 
- `Throughput (RPS)`: El número de peticiones por segundo que el sistema procesa exitosamente.
- `Latency` (Latencia): El tiempo que tarda el servidor en responder. Es vital mirar los `Percentiles (P95, P99)` en lugar del promedio, ya que los promedios ocultan experiencias de usuario muy lentas.
- `Error Rate`: El porcentaje de peticiones que fallan (ej. errores `500` o `504`) bajo carga. 
- `Resources Utilization`: El uso de `CPU`, `RAM` y `E/S` de disco durante la prueba.

### Flujo de Trabajo Recomendado
1. Definir el Escenario: Identificar las rutas más críticas (ej. `Login`, `Search`, `Payment`) y definir cuántos usuarios virtuales (`VUs`) simular.
2. Establecer una Línea Base (`Baseline`): ejecutar una prueba con poca carga para tener una referencia de rendimiento óptimo. 
3. Configurar el `Ramp-up`: No lances todo el tráfico de golpe; aumenta gradualmente el número de usuario para observar cómo se degrada el rendimiento.
4. Analizar y Optimizar: Si la latencia sube drásticamente, revisa los `logs` y el perfilador para encontrar la causa raíz.
5. Pruebas Continuas: Integrar los `tests` de carga en el `pipeline` de `CI/CD` para detectar regresiones de rendimiento en cada despliegue.

### Herramientas Destacadas
- `Grafana K6`: Una herramienta moderna de código abierto enfocada en desarrolladores. Permite escribir pruebas en `JavaScript`, facilitando su integración en el código y en sistemas de `CI/CD`. Es altamente escalable y eficiente en el uso de recursos.
- `Postman Performance Testing` `Postman` ha integrado capacidades de carga que permiten a los usuario simular tráfico real directamente desde sus colecciones de `API`, facilitando la transición de pruebas funcionales a pruebas de rendimiento sin cambiar de herramienta.
- `JMetter`: El estándar clásico de la industria, muy potente y con una gran interfaz gráfica, aunque con una curva de aprendizaje más pronunciada y mayor consumo de recursos.
