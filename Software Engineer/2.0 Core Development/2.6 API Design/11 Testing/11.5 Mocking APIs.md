## Definición 
El `API Mocking` es la técnica de crear un servidor simulado que imita el comportamiento de una `API` real. En lugar de realizar peticiones o un `backend` en producción o desarrollo, los clientes (como aplicaciones móviles o web) se comunican como un `Mock Server` que devuelve respuestas predefinidas (datos falsos, códigos de error, etc.).

### Por qué usar `API Mocking`? (Casos de Uso)
1. **Desarrollo en Paralelo**: El equipo de `frontend` puede empezar a construir la interfaz de usuario basándose en un contrato de `API` simulado mientras el `backend` aún está diseñando la lógica.
2. **Pruebas de Escenarios de Borde (`Edge Cases`)**: Es difícil obligar a una `API` real a devolver un error `500 Internal Server Error` o una latencia de 30 segundos. Con un `mock`, puedes simular estos fallos al instante para ver cómo reacciona tu aplicación.
3. **Aislamiento en Pruebas**: Permite realizar pruebas unitarias o de integración sin depender de la disponibilidad o el estado de una base de datos real.
4. **Demos y Prototipado**: Ideal para mostrar cómo funcionará una aplicación a clientes o inversores antes de que el `backend` sea funcional.

### `Mocking` vs. `Stubbing`
Aunque a menudo se confunden, tienen propósitos distintos:
- **`Stubbing`**: Es una respuesta fija e inamovible (ej. "Siempre devuelve este `JSON`"). Se usa para pruebas muy básicas.
- **`Mocking`**: Es más dinámico y sofisticado. Puede validar la lógica de la petición (ej. "Si la contraseña es `123`, devuelve éxito; si no, devuelve error") y simular comportamientos de red reales como la latencia.

### Características de un Buen Servidor `Mock`
Para que un `mock` sea útil en un entorno profesional, debe permitir:
- **Reglas y Criterios**: Devolver diferentes respuestas basadas en los parámetros de la consulta, el cuerpo del `JSON` o los `headers`.
- **Simulación de Latencia**: Añadir retrasos artificiales para probar `spinners` de carga y tiempos de espera.
- **`Proxying`**: Si una ruta no está `mockeada`, el servidor puede redirigir la petición a la `API` real.
- **Generación de Datos Dinámicos**: Usar librerías como `faker.js` para que cada respuesta tenga nombres, correos o `IDs` diferentes y no se sienta como datos estáticos.

### Herramientas Recomendadas

| Herramienta            | Lo mejor de...                                                                                             |
| :--------------------- | ---------------------------------------------------------------------------------------------------------- |
| `Mockoon`              | Open source, gratuito y con una interfaz visual muy intuitiva para crear reglas complejas.                 |
| `Postman Mock Servers` | Excelente si ya usas `Postman` para documentar, aunque con menos flexibilidad en reglas lógicas complejas. |
| `Prism (Stoplight)`    | Ideal para generar `mocks` automáticamente a partir de un archivo de especificación `OpenAPI/Swagger`.     |
| `Beeceptor`            | Herramienta basada en la web para `mocks` rápidos y captura de peticiones sin instalar nada.               |
