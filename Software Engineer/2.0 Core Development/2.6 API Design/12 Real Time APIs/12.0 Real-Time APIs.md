Una `API` en tiempo real es una interfaz diseñada para transmitir datos instantáneamente --generalmente en menos de 100ms-- desde el servidor al cliente. Esto permite crear experiencias interactivas donde la información parece fluir en vivo.

### Tecnologías y Protocolos Principales
Para lograr esta bidireccionalidad y baja latencia, existen tres enfoques técnicos principales:

| Tecnología           | Tipo de Comunicación              | Mejor para ...                                                 |
| :------------------- | --------------------------------- | -------------------------------------------------------------- |
| `WebSockets`         | Full-Duplex (Bidireccional)       | Chats, juegos multijugador, edición colaborativa.              |
| `Server-Sent Events` | Unidireccional (Server -> Client) | Tickets de bolsa, noticias en vivo, notificaciones de sistema. |
| `HTTP Long Polling`  | Simulación de tiempo real         | Sistemas `legacy` donde `WebSockets` no es una opción.         |
| `WebRTC`             | `Peer-to-Peer` (P2P)              | Videollamadas y `streaming` de audio de alta fidelidad.        |
### Patrones de Arquitectura
La mayoría de las `APIs` en tiempo real operan bajo el modelo `Pub/Sub (Publish/Subscribe)`:
1. **`Publishers` (Publicadores)**: Envían mensajes a un "tópico" o canal específico.
2. **`Brokers` (Intermediarios)**: Reciben el mensaje y gestionan quién debe recibirlo.
3. **`Suscribers` (Suscriptores)**: Clientes que están "escuchando" ese canal y reciben el mensaje al instante.

### Desafíos de Ingeniería
Implementar `APIs` en tiempo real no es solo "abrir un socket"; requiere manejar problema complejos de infraestructura.
- **`Concurrency` (Concurrencia)**: Mantener miles de conexiones abiertas simultáneamente consume mucha memoria y CPU.
- **`State Sync` (Sincronización de Estado)**: Asegurar que todos los clientes vean exactamente lo mismo al mismo tiempo (ej. en un documento compartido).
- **`Latency` (Latencia)**: Cada milisegundo cuenta. Minimizar los saltos de red y optimizar la serialización de datos es crítico.
- **`Fallback Strategies`**: Qué pasa si el cliente pierde la conexión? Necesitas un plan para reconectar y recuperar los mensajes perdidos.

### Casos de Uso del Mundo Real
- **Finanzas**: Actualización de precios de criptomonedas o acciones en milisegundos.
- **Logísticas**: Seguimiento GPS de un repartidor en un mapa en movimiento.
- **Colaboración**: Ver quién está escribiendo o qué cambios se hacen en herramientas como `Figma` o `Google Docs`.
- **Social**: Indicadores de "escribiendo..." y notificaciones de "`likes`" instantáneas.

## Notas Personales
Punto Crítico: No todas las `APIs` necesitan ser real-time. Si los datos cambian cada 5 minutos, un simple `Polling` es más eficiente y menos costoso de mantener. Reserva `WebSockets` para cuando la inmediatez sea una característica `core` del producto. Además, recuerda que mantener conexiones abiertas dificulta el escalado horizontal: necesitarás un `"backplane"` (como `Redis Pub/Sub`) para que los servidores se comuniquen entre sí y sepan a qué cliente enviarle la información.
