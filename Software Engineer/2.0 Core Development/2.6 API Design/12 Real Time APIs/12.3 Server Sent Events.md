## Definición
`Server-Sent Event (SSE)` es una tecnología que permite a un servidor enviar notificaciones automáticas y datos en tiempo real a una página web a través de una conexión `HTTP` estándar. A diferencia de `WebSockets`, `SSE` es unidireccional: los datos fluyen únicamente del servidor al cliente. Se basa en una conexión de larga duración donde el servidor mantiene la respuesta abierta para enviar "eventos" a medida que ocurren.

### Cómo funciona el Protocolo
`SSE` utiliza el protocolo `HTTP` convencional pero con una configuración específica que los transforma en un flujo de datos continuo:
- **Solicitud del Cliente**: El cliente realiza una petición `GET` normal al servidor.
- **Respuesta del Servidor**: El servidor responde con un `header` crítico: `Content-Type: text/event-stream`. También debe incluir `Cache-Control: no-cache` y `Connection: keep-alive`.
- **Flujo de Datos**: El servidor envía bloques de texto siguiendo un formato estricto:
	- `event: nombre_del_evento` (opcional)
	- `data: contenido del mensaje`
	- `id: identificador` (opcional, para recuperación de conexión)
	- `retry: milisegundos` (opcional, para definir el tiempo de reconexión)

### Características Principales
- **Unidireccionalidad**: Idea para escenarios donde el cliente solo necesita "escuchar" (ej. resultados deportivos o notificaciones de sistema).
- **Reconexión Automática**: El navegador (vía la interfaz `EventSource`) gestiona automáticamente la reconexión si el enlace se pierde. Gracias al campo `id`, el servidor puede saber cuál fue el último mensaje enviado y retomar desde allí.
- **Basado en Texto**: Los mensajes deben ser texto (usualmente `JSON` serializado), lo que los hace más ligeros de depurar que los `frames` binarios de `WebSockets`.
- **Eficiencia de Recursos**: Consumo menos recursos que el `Polling` constante y es más fácil de implementar en servidores que ya manejan `HTTP`, ya que no requiere un cambio de protocolo.

### SSE vs. `WebSockets`

| Característica    | Server-Sent Events (SSE)             | WebSockets                             |
| :---------------- | ------------------------------------ | -------------------------------------- |
| Dirección         | Unidireccional (Servidor -> Cliente) | Bidireccional (`Full-Duplex`)          |
| Protocolo         | HTTP Estándar                        | Protocolo propio (`ws://`)             |
| Reconexión        | Automática (Nativa)                  | Manual (Debe programarse)              |
| Formato           | Texto (`text/event-stream`)          | Binario y Texto                        |
| Límite Conexiones | Máximo 6 por domino (en `HTTP/1.1`)  | Ilimitado (según recurso del servidor) |
### Análisis Técnico y Limitaciones
Aunque `SSE` es muy eficiente, tiene una limitación histórica importante: en navegadores que usan `HTTP/1.1`, el número de conexiones `SSE` abiertas está limitado a 6 por dominio. Esto puede ser un problema si el usuario abre varias pestañas de la misma aplicación. El uso de `HTTP/2` resuelve esto gracias a la multiplexación de `streams`.
##### Implementación en el Cliente (JavaScript)
La `API` nativa `EventSource` hace que sea extremadamente sencillo de consumir:
```javascript
const eventSource = new EventSource('/api/updates');

eventSource.onmessage = (event) => {
    console.log("Nuevo mensaje:", event.data);
};

eventSource.addEventListener('ping', (event) => {
    console.log("Ping recibido a las:", event.data);
});
```

## Analogía
Imagina la suscripción a una Revista:
- **`Polling`**: Vas al quiosco cada mañana a preguntar: Ya salió el nuevo número? La mayoría de las veces el quiosquero te dice que no.
- **`Websockets`**: Tienes un walkie-talkie con la editorial. Puedes hablar con ellos y ellos contigo en cualquier momento.
- **`Server-Sent Event (SSE)`**: Te suscribes y la revista llega a tu buzón en cuanto sale. Tú no necesitas ir a ningún lado ni hablar con la editorial; ellos simplemente te entregan el contenido nuevo cada vez que está listo.
