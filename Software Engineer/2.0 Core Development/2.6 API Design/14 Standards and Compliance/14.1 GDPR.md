El `GDPR (General Protection Regulation)` es el marco legal más estricto del mundo sobre privacidad  y protección del datos personales. Para un arquitecto, no es solo un conjunto de leyes, sino un requerimiento no funcional sistémico. Define cómo se deben recolectar, procesar, almacenar y eliminar los datos de ciudadanos de la Unión Europea (UE), con un alcance extraterritorial: si tu SaaS de Comercio Exterior tocas datos de un ciudadano europeo, la ley te aplica, sin importar dónde esté tu servidor.

## Conceptos Clave
- **`Privacy by Design & Default (Art. 25)`**: La privacidad no es un parche; es parte de la arquitectura. El sistema debe estar configurado para proteger la privacidad de forma automática, limitando la recolección al mínimo necesario.
- **`Data Minimization (Minimización de Datos)`**: Solo se procesa lo estrictamente necesario para la finalidad declarada. Si no necesitas el teléfono personal para una declaración aduanera, no lo guardes.
- **`Right to Erasure (Derecho al Olvido - Art. 27)`**: Capacidad técnica de eliminar datos de un usuario de forma definitiva, incluyendo `backups` y `logs` (aquí es donde fallan la mayoría de los sistemas `legacy`).
- **`Data Pseudonymization & Encryption (Art. 32)`**: El uso de técnicas para que los datos no puedan atribuirse a un interesado sin información adicional, y el cifrado tanto en tránsito (`TLS`) como en reposo (`AES`).
- **`Lawfulness of Processing (Licitud)`**: Debes tener una base legal (consentimiento, contrato o interés legítimo) para cada datos que toques.

## Ejemplos Prácticos
##### `Implementation of Data Masking in API Resources`
Para cumplir con la minimización de datos en las respuestas de la `API`, se debe usar una capa de transformación que oculte datos sensibles por defecto.
```php
// Laravel Resource / Presenter Pattern
namespace App\Http\Resources\V1;

use Illuminate\Http\Resources\Json\JsonResource;

class UserProfileResource extends JsonResource
{
    public function toArray($request)
    {
        return [
            'id' => $this->id,
            'company_name' => $this->company_name,
            // Masking email to comply with security of processing
            'email' => $this->maskEmail($this->email),
            'role' => $this->role,
            // Sensitive PII only returned if explicitly authorized
            'personal_phone' => $request->user()->isAdmin() ? $this->personal_phone : 'REDACTED',
        ];
    }

    private function maskEmail($email) {
        $parts = explode("@", $email);
        return substr($parts[0], 0, 3) . '***@' . $parts[1];
    }
}
```
##### `Folder Structure for Data Sovereignty and Compliance`
Organización de módulos encargados específicamente del manejo de datos sensibles y solicitudes de borrado.
```
src/
├── core/
│   ├── compliance/
│   │   ├── gdpr/
│   │   │   ├── DataErasureService.php    # Handles logical/physical deletion
│   │   │   ├── ConsentManager.php        # Tracks user consent versions
│   │   │   └── PiiDetector.php           # Scans payloads for sensitive data
├── domain/
│   └── users/
│       ├── observers/
│       │   └── GDPRLogObserver.php       # Logs only non-PII metadata for audits
└── infrastructure/
    └── storage/
        ├── encrypted/                    # Storage for high-risk PII
        └── retention_policies/           # Automated logic for storage limitation
```

## Notas Personales
- **La trampa de los `Logs`**: Es muy fácil se `compliant` en la base de datos, pero los ingenieros solemos olvidar que los `Logs de Laravel/Nginx` a menudo capturan el `Request Body` completo. Si un usuario ejerce su "Derecho al Olvido", pero su email sigue en un `log` de error por 90 días, técnicamente sigues incumpliendo. Necesitas sanitizar los `logs` en el `middleware`.
- **Conflicto de Leyes (`Customs vs. GDPR`)**: Este es el problema real para tu SaaS. La ley aduanera suele exigir retención de datos por 5 o 10 años, mientras que el `GDPR` permite al usuario pedir su borrado. **Conclusión**: El cumplimiento legal de la ley aduanera suele anular el derecho al olvido, pero solo para los datos estrictamente necesarios para la declaración. El resto (datos de marketing, perfiles, preferencias) sí deben borrarse.
- **`Backups` como agujero negro**: Si restauras un `backup` de hace un mes para arreglar un `crash` podrías estar "resucitando" datos de usuario que ya pidieron ser borrados. Tu arquitectura debe tener un sistema de `Tombstone Records` (registros de borrado) que se re-aplique tras una restauración de base de datos.
