## Definición
La `CCPA (California Consumer Privacy Act)` es la respuesta de Estados Unidos al `GDPR`, centrada específicamente en los residente de California. A diferencia del `GDPR`, que se enfoca en el "procesamiento", la `CCPA` pone la lupa en la venta y el intercambio de información personal. Para un arquitecto de software, la `CCPA` introduce la necesidad técnica de gestionar estados de "`Opt-Out`" y de diferenciar claramente entre un "Proveedor de Servicios" (`Service Provider`) y un "Tercero" (`Third Party`).

## Conceptos Clave
- **`Verificable Consumer Request (VCR)`**: No puedes entregar datos solo porque alguien los pida. Debes implementar un flujo de verificación de identidad robusto en tus `APIs` antes de procesar una solicitud de acceso o borrado.
- **`The Sale vs. Share Distinction`**: Bajo `CCPA`, el concepto de "venta" es extremadamente amplio (incluye transferencia de datos por beneficios no monetarios). La `CPRA` (actualización de la ley) añadió el concepto de "compartir" para publicidad conductual.
- **`Opt-Out Mechanism`**: La obligación técnica de tener un `endpoint` o señal (como el `Global Privacy Control`) que, al activarse, detenga inmediatamente la transferencia de datos a terceros.
- **`Service Provider Status`**: Si tu SaaS actúa como "`Service Provider`" para tus clientes B2B, debes tener contratos que prohíban el uso de esos datos para tus propios fines. Esto te protege de ser considerado un "vendedor" de datos.
- **`Sensitive Personal Information (SPI)`**: Datos como geolocalización precisa, etnia o contenido de comunicaciones. Requieres un nivel de control y limitación de uso aún más estricto.

## Ejemplos Prácticos
##### `Identity Verification Flow for Data Access Requests`
Un patrón común es el uso de un `Verification Token` enviado por email antes de exponer los datos en la `API`.
```javascript
// Verification Middleware for Privacy Requests
const verifyPrivacyRequest = async (req, res, next) => {
    const { requestId, verificationToken } = req.body;
    
    // Check if the token matches the requested data and hasn't expired
    const isValid = await PrivacyService.validateToken(requestId, verificationToken);
    
    if (!isValid) {
        return res.status(401).json({ 
            error: "Identity verification failed. Please verify your email before proceeding." 
        });
    }
    
    next();
};

// API Endpoint for CCPA Right to Know
app.post('/api/v1/privacy/access-request', verifyPrivacyRequest, async (req, res) => {
    const userData = await DataExportService.collectAll(req.user.id);
    res.status(200).json(userData);
});
```
##### `Opt-Out Flag Propagation`
Cuando un usuario activa el `Do Not Sell`, este estado debe propagarse a todas la integraciones externas (como `Analytics` o `CRM`).
```
src/
├── app/
│   ├── Privacy/
│   │   ├── OptOutManager.php         # Centralizes the opt-out state
│   │   ├── Signals/
│   │   │   └── GpcHandler.php        # Global Privacy Control signal detector
│   │   └── Synchronizers/
│   │       ├── HubSpotSync.php       # Updates third-party opt-out state
│   │       └── SegmentSync.php       # Disables tracking if opted-out
```

## Notas Personales
- **El mito de "Somo B2B, no aplica"**: Muchos ingenieros creen que por ser un SaaS B2B están exentos. Con la llegada de la `CPRA`, los datos de los empleados de tus clientes (nombres, emails corporativos) se consideran `Personal Information`. Si tu SaaS de Comercio Exterior tiene usuarios en California, el `CCPA` vive en tus bases de datos.
- **El `"Vendor Sprawl"` en la refactorización**: Al moverte de MVC a una arquitectura de servicios, es probable que añadidas microservicios de terceros (`logs`, monitoreo, búsqueda). Cada vez que envías un `user_id` o una `IP` a un tercero, estás abriendo un flanco de `CCPA`. Estrategia: Usa un `Data abstraction layer` que ofusque los datos antes de enviarlos a herramientas de observabilidad como `New Relic` o `Datadog`.
- **`Audit Trail vs. Opt-Out`**: Al igual que con el `GDPR`, si un cliente pide "No vender/compartir", pero tu sistema de `logs` de auditoría (necesario para aduanas) envía esos datos a una plataforma de análisis externa, podrías estar en falta. La arquitectura debe distinguir entre `Logs Operacionales Legales` (permitidos) y `Data Sharing for Analytics` (sujeto a `Opt-Out`).
