## Definición
El `PCI DSS (Payment Card Industry Data Security Standard)` es el estándar de seguridad global obligatorio para cualquier entidad que procese. almacene o transmita datos de tarjetas de pago (`PAN - Primary Account Number`). Para un arquitecto de `software`, `PCI DSS` no es un `"check"` legal, sino una disciplina de aislamiento de red. El objetivo fundamental es proteger el `CDE (Cardholder Data Environment)` y reducir la superficie de ataque mediante endurecimiento de sistemas y la restricción estricta del acceso a los datos.

## Conceptos Clave
- **`Scope Minimization` (Reducción de Alcance)**: La regla de oro. La mejor forma de cumplir con `PCI` es no tocar los datos. Si el dato no entra en tus servidores, no está en `"scope"`.
- **`Tokenization`**: El proceso de reemplazar el `PAN` por un valor no sensible (`token`). El `PAN` viven en el servidor del proveedor de pago (`Stripe`, `Adyen`), y tu SaaS solo maneja el `token`.
- **Los 6 Pilares**
	- Redes Seguras (`Firewalls`)
	- Protección de datos almacenados (`Cifrado`)
	- Gestión de vulnerabilidades (`Antivirus/Parches`)
	- Control de acceso (Mínimo privilegio/MFA)
	- Monitoreo y Pruebas (`Logs`/`Pentesting`)
	- Políticas de seguridad
- **`CDE Isolation`**: Segmentar físicamente o lógicamente la red donde se procesan tarjetas del resto de la infraestructura (ej. tu lógica de comercio exterior).
- **`SAQ (Self-Assessment Questionnaire)`**: El nivel de auditoría depende de tu volumen de transacciones y de cómo manejas los datos (`SAQ-A` para dirección total, `SAQ-D` para almacenamiento propio).

## Ejemplos Prácticos
##### `PCI-Compilant Folder Structure (Isolation Pattern)`
Es una arquitectura desacoplada, los servicios que tocan pagos deben estar aislados para que una vulnerabilidad en el módulo de "Logistica" no comprometa el "CDE".
```
src/
├── services/
│   ├── payments/
│   │   ├── CardVaultProxy.php        # Transparent proxy to Payment Gateway
│   │   ├── TokenManager.php          # Handles only non-sensitive tokens
│   │   └── providers/
│   │       └── StripeProvider.php    # Implementation of PCI-DSS via Elements/iFrames
├── infrastructure/
│   └── security/
│       ├── SensitiveDataMasker.php   # Redacts PANs from logs automatically
│       └── NetworkSegments.php       # Logic for VPC Peering/Security Groups
```
##### `Log Redaction Middleware (Anti-Leak Pattern)`
Uno de los fallos de cumplimiento más comunes es registrar accidentalmente el número de tarjeta en los `logs` de depuración.
```javascript
// Middleware to prevent PAN leakage in application logs
const pciDataMasker = (req, res, next) => {
    const panRegex = /\b(?:4[0-9]{12}(?:[0-9]{3})?|5[1-5][0-9]{14}|3[47][0-9]{13})\b/g;
    
    // Deep clone and mask body if it exists
    if (req.body) {
        const bodyString = JSON.stringify(req.body);
        if (panRegex.test(bodyString)) {
            console.warn(`[SECURITY ALERT] Blocked request body containing PAN from IP: ${req.ip}`);
            req.body = JSON.parse(bodyString.replace(panRegex, "XXXX-XXXX-XXXX-XXXX"));
        }
    }
    next();
};
```

## Notas Personales
- **La paradoja de los pagos B2B**: En un SaaS de Comercio Exterior, la mayoría de los pagos son vía transferencia bancaria (`Wire Transfer`), lo que te exime de PCI. Sin embargo, si decides añadir una opción de "Pago de cuotas con tarjeta" para pequeñas comisiones, metes a toda tu infraestructura en el rada de `PCI`. Decisión de Arquitecto: Usa siempre una solución de `iFrame` (`Strioe Elements`) para que los datos nunca toquen tus servidores.
- **El peligro de los `Logs` en la refactorización**: Al moverte los MVC a `APIs`, es muy común que los desarrolladores activen el `DEBUG=true` o `logueen` todos los `Request $request`. Si un campo de tarjeta llega a tus `logs` (`CloudWatch`, `ELK`), habrás fallado la auditoría instantáneamente. Necesitas un `masker` a nivel de infraestructura, no de aplicación.
- **Segmentación de Red**: Si tu SaaS crece, no mezcles el servicio de pagos con el de generación de `PDFs` de aduanas. Si están en la misma red o comparten base de datos, el auditor de `PCI` te exigirá que el servicio de `PDFs` también cumpla con los 12 requisitos de seguridad, lo que triplicará tu deuda técnica y operativa.
