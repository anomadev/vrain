## Definición
La `HIPPA (Health Insurance Portability and Accountability Act)` es la ley federal de EE. UU. que establece los estándares nacionales para la protección de la información de salud sensible (`PHI - Protected Health Information`) para que no sea revelada sin el consentimiento o conocimiento del paciente. Desde la arquitectura de software, `HIPPA` se traduce en un marco de gobernanza de datos de salud, donde la seguridad no es opcional, sino una combinación de protecciones técnicas, administrativas y físicas que garantizan la integridad y confidencialidad absoluta del dato médico.

## Conceptos Clave
- **`PHI (Protected Health Information)`**: Cualquier información en un registro médico que pueda ser usada para identificar a un individuo y que fue creada, usada o compartida en el curso de la prestación de un servicio de salud.
- **`Business Associates (BA)`**: Si tu SaaS procesa `PHI` para un entidad cubierta (hospital, clínica), eres un Socio Comercial y debes firmar un `BAA (Business Associate Agreement)`, asumiendo responsabilidad legal por la seguridad de los datos.
- **`Security Rule - Technical Safeguards`**:
	- **`Access Control`**: Limitar el acceso a `PHI` solo a personas o procesos autorizados. 
	- **`Audit Controls`**: Registrar y examinar la actividad en los sistemas que contienen `PHI`.
	- **`Integrity`**: Asegurar que la `PHI` no sea alterada o destruida de forma no autorizada.
	- **`Transmission Security`**: Cifrar los datos siempre que viajes por la red.
- **`Minimum Necessary Rule`**: Al igual que la minimización de datos de `GDPR`, exige que solo se acceda a la cantidad mínima de `PHI` necesaria para realiza una tarea específica.

## Ejemplos Prácticos
##### `PHI Isolation and Encryption Strategy`
Arquitectónicamente, la `PHI` nunca debe mezclarse con datos genéricos del sistema sin capas de cifrado adicionales.
```
src/
├── app/
│   ├── PHI/
│   │   ├── Encryption/
│   │   │   ├── KMSService.php          # Interface with Key Management Service (AWS KMS/Azure Vault)
│   │   │   └── PhiEncryptor.php        # Application-level field encryption
│   │   ├── Auditing/
│   │   │   └── AccessLogger.php        # Mandatory audit trail for every PHI read/write
│   │   └── AccessControl/
│   │       └── PhiPolicyEvaluator.php  # Validates 'Minimum Necessary' logic
```
##### `Database Field-Level Encryption`
En lugar de cifrar toda la base de datos (que es el mínimo), `HIPAA` sugiere cifrar los campos sensibles a nivel de aplicación para proteger el dato incluso si la base de datos se ve comprometida.
```javascript
// Example of field-level encryption before persisting to Database
const savePatientData = async (patient) => {
    const sensitiveFields = ['ssn', 'birthDate', 'medicalHistory'];
    
    const encryptedData = { ...patient };
    
    for (const field of sensitiveFields) {
        // Use a unique IV (Initialization Vector) per record/field
        encryptedData[field] = await EncryptionService.encrypt(patient[field]);
    }
    
    // Audit the creation of PHI
    await AuditLogger.logPhiAccess({
        userId: currentUser.id,
        action: 'CREATE_PHI',
        resourceId: patient.id,
        justification: 'New patient intake'
    });

    return await PatientRepository.save(encryptedData);
};
```

## Notas Personales
1. **Por qué aprender `HIPAA` si hago Comercio Exterior?** Aunque tu SaaS no maneje `PHI`, los `Technical Safeguards` de `HIPAA` son el "estándar de oro" para cualquier industria de alta criticidad. Si diseñas tu sistema de Comercio Exterior para pasar una auditoría `HIPAA`, estarás automáticamente blindado contra casi cualquier otra normativa de seguridad de datos.
2. **La trampa del `SaaS Multi-tenant`**: En una arquitectura compartida, el riesgo de `HIPAA` es que un error en el `where user_id = ?` exponga datos médicos de un paciente a otro cliente. La seguridad a nivel de fila (`RLS`) y el aislamiento lógico de bases de datos son patrones que `HIPAA` te obliga a perfeccionar.
3. **`Logs y PHI`**: Al igual que con `PCI` y `GDPR`, el mayor error es el `leak` de `PHI` en `logs` de error o en herramientas de `APM` (como `Sentry` o `New Relic`). Si un diagnóstico médico termina en un `log` de texto plano en `CloudWatch`, tienes un incidente de seguridad mayor. La sanitización de `logs` es una disciplina que `HIPAA` lleva al extremo.
