## Definición
La `PII` (Información de Identificación Personal) es cualquier dato que pueda utilizarse, por sí solo o combinado con otra información, para identificar contactar o localizar a una persona específica. Desde la arquitectura de software, la `PII` debe tratarse como "Residuos Tóxicos": es valiosa mientras se usa, pero representa un riesgo masivo si se almacena sin control. No se trata solo de nombres; es el rastro digital que vincula a un individuo con un registro en tu base datos.

## Conceptos Clave
- **`PII Sensible vs. No Sensible`**: 
	- **Sensible**: Datos que, si se filtran, causan daño directo (`SSN`, biométricos, registros financieros, salud).
	- **No Sensible**: Datos públicos o de bajo riesgos (nombre completo, dirección, código postal), pero que combinados pueden identificar a alguien.
- **Identificadores Directos vs Indirectos (`Quasi-identifiers`)**: Un pasaporte es directo. Una combinación de "Fecha de nacimiento + Código Postal + Género" es un identificador indirecto que puede identificar al 87% de la población de EE.UU.
- **`Linkability` (Vinculación)**: La capacidad de unir piezas de datos de diferentes fuentes para revelar la identidad. Tu `API` debe prevenir que un ataque "una los puntos".
- **`De-identification` (Desidentificación)**: El proceso de eliminar o enmascarar identificadores. Incluye la `Seudonimización` (reversible con una llave) y la Anonimización (irreversible).
- **Principio de Privacidad por Defecto**: Los sistemas deben diseñarse para que, en su configuración básica, el acceso a la `PII` sea nulo o mínimo.

## Ejemplos Prácticos
##### `PII Protection & Masking Layer (Service Pattern)`
En lugar de permitir que los servicios lean `PII` directamente, se utiliza un servicio de "bóveda" o una capa de transformación.
```
src/
├── app/
│   ├── Privacy/
│   │   ├── Vault/
│   │   │   ├── DetokenizerService.php    # Retrieves real PII from secure storage
│   │   │   └── PiiVaultRepository.php    # Interface for the dedicated PII DB
│   │   ├── Masking/
│   │   │   ├── PhoneMasker.php           # Logic to return +52 ****** 89
│   │   │   └── AddressObfuscator.php     # Logic to return only City/State
│   │   └── Scoping/
│   │       └── PiiAccessPolicy.php       # ABAC/RBAC rules for PII access
```
##### `Data Redaction in API Responses`
Implementación de un patrón para asegurar que los datos sensible no se "escapen" por pereza del desarrollador al retornar un objeto.
```javascript
// Generic PII Redactor for API responses
const redactPii = (data, scope = 'public') => {
    const sensitiveFields = ['ssn', 'tax_id', 'private_phone', 'birth_date'];
    
    return Object.keys(data).reduce((acc, key) => {
        if (sensitiveFields.includes(key) && scope !== 'internal_admin') {
            acc[key] = '[REDACTED]';
        } else {
            acc[key] = data[key];
        }
        return acc;
    }, {});
};

// Usage in an Express/Node controller
app.get('/api/v1/brokers/:id', async (req, res) => {
    const broker = await BrokerService.getById(req.params.id);
    const safeData = redactPii(broker, req.user.role);
    res.json(safeData);
});
```

## Notas Personales
- **La Ceguera del B2B**: En el SaaS de Comercio Exterior, es fácil decir "solo manejamos empresas". Error. Los Pedimentos, las Cartas de Encomienda y los Manifiestos contienen `PII` de: representantes legales, choferes de camiones (licencias), agentes aduanales y clientes finales. Al refactorizar tu `MVC`, debes etiquetar estos campos como `PII` aunque vivan en tables de "Negocio".
- **La Refactorización y el `Data Sprawl`**: Al moverte de un monolito a una arquitectura de servicios, el riesgo es que la `PII` empiece a replicarse en múltiples bases de datos de microservicios para evitar `Joins`. Regla de Oro: La `PII` debe vivir en un solo microservicio (`User/Identity Service`). Los demás servicios deben usar un `UUID` o un `Token`, nunca el nombre o el email del usuario como llave.
- **`Analytics vs. PII`**: Es común enviar datos a herramientas como `Mixpanel` o `google Analytics`. Si envías el email de representante legal en un evento de "Declaración Creada", acabas de meter a tu herramienta de marketing en el rada de cumplimiento legal. Sanitiza siempre los eventos de analítica.
