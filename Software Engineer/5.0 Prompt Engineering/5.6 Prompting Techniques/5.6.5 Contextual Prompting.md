## Definición
El `Contextual Prompting` consiste en incluir información de fondo específica, hechos, documentos o estados previos dentro del prompt ([[5.1.2 What is a Prompt]]) para "anclar" (grounding) la respuesta del modelo a una realidad concreta. Su objetivo es maximizar la relación **Señal/Ruido ($SNR$)**, proporcionando al modelo los datos necesarios para resolver la tarea sin que tenga que adivinar o alucinar basándose en información genérica.

## Conceptos Clave
- **Grounding (Anclaje)**: Técnica de forzar al modelo a responder basándose *únicamente* en el contexto proporcionado.
- **Signal-to-Noise Ratio ($SNR$)**: La eficiencia del prompt. Demasiado contexto irrelevante aumenta el ruido y puede causar que el modelo ignore la información crítica ("*Lost in the Middle*").
- **Temporal Context**: Información sobre el "ahora" (fecha, hora, estado actual del sistema) que el modelo no posee de forma nativa debido a sus fecha de corte de entrenamiento.
- **Domain Specificity**: Inyección de terminología, reglas de negocio o manuales técnicos que son privados o demasiado recientes para el modelo.

## Escenarios de Aplicación e Identificación
¿Dónde aplicarlo?
- **Customer Support Agents**: Inyectar el historial de compras y tickets anteriores del usuario para que el agente sepa quién es y qué problemas ha tenido.
- **Análisis de Código**: Proporcionar el contenido de archivos relacionados o la documentación de una librería interna antes de pedir un refactor.
- **Sistemas RAG ([[5.1.5 Common Terminology]])**: Buscar en una base de datos vectorial los fragmentos de un PDF y pegarlos en el prompt antes de que el usuario haga una pregunta sobre dicho documento.
**¿Cómo identificarlo?**
- **Directivas de Referencia**: Uso de frases como "*Based on the following documentation...*", "*Using the provided context...*" o "*According to the user's history...*".
- **Delimitadores de Datos**: Verás bloques de texto claramente separados por etiquetas como `[CONTEXT]`, `---`,  o `### DOCUMENTATION ###`.

## Notas Personales
En el diseño de sistemas, el contexto es un arma de doble filo. Exista la tentación de "darle todo el manual" al modelo. Sin embargo, como arquitecto, mi regla de oro es: "**Solo inyecta lo que el modelo necesita para este turno específico**". La relevancia es más importante que la completitud. Si el contexto está sucio, la respuesta será basura (GIGO - Garbage In, Garbage Out).
