 Aunque los `Form Requests` son la opción preferida para la mayoría de los escenarios, existen momentos donde necesitas un control total sobre el proceso de validación. La Validación Manual te permite crea una instancia del validador en cualquier parte de tu código, no solo al inicio de un controlador, lo cual es vital para procesos complejos, `APIs` personalizadas o validación de datos que no vienen directamente de una petición `HTTP`.
##### Cuándo usar Validación Manual?
Generalmente, querrás usar este enfoque si:
- Necesitas validar datos en un `Service`, un `Job` o un comando de consola.
- La lógica de validación depende de condiciones que solo conoces a mitad de la ejecución del método.
- Estás manejando peticiones `AJAX` muy específicas y quieres un formato de respuesta de error único.

##### Implementación con la `Facade Validator`
Para crear un validador manualmente, utilizamos el método `make` de la `facade` `Validator`. Este método recibe tres argumentos: los datos a validar, las reglas y (opcionalmente) los mensajes personalizados.
```php
use Illuminate\Support\Facades\Validator;
use Illuminate\Http\Request;

public function store(Request $request)
{
    $validator = Validator::make($request->all(), [
        'title' => 'required|unique:posts|max:255',
        'body' => 'required',
    ]);

    if ($validator->fails()) {
        return redirect('post/create')
                    ->withErrors($validator)
                    ->withInput();
    }

    // Los datos son válidos...
    $validated = $validator->validated();
}
```

##### El método `fails()` vs `passes()`
Una vez creada la instancia, puedes interrogarla:
- `$validator->fails()`: Devuelve `true` si alguna regla se rompió.
- `$validator->passes()`: El opuesto semántico.
- `$validator->errors()`: Devuelve una instancia de `MessageBag` con todos los fallos encontrados.

##### Redirección Automática con `validate()`
Si estás dentro de un controlador y usas la validación manual pero quieres que Laravel maneje la redirección por ti (como lo hace con los `Form Requests`), puedes llamar al método `validate()` directamente sobre la instancia al validador:
```php
Validator::make($request->all(), [
    'title' => 'required',
])->validate();

// Si llegas aquí, la validación pasó.
```
- Nota de experto: Si la validación falla, este método lanzará una `ValidationException` que Laravel capturará automáticamente para redirigir al usuario o enviar una respuesta `JSON 422`.

##### Comparativa: Manual vs. `Form Requests`
| Característica | Validación Manual                    | `FormRequest`                        |
| :------------- | ------------------------------------ | ------------------------------------ |
| Ubicación      | Cualquier parte del código           | Clase separada (`app/Http/Requests`) |
| Reutilización  | Baja (a menos que se use un `Trait`) | Alta                                 |
| Legibilidad    | Puede ensuciar el controlador        | Mantiene el controlador `"Skinny"`   |
| Control        | Máximo (paso a paso)                 | Automatización                       |
| Uso Ideal      | `Jobs`, Comandos, Lógica dinámica    | Formularios estándar de la App       |
##### Consejos Pro
1. Validación de `Arrays`: Si estás validando una lista (ej. un carrito de compras), usa la notación de punto: `'items.*.id' => 'required|exists:products,id'`.
2. Mensajes Condicionales: Puedes usar el método `after()` para añadir validaciones personalizadas que se ejecuten solo si las reglas básicas pasaron.
```php
$validator->after(function ($validator) {
    if ($this->somethingIsWrong()) {
        $validator->errors()->add('field', 'Algo salió mal con este campo específico.');
    }
});
```
3. No repitas lógica: Si te encuentras escribiendo el mismo `Validator::make` en tres controladores diferentes, es una señal clara de que debes refactorizar a un `Form Request`.
