## Definición
Laravel Sanctum permite emitir múltiples tokens de acceso de larga duración para tus usuarios. Estos tokens pueden usarse para autenticar peticiones a través de la cabecera `Authorization: Bearer`. Además, ofrece una forma segura de autenticar `SPAs` mediante cookies de sesión, aprovechando la protección CSRF integrada de Laravel.

### Los dos pilares de `Sanctum`
##### `Tokens` de Acceso Personal (`mobile`/`desktop`)
`Sanctum` permite a los usuarios generar "tokens de acceso personal" (similar a los tokens de GitHub). Estos son perfectos para aplicaciones móviles o scripts de terceros.
- **Almacenamiento:** Los tokens se guardan en una tabla `personal_access_tokens`.
- **Seguridad:** Los tokens son `hasheados` usando `SHA-256` antes de guardarse.

```php
// Generar un token para un usuario
$token = $user->createToken('nombre-del-dispositivo')->plainTextToken;

// Acceder con el token en el cliente
// Authorization: Bearer {token}
```
##### Autenticación de `SPA` (`Single Page Application`)
Este es el uso más común para frameworks como Vue, React o Next.js que viven en el mismo dominio (o subdominio) que la API.
- **Mecanismo:** No usa tokens en las cabeceras; usa **cookies de sesión** tradicionales.
- **Ventaja:** Es más seguro que almacenar tokens en `localStorage`, ya que las cookies `httpOnly` protegen contra ataques `XSS`.

### Configuración Esencial 
Para que `Sanctum` funcione correctamente, especialmente en `SPAs`, debes configurar tres elementos clave:
1. **`Stateful Domains`:** En `config/sanctum.php`, define los dominios que deben ser tratados como "autenticados por sesión".
2. **`Middleware`:** Asegúrate de que el middleware `EnsureFrontendRequestsAreStateful` esté presente en tu grupo de rutas `api` en `bootstrap/app.php`.
3. **`CORS`:** Configura `config/cors.php` para permitir `supports_credentials => true`.

### Habilidades (`Abilities`)
`Sanctum` permite asignar permisos específicos a los tokens. Esto es útil para limitar qué puede hacer una aplicación móvil o un servicio externo sin dar acceso total a la cuenta del usuario.

```php
// Crear token con permisos limitados
$user->createToken('token-name', ['post:update', 'post:delete']);

// Verificar habilidad en el controlador
if ($user->tokenCan('post:update')) {
    // ...
}
```

## Analogía 
Imagina que tu aplicación es un **Hotel**.
- **`SPA Authentication`** es como si fueras un **Huésped de Larga Estancia**. El hotel te reconoce por tu cara (sesión/cookie) mientras estés dentro de las instalaciones del mismo dueño. No necesitas mostrar tu identificación en cada puerta.
- **`API Tokens`** son como una **Tarjeta Magnética Temporal**. Se la das a un servicio de limpieza (App Móvil) para que entre. La tarjeta tiene límites (Abilities): solo abre la puerta de la habitación, no la de la caja fuerte ni la del gimnasio. Si pierdes la tarjeta, puedes cancelarla sin cambiar las cerraduras del hotel.

## Buenas Prácticas
- **Protección de Rutas:** Usa siempre el middleware `auth:sanctum` para proteger tus rutas de API. Laravel se encargará de decidir si usa tokens o sesiones automáticamente.
- **Expiración de `Tokens`:** Configura el tiempo de vida de los tokens en `config/sanctum.php` (`expiration`) para forzar a los usuarios de apps móviles a re-autenticarse periódicamente.
- **`Sanctum vs. Passport`:** No uses Passport a menos que necesites explícitamente flujos de OAuth2 (como "`Login with Google`" para otros desarrolladores). `Sanctum` es más rápido, consume menos memoria y es más fácil de mantener.
- **Limpieza:** Implementa una interfaz para que los usuarios puedan revocar sus propios tokens. Esto es vital para la seguridad en caso de robo de dispositivos.
