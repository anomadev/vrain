## Definición
Laravel Passport utiliza la librería **League OAuth2 Server** para proporcionar una implementación completa de un servidor de autorización. Permite que otras aplicaciones soliciten permiso para acceder a los datos de tus usuarios de forma segura sin que estos tengan que compartir sus contraseñas, utilizando flujos de tokens estandarizados a nivel mundial.

### Cuando elegir `Passport` sobre `Sanctum`?
La decisión depende de quién consume la API:
- **`Sanctum`:** Si solo tú (tu propio `frontend` en `Vue/React` o tu App móvil) consumes la API.
- **`Passport`:** Si planeas permitir que **desarrolladores externos** construyan aplicaciones que se integren con tu plataforma (estilo "`Login with Tu App`" o acceso a datos vía API para terceros).

### Tipos de `Grants` (Subvenciones) Principales
OAuth2 es complejo porque tiene diferentes formas de emitir tokens según el caso de uso. Passport maneja los más importantes:
1. **`Authorization Code Grant`:** El flujo más común (y seguro). El usuario es redirigido a tu app, aprueba el acceso y es redirigido de vuelta con un código que se canjea por un token.
2. **`Client Credentials Grant`:** Ideal para comunicación **máquina a máquina (M2M)**. No hay un usuario humano involucrado; una aplicación se identifica ante la tuya usando un `client_id` y `client_secret`.
3. **`Password Grant (Obsoleto)`:** Permitía enviar usuario/`password` directamente. Se desaconseja totalmente en favor de los anteriores por razones de seguridad.
4. **`Personal Access Tokens`:** Permite a los usuarios generar tokens manualmente desde su panel de control para usarlos en scripts o herramientas de línea de comandos.

### `Scopes` (Alcances)
Passport permite definir **Scopes**, que son permisos granulares asociados a un token. Esto asegura que una aplicación externa solo pueda hacer aquello para lo que fue autorizada.

```php
// Definición en AuthServiceProvider
Passport::tokensCan([
    'view-reports' => 'Ver informes de ventas',
    'manage-inventory' => 'Modificar stock de productos',
]);
```

## Analogía
Imagina que tu aplicación es un **País**.
- **`Sanctum`** es como un **DNI o Cédula**: Te sirve para moverte dentro de tu propio país y hacer trámites internos fácilmente.
- **`Passport`** es, literalmente, un **Pasaporte con Visas**.
    - Si un ciudadano de otro país (App externa) quiere entrar al tuyo, debe ir al **Consulado** (`Authorization Server`).
    - Tú (el Usuario) le das permiso.
    - El Consulado le pone una **Visa (`Scope`)** en su pasaporte que dice: "Solo puede visitar museos (`view-only`), no puede trabajar (`update`)".

## Buenas Prácticas
- **Seguridad de `Keys`:** `Passport` genera claves `RSA` para firmar los tokens. Asegúrate de que los archivos `oauth-private.key` y `oauth-public.key` estén fuera de tu repositorio Git.
- **Uso de `UUIDs`:** Si tu aplicación maneja millones de registros o integraciones externas, considera usar `UUIDs` para los `client_id` para evitar la enumeración de recursos.
- **Vida útil de Tokens:** Configura tiempos de expiración cortos para los `access_tokens` y usa `refresh_tokens` para renovarlos. Esto limita el daño si un token es interceptado.
- **Consume tus propias `APIs`:** Si usas Passport, puedes usar el middleware `CreateFreshApiToken` para que tu propio `frontend` pueda consumir la `API` mediante cookies de sesión, evitando manejar tokens en el lado del cliente (`Javascript`).
