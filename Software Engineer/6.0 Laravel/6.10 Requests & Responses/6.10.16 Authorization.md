## Definición
Es la capa lógica encargada de validar permisos sobre recursos. Laravel implementa este proceso mediante dos aproximados principales: `Gates` (puertas), que son ideales para acciones simples o globales, y `Policies` (políticas), que son clases dedicadas a organizar la lógica de autorización alrededor de modelos de `Eloquent` específicos.

## Conceptos Clave
- `Gates`: `Closures` que determinan si un usuario puede realizar una acción. Se definen normalmente en el `AuthServiceProvider`.
- `Policies`: Clases que organizan la lógica de autorización para un modelo específico (ej. `PostPolicy` para el modelo `Post`).
- `Method can`: Método disponible en el modelo `User` para verificar permisos en el código.
- `Directive @can`: Herramienta de `Blade` para mostrar u ocultar elementos de la interfaz basados en permisos.
- `Middleware can`: filtro a nivel de ruta para bloquear accesos no autorizados antes de llegar al controlador.
- `Method authorize`: `Helper` disponible en los controladores para lanzar una excepción `403 Forbidden` automáticamente si falla la validación.

## Ejemplos Prácticos
##### Definición de un `Gate` (Simple/Global)
```php
// app/Providers/AuthServiceProvider.php
use Illuminate\Support\Facades\Gate;

public function boot(): void
{
    // Simple gate to check if the user is an administrator
    Gate::define('access-admin-panel', function ($user) {
        return $user->role === 'admin';
    });
}
```

##### Creación y uso de una `Policy`
```php
// Generating the policy: php artisan make:policy PostPolicy --model=Post

// app/Policies/PostPolicy.php
public function update(User $user, Post $post): bool
{
    // Only the owner or an admin can update the post
    return $user->id === $post->user_id || $user->role === 'admin';
}

// Using it in a Controller
public function edit(Post $post)
{
    $this->authorize('update', $post); // Throws 403 if unauthorized
    return view('posts.edit', compact('post'));
}
```

##### Verificación en `Blade`
```php
@can('update', $post)
    <button>Edit Post</button>
@endcan
```

## Analogía
Imagina un Hotel de Lujo:
- `Authentication`: Es la recepción. Das tu identificación, comprueban que tienes una reserva y te dan una llave.
- `Authorization`: Es la **llave electrónica**. Aunque la llave demuestra que eres un huésped (estás autenticado), no te permite abrir cualquier puerta. Si intentas entrar al penthouse o al spa privado sin haber pagado por ello, la cerradura electrónica (`Gate/Policy`) se pondrá en rojo y te denegará el paso.

## Buenas Prácticas
- **Políticas por defecto**: Siempre que una acción esté vinculada a un modelo de `Eloquent`, usa una **`Policy`** en lugar de un **`Gate`**. Mantiene el código limpio y el `AuthServiceProvider` ligero.
- **Usa el método `authorize`:** En los controladores, prefiere `$this->authorize()` o `Gate::authorize()`. Esto asegura que Laravel maneje la excepción `403` y la respuesta de forma estandarizada.
- **Método `before`:** En tus `Policies`, utiliza el método `before` para conceder acceso total a super-administradores sin tener que repetir la lógica en cada método.
- **Principio de Responsabilidad Única:** No pongas lógica de negocio dentro de la autorización (como enviar correos o actualizar campos). Su única misión es devolver `true` o `false`.

## Escenarios y Aplicaciones
- **Control de Propiedad:** El escenario más común; asegurar que solo el creador de un comentario pueda editarlo o borrarlo.
- **Sistemas de Roles (`RBAC`):** Restringir el acceso a paneles de administración o configuraciones críticas solo a ciertos roles.
- **Limitaciones de Plan (`SaaS`):** Autorizar la creación de recursos solo si el usuario no ha excedido el límite de su suscripción actual.
- **Filtros de Interfaz:** Ocultar botones de "Eliminar" para usuarios que solo tienen permisos de lectura.

## Notas Personales
- Es común confundir `Authorization` con `Authentication` al principio. Recordar: `Authn` = Identidad, `Authz` = Permisos.
- Un `"gotcha"` frecuente es olvidar registrar la `Policy` si no estás siguiendo las convenciones de nombres de Laravel. Si tu modelo es `Post`, tu política sebe ser `PostPolicy` para que el `auto-discovery` funcione.
- La autorización es la base de la `Clean Architecture` en la capa de datos; evita que se filtran permisos en las consultas `SQL` al delegar esa validación al objeto de usuario.
