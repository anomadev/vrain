## Definición
Las `Policies` son clases de PHP que organizan la lógica de autorización en torno a un modelo de `Eloquent` específico (como `User`, `Post` o `Invoice`). Actúan como un "contrato de permisos" para un recurso, donde cada método de la clase corresponde a una acción del controlador (`view`, `create`, `update`, `delete` ), permitiendo que Laravel verifique automáticamente si el usuario tiene derecho a interactuar con esa instancia específica del modelo.

## Conceptos Clave
- **`Policy Discovery`**: Laravel asocia automáticamente modelos con sus políticas si siguen la convención (Modelo en `app/Models` y Política en `app/Policies` con el sufijo `Policy`).
- **Métodos `CRUD`**: Métodos estándar como `viewAny`, `view`, `create`, `update`, `delete`, `restore` y `forceDelete`.
- **Método `before()`**: Un "comodín" que se ejecuta antes de cualquier otro método de la política. Ideal para otorgar acceso total a `Super Administradores`.
- **`Middleware can`**: Permite proteger rutas verificando la política antes de ejecutar el controlador.
- **`Dependecy Injection`**: Laravel inyecta automáticamente al usuario autenticado como primer argumento en cada método de la política.

## Ejemplos Prácticos
##### Crear una `Policy` con modelo vinculado
```
# Generating a policy linked to a model
php artisan make:policy PostPolicy --model=Post
```

##### Lógica dentro de la `Policy`
```php
// app/Policies/PostPolicy.php
namespace App\Policies;

use App\Models\Post;
use App\Models\User;

class PostPolicy
{
    /**
     * Determine if the user can update the post.
     * Only the author or an admin can update.
     */
    public function update(User $user, Post $post): bool
    {
        return $user->id === $post->user_id || $user->is_admin;
    }
}
```

##### Uso en el `Controller` y `Blade`
```php
// In a Controller method
public function edit(Post $post)
{
    // Throws 403 Forbidden if the policy returns false
    $this->authorize('update', $post);

    return view('posts.edit', compact('post'));
}
```

```php
{{-- In a Blade view --}}
@can('update', $post)
    <button>Edit Post</button>
@endcan
```

## Analogía
Imagina el **Registro de la Propiedad**.
- **Autenticación**: Es mostrar tu identificación para entrar al edificio.
- **`Policy`**: Es el **expediente de una casa específica**. Cuando quieres hacer una remodelación (`Update`), el notario (`Laravel`) no solo verifica que seas un ciudadano válido, sino que consulta el expediente (`Policy`) de esa casa (`Modelo`) para confirmar que tu nombre aparece en las escrituras. No importa cuántas casas haya en la ciudad; cada una tiene sus propias reglas de acceso y propiedad definidas en su expediente.

## Buenas Prácticas
- **Usar el `flag` `--model`**: Siempre genera políticas vinculadas a un modelo para que Laravel pre-rellene los métodos estándar y facilite el _`discovery`_.
- **Filtros Globales con `before`**: Define la lógica de "Dios" (`Super Admin`) una sola vez en el método `before` para evitar repetir `|| $user->is_admin` en cada método.
- **No inyectar lógica de negocio**: Una política debe responder "¿Puede?", no "¿Cómo lo hace?". No actualices registros ni envíes emails dentro de una `Policy`.
- **Manejo de Invitados**: Si permites que usuarios no autenticados vean recursos, define el parámetro `User` como opcional (`?User $user`) en la firma del método.

## Escenarios
- **Propiedad de Recursos (Ownership)**: El uso más común. Verificar que un usuario solo pueda editar sus propios posts o comentarios.
- **`Multi-tenant SaaS`**: Asegurar que un usuario solo pueda ver facturas o clientes que pertenecen a su empresa (`tenant_id`).
- **Sistemas de Jerarquía**: Permitir que un mánager vea los reportes de sus subordinados, pero no los de otros departamentos.
- **Control de Estado**: Impedir que un pedido sea cancelado si ya ha sido enviado (`$order->status !== 'shipped'`).

## Notas Personales
- El método `viewAny` es engañoso. No filtra la consulta de la base de datos; solo decide si el usuario puede entrar a la página de índice. El filtrado de datos debe hacerse con `Query Scopes`
- Si nombre del método es la política no coincide con el del controlador (ej. `publish`), puedes llamarlo explícitamente: `$this->authorize('publish', $post)`.
- Las `Policies` son el lugar ideal para mantener el código `DRY` cuando los mismos permisos se usan en Controladores, `Blade` y `APIs`.
