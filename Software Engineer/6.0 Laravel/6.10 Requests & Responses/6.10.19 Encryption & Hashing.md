## Definición
La `Encriptación` es un proceso de dos vías (bidireccional): transforma un texto plano en un mensaje ilegible que puede ser revertido a su estado original mediante una clave secreta. El `Hashing`, por el contrario, es un proceso de una sola vía (unidireccional): genera una "huella digital" fija de un datos que es matemáticamente imposible de revertir; su propósito no es recuperar el dato, sino verificar que coincida.

## Conceptos Clave
- `APP_KEY`: La clave de 32 caracteres definida en `.env`. Es el corazón de la encriptación en Laravel; si se pierde, los datos encriptados son irrecuperables.
- `AES-256` y `AES-128`: Los algoritmos de cifrado simétrico que utiliza Laravel (`OpenSSL`).
- `Bcrypt` y `Argon2`: Algoritmos de `hashing` diseñados para ser lentos y resistentes a ataques de fuerza bruta. `Argon2` es el estándar actual recomendado por `OWASP`.
- `Salt (Sal)`: Valor aleatorio añadido automáticamente por Laravel al `hashing` para asegurar que dos contraseñas iguales produzcan `hashes` diferentes.
- `Payload`: El paquete que contiene el dato cifrado, el `IV` (Vector de Inicialización) y el `MAC` (Código de Autenticación de Mensaje) para asegurar que el dato no fue manipulado.

## Ejemplos Prácticos
##### Encriptación `(Two-way)`
```php
use Illuminate\Support\Facades\Crypt;

// Encrypting a sensitive value (e.g., an API token)
$encrypted = Crypt::encryptString('secret-api-token');

// Decrypting the value back to plain text
try {
    $decrypted = Crypt::decryptString($encrypted);
} catch (DecryptException $e) {
    // Handle cases where the key or payload is invalid
}
```

##### `Hashing (One-way)`
```php
use Illuminate\Support\Facades\Hash;

// Creating a secure hash for a password
$hashedPassword = Hash::make('user-password');

// Verifying a password during login
if (Hash::check('plain-text-password', $hashedPassword)) {
    // The passwords match
}

// Checking if the hash needs to be updated (e.g., algorithm changed)
if (Hash::needsRehash($hashedPassword)) {
    $hashedPassword = Hash::make('plain-text-password');
}
```

## Analogía
- **Encriptación (El Cofre)**: Metes una carta en un cofre y lo cierras con una llave. Puedes enviarlo a cualquier parte y, mientras el destinatario tenga la misma llave, podrá abrirlo y leer la carta original.
- **`Hashing` (La Hamburguesa)**: Metes una vaca (el dato) en una procesadora industrial. El resultado es carne picada (el `hash`). No importa cuánto lo intentes, no puedes "des-picar" la carne para reconstruir la vaca. Sin embargo, si metes exactamente la misma vaca, la hamburguesa resultante será idéntica, lo que te permite comparar resultados.

## Buenas Prácticas
- **Jamás cambies el `APP_KEY` en producción**: Si lo haces, todos los datos encriptados en tu base de datos (y las sesiones de usuario) quedarán inutilizados.
- **Usa el `cast` `hashed`**: En Laravel 10/11+, define tus campos de contraseña en el modelo usando `protected function casts(): array { return ['password' => 'hashed']; }` para automatizar el proceso.
- **`PII (Personally Identifiable Information)`**: Encripta datos sensibles como números de seguridad social o números de teléfono si el negocio requiere leerlos de vuelta, pero recuerda que no podrás buscar eficientemente por esos campos (`WHERE`) en la base de datos.
- **`Hashing` para todo lo que no necesites leer**: Si el dato solo sirve para validación (como contraseñas o `tokens` de un solo uso), usa **`Hash`**, nunca encriptación.

## Escenarios y Aplicaciones
- **Encriptación**: Almacenamiento de credenciales de terceros `(API Keys` de clientes), datos médicos o financieros que deben mostrarse al usuario más tarde.
- **`Hashing`**: Contraseñas de usuario, `tokens` de recuperación de cuenta, verificación de integridad de archivos.

## Notas Personales
- Las búsquedas en la base de datos sobre campos encriptados son imposibles mediante `SQL` estándar (ya que el resultado cambia con cada encriptación debido al `IV`). Si necesitas buscar, considera usar un `Blind Index` (un `hash` del dato original guardado en una columna separada).
- La encriptación de Laravel es **determinística en integridad paro no en salida**: Encriptar el mismo `string` dos veces resultará en dos cadenas diferentes, pero ambas se desencriptarán al mismo valor. Esto es una característica de seguridad para evitar ataque de análisis de patrones.
