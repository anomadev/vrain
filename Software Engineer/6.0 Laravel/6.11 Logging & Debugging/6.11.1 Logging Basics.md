## Definición
El `logging` es el proceso de registrar los eventos, errores y mensajes informativos generados por la aplicación durante su ejecución. En Laravel, este sistema está centralizado en el archivo de configuración `config/logging.php`, permitiendo enviar mensajes a múltiples destinos (canales) como archivos locales, servicios en la nube o herramientas de mensajería instantánea.

## Conceptos Clave
- **`Monolog`**: La librería PHP subyacente que maneja la lógica de envío y formateo de `logs`.
- **`Channels` (Canales)**: Destinos específicos para los `logs` (ej. `stack`, `single`, `daily`, `slack`, `syslog`).
- **`Stacks`**: Un canal especial que agrupa múltiples canales, permitiendo enviar un mismo mensaje a varios destinos simultáneamente.
- **Niveles de `Log (RFC 5424)`**: Estándar de severidad que incluye 8 niveles: `emergency`, `alert`, `critical`, `error`, `warning`, `notice`, `info`, y `debug`.
- **Contexto (`Context`)**: Información adicional en forma de `array` que se adjunta al mensaje para facilitar el depurado.

## Ejemplos Prácticos
##### Uso básico de la `Facade Log`
```php
use Illuminate\Support\Facades\Log;

// Recording different severity levels
Log::info('A user has logged in.', ['user_id' => $user->id]);
Log::error('Connection to payment gateway failed.', ['gateway' => 'Stripe']);
Log::debug('SQL Query execution time', ['ms' => 120]);
```
##### Escritura en canales específicos 
```php
// Sending a log only to the 'slack' channel instead of the default stack
Log::channel('slack')->critical('The server is running out of disk space!');
```
##### Registro de excepciones
```php
try {
    // Some risky operation
} catch (Exception $e) {
    Log::error('Operation failed', [
        'message' => $e->getMessage(),
        'file' => $e->getFile(),
        'line' => $e->getLine()
    ]);
}
```

## Analogía
Imagina que tu aplicación es un **Barco en alta mar**.
- El `Logging` es el Diario de Bitácora del capitán.
- Si el cocinero se queda sin sal, es una `info` o `notice`.
- Si hay una tormenta ligera, es un `warning`.
- Si el motor se detiene, es un `critical`.
- Si el barco se está hundiendo, es un `emergency`.
El capitán no solo anota el problema, sino las coordenadas y la hora (Contexto) para que, si el barco naufraga, los investigadores puedan reconstruir los hechos.

## Buenas Prácticas
- **Niveles Adecuados**: No uses `error` para cosas que no detienen el flujo; usa `info` o `notice`. Reserva `emergency` para fallos totales del sistema.
- **`Log Rotation`**: Usa el canal `daily` en lugar de `single` para evitar archivos gigantescos de varios GB que saturen el disco.
- **No `Loguear` Datos Sensibles**: Jamás registres contraseñas, números de tarjeta de crédito o `tokens` de sesión (`PII`).
- **Información de Contexto**: Un `log` que dice "Error en el sistema" no sirve de nada. Usa siempre el `array` de contexto para incluir `IDs` de modelos o estados de variables.

## Escenarios y Aplicaciones
- **Depuración en Producción**: Identificar por qué un `job` falló sin tener acceso a un entorno interactivo.
- **Auditoría de Seguridad**: Registrar intentos fallidos de `login` o cambios de permisos en usuarios.
- **Monitoreo de Rendimiento**: Registrar tiempos de respuesta de `APIs` externas que son críticas para el negocio.
- **Alertas Críticas**: Configurar el canal de `Slack` para que el equipo reciba notificaciones inmediatas ante errores 500 recurrentes.

## Notas Personales
- El archivo `laravel.log` puede crecer muy rápido si dejas el nivel en `debug` en producción. Asegúrate de que `LOG_LEVEL` es tu `.env` sea `info` o `error` en entornos productivos.
- El canal `null` es muy útil para `tests` unitarios cuando no quieres que la ejecución de las pruebas llene tus archivos de `logs` reales.
- Si usas microservicios, considera un canal `syslog` o integraciones con `ELK Stack` (`Elasticsearch`, `Logstash`, `Kibana`) para centralizar `logs` de múltiples fuentes.
