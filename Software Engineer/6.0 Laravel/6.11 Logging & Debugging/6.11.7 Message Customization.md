## Definición
El renderizado de excepciones es el mecanismo mediante el cual Laravel convierte una excepción lanzada en el código en una respuesta `HTTP`. Personalizar este proceso permite interceptar tipos específicos de errores (como un `InvalidOrderException`) para devolver una vista personalizada, un mensaje de error específico o una estructura `JSON` estructurada, en lugar de la página de error genérica del `framework`.

## Conceptos Clave
- **`Exception Handler`**: Centraliza la lógica de reporte y renderizado. En Laravel 11+, esto se configura en `bootstrap/app.php` usando el método `withExceptions`.
- **`renderable`**: Un método o clausura que defina cómo se debe mostrar una excepción específica al usuario.
- **`reportable`**: Se utiliza para ejecutar lógica adicional (como enviar un `log` o una alerta a `Sentry`) sin detener el proceso de renderizado.
- **Contexto de la excepción**: Datos adicionales que se pueden adjuntar a la excepción para que el `Hanlder` tenga más información al decidir cómo renderizarla.
- **`Mapping`**: La capacidad de "mapear" una excepción propia a un código de estado `HTTP` o a una respuesta personalizada.

## Ejemplos Prácticos

##### Personalización en `bootstrap/app.php` (Laravel `11+`)
```php
// bootstrap/app.php
->withExceptions(function (Exceptions $exceptions) {
    // Customizing the response for a specific exception type
    $exceptions->render(function (NotFoundHttpException $e, Request $request) {
        if ($request->is('api/*')) {
            return response()->json([
                'message' => 'The requested resource was not found in our API.',
                'status' => 404
            ], 404);
        }
    });
})
```

##### Excepciones que se renderizan a sí misma (`Self-Rendering`)
```php
namespace App\Exceptions;

use Exception;
use Illuminate\Http\Request;
use Illuminate\Http\Response;

class BusinessLogicException extends Exception
{
    /**
     * Render the exception into an HTTP response.
     */
    public function render(Request $request): Response
    {
        return response()->view('errors.business', [
            'message' => $this->getMessage()
        ], 422);
    }
}
```

## Analogía
Imagina un Traductor en una embajada. Cuando un ciudadano extranjero (el código) comete una infracción o tiene un problema, el traductor (`Exception Handler`) escucha la queja técnica en su idioma original. Luego, el traductor decide cómo comunicarlo al oficial local (el usuario):
-  Si el oficial es Diplomático (Navegador Web), el traductor le entrega una carta formal y elegante (Vista `HTML`).
- Si el oficial es un Agente de Logística (`API`), el traductor le entrega una ficha técnica con códigos y datos precisos (`JSON`).

## Buenas Prácticas
- **Identificar el Cliente**: Siempre verifica si la petición espera JSON (`$request->expectsJson()`) antes de devolver una respuesta personalizada, para no romper la integración con el `frontend`.
- **Excepciones Personalizadas**: Para errores de negocio complejos, crea clases de excepción dedicadas en lugar de lanzar una `Exception` genérica. Esto facilita el filtrado en el `Handler`.
- **No revelar secretos**: En el renderizado personalizado, asegúrate de no incluir fragmentos de SQL o rutas internas del servidor que puedan comprometer la seguridad.
- **Usar el método `render` en la clase**: Si una excepción siempre debe mostrarse de la misma forma, define el método `render` dentro de la propia clase de excepción para mantener el `Handler` limpio.

## Escenarios y Aplicaciones
- **`APIs` Uniformes**: Asegurar que todos los errores de la API sigan el mismo formato JSON `{ "error": "msg", "code": 400 }`.
- **Manejo de Errores de Pago**: Capturar excepciones de pasarelas (`Stripe`/`PayPal`) y redirigir al usuario a una página de "Pago Fallido" con instrucciones claras.
- **Acciones Expiradas**: Manejar el `TokenMismatchException` (CSRF) para redirigir al usuario al `login` con un mensaje amigable en lugar de mostrar un error `"419 Page Expired"` seco.

## Notas Personales
- Si personalizas el renderizado de una excepción básica (como `NotFoundHttpException`), podrías accidentalmente deshabilitar las páginas de error personalizadas de la carpeta `resources/views/errors`.
- El orden de las clausuras en el `Handler` importa: Laravel ejecutará la primera que coincida con el tipo de excepción lanzada.
- En el desarrollo de SaaS, esta técnica es vital para manejar errores de "Suscripción Expirada" sin ensuciar cada controlador con validaciones `if/else`.
