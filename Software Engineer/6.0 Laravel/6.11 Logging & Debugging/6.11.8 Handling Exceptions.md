## Definición
El manejo de excepciones es el proceso de interceptar, procesar y responder a errores inesperados que ocurren durante la ejecución de una aplicación. Laravel centraliza este comportamiento, permitiendo decidir de manera granular qué errores deben registrarse en los `logs` (`Report`) y qué respuesta debe ver el usuario final (`Render`), evitando que la aplicación colapse o exponga datos sensibles.

## Conceptos Clave
- **`reportable`**: Clausura que permite ejecutar lógica personalizada cuando ocurre una excepción (ej. enviar una alerta a `Sentry` o `Slack`) sin detener el flujo de renderizado.
- **`renderable`**: Clausura que define cómo se debe transformar una excepción en una respuesta `HTTP` (`HTML` o `JSON`).
- **`stop (Ignore)`**: Capacidad de indicar al `framework` que ignore ciertas excepciones para que no sean reportadas ni procesadas por el manejador global.
- **`Custom Exceptions`**: Clases que heredan de `Exception` y pueden contener sus propios métodos `report` y `render`, permitiendo que el objeto se "maneje a sí mismo".
- **`Global Handler`**: En Laravel `11+`, se configura en `bootstrap/app.php` mediante el método `withExceptions`.

## Ejemplos Prácticos

##### Manejo centralizado (Laravel `11+`)
```php
// bootstrap/app.php

->withExceptions(function (Exceptions $exceptions) {
    // Reporting specific exceptions to external services
    $exceptions->reportable(function (InvalidOrderException $e) {
        // Custom reporting logic (e.g., Slack notification)
    });

    // Custom rendering for API clients
    $exceptions->render(function (Throwable $e, Request $request) {
        if ($request->is('api/*')) {
            return response()->json([
                'error' => 'An unexpected error occurred.',
                'code' => $e->getCode()
            ], 500);
        }
    });
})
```

##### Ignorar excepciones comunes
```php
->withExceptions(function (Exceptions $exceptions) {
    $exceptions->stopReporting([
        \App\Exceptions\UserNotPremiumException::class,
    ]);
})
```

##### Excepción con lógica interna (`Self-Handling`)
```php
namespace App\Exceptions;

use Exception;

class ServiceUnavailableException extends Exception
{
    public function report(): void
    {
        // Automatically log this specific service failure
    }

    public function render($request)
    {
        return response()->view('errors.service-down', [], 503);
    }
}
```

## Analogía
Imagina el **Sistema de Triage en una Sala de Urgencias**:
1. **`Excepción`**: Un paciente llega con un problema.
2. **`Reportable` (El Registro)**: El administrativo anota el caso en la base de datos del hospital para estadísticas.
3. **`Renderable` (El Tratamiento)**: El médico decide si el paciente necesita una curación rápida (Mensaje de error) o una cirugía (Página de error 500).
4. **`Ignore`**: Si el paciente llega solo por un rasguño menor (Excepción trivial), el hospital decide no registrarlo en las estadísticas críticas para no saturar el sistema.

## Buenas Prácticas
- **No captures todo**: Evita el uso excesivo de bloques `try-catch` genéricos. Deja que las excepciones "burbujeen" hasta el `Global Handler` para mantener tus controladores limpios.
- **Tipado Estricto**: Usa excepciones específicas (`PaymentFailedException`) en lugar de genéricas (`Exception`). Esto permite aplicar lógica de recuperación distinta para cada caso.
- **Entornos de ejecución**: Asegúrate de que las excepciones detalladas solo se rendericen cuando `APP_DEBUG=true`.
- **Contexto en `Logs`**: Si usas `reportable`, adjunta el ID del usuario o el `payload` de la petición para que el `debug` sea más sencillo.

## Escenarios y Aplicaciones
- **`APIs` Externas**: Capturar fallos de conexión con servicios de terceros y devolver una respuesta amigable al `frontend` en lugar de un error de `cURL`.
- **Lógica de Negocio**: Lanzar excepciones cuando un usuario intenta realizar una acción sin saldo o permisos, centralizando la respuesta visual en el `Handler`.
- **`Jobs` en Cola**: Manejar excepciones en procesos asíncronos para decidir si el `Job` debe reintentarse o marcarse como fallido definitivamente.

## Notas Personales
- Las excepciones lanzadas dentro de un `Job` en cola (`Queued Job`) son capturadas por el `Worker`. Si el `Handler` las "detiene" (`stopReporting`), podrías no enterarte nunca de por qué fallan tus procesos en segundo plano.
- No uses excepciones para controlar el flujo normal de la aplicación (ej. redirecciones de login). Las excepciones son costosas en términos de rendimiento y deben reservarse para estados verdaderamente excepcionales.
- La integración de `reportable` con servicios como `Sentry` o `Bugsnag` es lo que separa a una aplicación amateur de una profesional en producción.
