## Definición
El `testing` automatizado consiste en escribir `scripts` que verifiquen si el código se comporta como se espera. En Laravel, esto se divide principalmente en dos niveles: `Unit Tests` (Pruebas Unitarias), que validan una pieza pequeña y aislada de lógica (una sola clase o método), y `Feature Tests` (Pruebas de Funcionalidad), que verifican el comportamiento de una característica completa, involucrando múltiples componentes como rutas, controladores y bases de datos.

## Conceptos Clave
- **`PHPUnit & Pest`**: Las herramientas base. Laravel `11/12` utiliza `Pest` por defecto por su sintaxis expresiva, aunque mantiene compatibilidad total con `PHPUnit`.
- **`AAA Pattern (Arrage, Act, Assert)`**: La estructura estándar para escribir cualquier `test`: Preparar el escenario, ejecutar la acción y verificar el resultado.
- **`Mocks & Fakes`**: Objetos que simulan el comportamiento de servicios reales (como el envío de emails o una `API` de pagos) para evitar efectos secundarios durante las pruebas.
- **`Database Migrations & Refresh`**: Laravel utiliza un `trait` (`RefreshDatabase`) para asegurar  que cada `test` se ejecute en un entorno de base de datos limpio y controlado.
- **`Assertions`**: Métodos que comprueben condiciones (ej.: `assertStatus(200)`, `assertSee('Welcome`)`).

## Ejemplos Prácticos

##### `Unit Test (Isolation)`
```php
// Isolated logic: no DB, no HTTP
test('it can calculate the total price with tax', function () {
    $calculator = new PriceCalculator();
    
    // Act
    $total = $calculator->calculate(100, 0.16); // 100 + 16% tax
    
    // Assert
    expect($total)->toBe(116.0);
});
```

##### `Feature Test (Full Cycle)`
```php
// Testing a complete request
use function Pest\Laravel\{post, get};
use App\Models\User;

test('authenticated users can create a post', function () {
    // Arrange: Create a user and act as him
    $user = User::factory()->create();
    
    // Act: Send a POST request to the endpoint
    $response = $this->actingAs($user)
        ->post('/posts', [
            'title' => 'My New Post',
            'body' => 'Content of the post',
        ]);
        
    // Assert: Check DB and Response
    $response->assertStatus(201);
    $this->assertDatabaseHas('posts', ['title' => 'My New Post']);
});
```


## Analogía
Imagina que estás construyendo un Coche:
- **`Unit Test`:** Es probar una bujía fuera del motor. Te aseguras de que genere chispa cuando recibe corriente. No te importa el resto del coche, solo la bujía.
- **`Feature Test`:** Es girar la llave en el contacto. Quieres ver si la bujía hace chispa, el motor arranca, el tablero se enciende y el coche está listo para conducir. Pruebas la interacción de todas las piezas juntas.

## Buenas Prácticas
- **Pareto en `Testing`**: El 20% de tus `test` (generalmente `Feature Test`) cubrirán el 80% de los fallos más críticos en producción. Prioriza probar rutas y flujos críticos de negocio.
- **`Test` de Regresión**: Cada vez que encuentres un `bug`, escribe un `test` que lo produzca antes de arreglarlo. Así te aseguras de que nunca vuelva a ocurrir.
- **`Tests` Rápidos**: Los `Unit Tests` deben ser instantáneos. Si un `Unit Test` toca la base de datos o hace una petición web, ya no es un `Unit Test`; es un `Integration` o `Feature Test`.
- **`Clean Architecture`**: Escribir código fácil de `testear` suele ser sinónimo de buen diseño. Si un método es difícil de probar, probablemente tiene demasiadas responsabilidades (rompe el `SRP` de `SOLID`).

## Escenarios y Aplicaciones
- **Refactorización:** Quieres cambiar Laravel MVC a una arquitectura de capas. Si tienes `Feature Tests`, puedes cambiar toda la lógica interna y saber que el resultado final sigue siendo el mismo.
- **`CI/CD`:** Automatizar los `tests` en `GitHub Actions` para prohibir el `"merge"` de cualquier código que rompa una funcionalidad existente.
- **`TDD (Test Driven Development)`:** Escribir el test antes que el código para definir claramente qué debe hacer la funcionalidad antes de implementarla.

## Notas Personales
- Un error común es intentar `testearlo` todo al 100%. El `"Code Coverage"` del 100% es una métrica de vanidad. Enfócate en la lógica de negocio y los flujos donde se mueve el dinero o los datos sensibles.
- `Pest` ha ganado la batalla por su limpieza visual, pero entender `PHPUnit` es fundamental porque es el motor que corre por debajo.
- En el desarrollo de un `SaaS`, los `Feature Tests` son vitales para asegurar que las actualizaciones no rompan el proceso de suscripción o pago de los clientes.
