## Definición
`Pest` es un `framework` de pruebas para PHP con un enfoque funcional, construido sobre la robustez de `PHPUnit`. Su objetivo principal es mejorar le experiencia del desarrollador (`DX`) mediante una sintaxis minimalista, fluida y altamente expresiva. En Laravel, `Pest` es ahora el motor de pruebas predeterminado, integrándose perfectamente con sus funcionalidades de `mocking`, base de datos y aserciones.

## Conceptos Clave
- **`Expectations API`**: Un conjunto de métodos encadenables que comienzan con `expect($value)` y permiten realizar aserciones de forma muy legible (ej. `->toBe()`, `->toContain()`).
- **`Functional Syntax`**: En lugar de definir clases y métodos `public function test_something()`, se utilizan las funciones globales `test()` o `it()`.
- **`Architecture Testing`**: Una característica única de `Pest` que permite validar reglas de arquitectura, como asegurar que no se usen `dd()` o que los controladores no dependan directamente de modelos.
- **`Higher Order Expectations`**: Permite realizar acciones y aserciones directamente sobre el objeto que se está probando sin variables intermedias.
- **`Plugins`**: Ecosistema extendido que incluye herramientas para cobertura de código(`--coverage`), vigilancia de archivos (`--watch`) y `tests` paralelos.

## Ejemplos Prácticos

##### `Test` Básico con `Expectations`
```php
// testing a simple logical output
test('it can sum two numbers', function () {
    $result = 5 + 5;

    expect($result)->toBe(10)
        ->toBeInt()
        ->not->toBeString();
});
```

##### `Feature Test` en Laravel
```php
// Using 'it' for more natural language (it should...)
it('allows a user to view the homepage', function () {
    $this->get('/')
        ->assertStatus(200)
        ->assertSee('Welcome');
});
```

##### `Architecture Testing (SOLID enforcement)`
```php
// Ensuring architectural integrity
test('controllers do not access models directly')
    ->expect('App\Http\Controllers')
    ->not->toUse('App\Models');

test('no debugging statements left in code')
    ->expect(['dd', 'dump', 'ray'])
    ->not->toBeUsed();
```

## Analogía
Imagina que escribir `tests` en **`PHPUnit`** es como llenar un **Formulario Notarial**: es formal, requiere una estructura rígida (clases, `namespaces`, métodos públicos) y mucha burocracia visual.
Escribir `tests` en **`Pest`** es como escribir en una **Nota Adhesiva (`Post-it`)**: vas directo al grano, el lenguaje es natural y solo escribes lo que realmente importa. El notario (`PHPUnit`) sigue trabajando en la oficina de atrás revisando todo, pero tú solo interactúas con la simplicidad de la nota.

## Buenas Prácticas
- **Usar `it()` para lenguaje natural:** Prefiere `it('should...')` para pruebas que describan comportamientos y `test('description')` para validaciones técnicas.
- **Agrupar con `describe()`:** Utiliza bloques `describe` para organizar `tests` relacionados, manteniendo el archivo limpio y jerárquico.
- **`Architecture Testing`:** Implementa `tests` de arquitectura desde el inicio del proyecto para garantizar que el equipo respete patrones como `Clean Architecture` o la separación de capas.
- **`Dataset Driven Testing`:** Utiliza el método `with()` para ejecutar el mismo `test` con múltiples juegos de datos, evitando la duplicación de lógica.

## Escenarios y Aplicaciones
- **Prototipado Rápido:** La baja fricción de `Pest` lo hace ideal para escribir pruebas mientras se desarrolla la funcionalidad (`TDD`).
- **Refactorización de Legado:** Gracias a su legibilidad, es más fácil documentar qué hace el código antiguo mediante pruebas escritas en `Pest`.
- **Validación de Estándares:** Ideal para equipos grandes donde se quiere asegurar que nadie use funciones prohibidas o rompa la estructura de carpetas definida.

## Notas Personales
- A veces, al depurar errores profundos en el `framework`, el `stack trace` puede verse alterado por la capa de `Pest`, pero las herramientas de error integradas suelen compensarlo.
