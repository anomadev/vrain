## Definición
`PHPUnit` es un `framework` de pruebas orientado a objetos para PHP, basado en la arquitectura `xUnit`. Su función principal es permitir a los desarrolladores es escribir `"Test Cases"` (clases de prueba) que verifican unidades aisladas de código. En el contexto de Laravel, `PHPUnit` es el motor que ejecuta las pruebas, proporcionando una `suite` completa de herramientas para realizar aserciones, manejar el ciclo de vida de los `tests` y simular dependencias.

## Conceptos Clave
- **`Test Case`**: La clase base de la cual deben heredar todas las clases de prueba. Proporciona los métodos necesarios para realizar verificaciones.
- **`Assertions`** (Aserciones): Métodos que comparan un valor esperado con uno real (ej. `assertEqueals`, `assertTrue`, `assertInstanceOf`). Si la condición no se cumple, el `test` falla.
- **`Fixtures`**: El estado necesario para ejecutar `test`. Se gestiona mediante los métodos `setUp()` (antes de cada `test`) y `tearDown()` (despues de cada `test`).
- **`Data Providers`**: Métodos que permiten ejecutar el mismo `test` múltiples veces con diferentes conjuntos de datos de entrada.
- **`Annotations / Attributes`**: Comentarios especiales (o atributos PHP `8+`) que indican a `PHPUnit` metadatos sobre el `test` (ej. `@test`, `@depends`, `#[DataProvider]`).
- **`Stubs & Mocks`**: Objetos que imitan el comportamiento de clases reales para aislar la unidad que se está probando.

### Ejemplos Prácticos

##### Estructura de una Clase de Prueba
```php
namespace Tests\Unit;

use PHPUnit\Framework\TestCase;
use App\Services\Calculator;

class CalculatorTest extends TestCase
{
    private Calculator $calculator;

    // Configuración inicial antes de cada método de prueba
    protected function setUp(): void
    {
        $this->calculator = new Calculator();
    }

    /** @test */
    public function it_can_add_two_numbers(): void
    {
        $result = $this->calculator->add(10, 5);

        // Verificación del resultado esperado
        $this->assertEquals(15, $result);
    }
}
```

##### Uso de `Data Providers`
```php
/**
 * @dataProvider additionProvider
 */
public function test_add_with_multiple_datasets($a, $b, $expected): void
{
    $this->assertEquals($expected, $this->calculator->add($a, $b));
}

public function additionProvider(): array
{
    return [
        'positive numbers' => [5, 5, 10],
        'negative numbers' => [-1, -1, -2],
        'zero'             => [0, 5, 5],
    ];
}
```

## Analogía
Imagina que eres un **Ingeniero de Calidad en una fábrica de piezas de LEGO**. **`PHPUnit`** es la **Plantilla de Acero** que usas para medir cada pieza nueva.
- **El `Test Case`** es el manual de medidas.
- **La Aserción** es el acto de pasar la pieza por el molde: si encaja perfectamente (`True`), la pieza pasa a producción; si hay una micra de diferencia (`False`), se descarta el lote. No pruebas el juguete armado (eso sería un `Feature Test`), pruebas que cada "botoncito" de la pieza tenga el diámetro exacto para que pueda conectarse con cualquier otra.

## Buenas Prácticas
- **Independencia de los `Tests`:** Un `test` nunca debe depender del resultado de otro. Cada prueba debe ser capaz de ejecutarse de forma aislada.
- **Nombres Descriptivos:** El nombre del método debe explicar claramente qué se está probando (ej. `test_user_cannot_purchase_without_balance`).
- **Una sola Unidad:** Un test unitario debe probar solo una clase. Si necesitas tocar la base de datos o una `API`, usa `Mocking` para esas dependencias.
- **`F.I.R.S.T. Principles`:** Los `tests` deben ser _`Fast`_ (Rápidos), _`Independent`_ (Independientes), _`Repeatable`_ (Repetibles), _`Self-validating`_ (Auto-validados) y _`Timely`_ (Oportunos).

### Escenarios y Aplicaciones
- **Lógica Matemática o de Algoritmos:** Validar cálculos de impuestos, descuentos o transformaciones de datos complejas.
- **Validación de `Services`:** Probar que los servicios de tu aplicación (capa de lógica) procesan la información correctamente antes de guardarla.
- **Librerías y Paquetes:** Si estás creando un paquete para ser distribuido, `PHPUnit` es esencial para asegurar la compatibilidad entre versiones.
- **Refactorización de Código Crítico:** Antes de cambiar una función vital, asegúrate de tener una suite de `PHPUnit` que garantice que el resultado final no cambie.

## Notas Personales
- `PHPUnit` es más "verboso" que `Pest`, pero su estructura de clases es muy útil para aplicar principios de **`Clean Architecture`** y **`SOLID`** en los mismos `tests` (como usar herencia para crear `TestCases` personalizados).
- Un error común es olvidar que Laravel extiende `PHPUnit`. Al testear en Laravel, asegúrate de heredar de `Tests\TestCase` (la de Laravel) para tener acceso a la aplicación, y no directamente de `PHPUnit\Framework\TestCase` a menos que sea una prueba unitaria pura sin dependencias del `framework`.
- El uso de `Data Providers` es la forma más eficiente de aplicar el principio de Pareto en `testing`: con un solo método de prueba puedes cubrir decenas de casos de borde (`edge cases`).
