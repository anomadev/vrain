Los entornos de vista previa permiten crear automáticamente entornos totalmente aislados para cada `pull request`. Esto permite probar los cambios de forma segura, compartir vistas previas con el equipo y detectar problemas antes del `merge`.
Cada aplicación de una organización puede tener sus propias reglas de automatización y flujos de trabajo de vista previa.

> [!info] Los `preview environments` están disponibles para los clientes con planes `Growth` y `Business`

Cuando configuras la automatización de un entorno de vista previa en la aplicación:
1. Escucha las `pull requests` o los eventos de nuevas ramas en el repositorio.
2. Cuando se activa, se crea un nuevo `preview environment`.
3. El entorno se basa en las especificaciones y recursos de otro entorno.
4. El entorno recibe un dominio único para previsualizar los cambios.
5. Si la automatización se activa mediante la `pull-request`
6. Cloud añadirá un comentario al `pull-request` con el enlace del `environment`
7. Por defecto, Cloud despliega el `environment` automáticamente y actualiza el comentario del `pull-request` con un enlace al dominio cuando finaliza el despliegue.
8. El `preview environment` se mantiene de forma independiente hasta que se elimina la rama o cierra la `pull-request`. Si la autodestrucción está habilitada, el `environment` y todos los recursos que se hayan creado se eliminarán automáticamente; de lo contrario, será necesaria una limpieza manual.

### Best Practices
- Si despones de un entorno de desarrollo o pruebas, replícalo para que los `preview environments` compartan su configuración y recursos. De los contrario, cree un `template environment` específicamente para entornos de vista previa.
- Utiliza la `template environment` para personalizar cualquier configuración específica del `preview environment`.
- Utiliza `feature flags` o variables de entorno para controlar el comportamiento de `only-preview`.
- Utiliza datos de prueba predefinidos en lugar de copiar los datos de producción.

> [!warning]  Los preview environment reutilizan la `APP_KEY` del entorno replicado. Evita conectarlos a la bases de datos de producción, cachés u otros recursos confidenciales. No es recomendable replicar el entorno de producción. Si comparte recursos entre entornos, ten en cuenta que ejecutar migraciones, realizar cambios etc. puede afectar a otros entornos.

### Configuring preview environments
1. Dirígete a la organización
2. Selecciona el `environment` en la `application` que quieres configurar
3. Navega a `Settings > Preview Environments` en la configuración de la `application`
4. `Get Started > New Automation` para crear una nueva automatización de `preview environment`

#### Automation settings
- `Automation name`: nombre descriptivo
- `Environment to replicate`: selecciona un entorno existente que usara como plantilla que será usado por cada `preview environment`.
- `Automation triggers on`
	- `New pull requests`: crea automáticamente un `preview environment` cuando una `pull request` es abierta contra la rama objetivo.
	- `New branches`: crea un `preview environment` cuando una nueva rama es agregada.
- `Target branch`: si utiliza disparadores de `pull requests` selecciona la rama que desea escuchar.
- `Filter by branch name` (opcional): ejecuta la automatización únicamente si el nombre de la rama coincide con un patrón específico.

#### Replication options
Elija qué recursos duplicar, reutilizar o ignorar para los `preview environments`:
- **`Compute clusters`**: replica la configuración de computación del `template environment`.
- **`Database`**: Opcional, puede adjuntar una instancia de base de datos.
- **`Cache`**: Opcional, adjunte un recurso de caché. Al reutilizar una instancia de caché en diferentes `preview environments` Laravel Cloud inyectara una variable de entorno `CACHE_PREFIX` para proporcionar aislamiento de clave lógica entre entornos. 
  Si seleccionas `Create new cache` para cada entorno, se aprovisionará una instancia de caché independiente para cada uno y no se requerirá ninguna configuración adicional.
- **`Object storage`**: Opcional, adjunto un almacenamiento de objetos.
- `Web Sockets`: Opcional, adjunta una aplicación `Web Socket`.
- `Custom environment variables`: Especifica las variables que se deben sobrescribir o agregar para los `preview environments`.

#### Deployment options
- **`Auto deploy on trigger`**: Habilita para desplegar automáticamente el `preview environment` al crearlo. De estar deshabilitado se requiere despliegue manual.
- **`Auto destroy environment and resources`**: Habilita para destruir automáticamente el entorno y sus recursos cuando se fusiones o cierre la `pull request`. De estar deshabilitado, los entornos deben eliminarse manualmente.
- **`Initial deploy commands`**: Opcional, especifica los comandos de `shell` que se ejecutarán después del primer despliegue.

#### Security
Puede habilitar la autenticación básica HTTP para `preview environments` creados mediante automatización. Cuando está habilitada, cada entorno requerirá que los visitantes ingresen el nombre de usuario y contraseña que haya configurado antes que el trafico llegue a su aplicación.

### Managing preview environments
- Los entornos activos aparecen en el selector de aplicaciones o en la sección `Recently deployed` de la descripción general de la aplicación.
- Los `preview environments` reciben el nombre del `pull request` o rama.
- Cuando agrega nuevos `commits` a la `pull request`, el `preview environment` asociado se reconstruirá y se volverá a implementar automáticamente.
