Cada entorno desplegado en Laravel Cloud se ejecuta en al menos una instancia de `compute`, a la que se le asigna una cantidad específica de potencia de procesamiento de CPU y RAM disponible.
Es la responsable de gestionar el tráfico HTTP  entrante al entorno. Además puede agregar instancias de `compute workers` al entorno para procesar trabajo en cola o ejecutar otros procesos en segundo plano.

Los recursos informáticos asignado a un entorno se pueden gestionar desde el panel de control de la infraestructura del entorno.

### App Compute Cluster
Los `compute cluster` pueden gestionar el tráfico web, escalar a cero, administrar procesos de larga duración como los `queue workers` e incluso ejecutar tareas programadas. 

#### Laravel Octane
Laravel Cloud facilita la ejecución de tu aplicación mediante `Octane`, que inicia la aplicación una sola vez y la mantiene en memoria para cada solicitud.

#### Inertia SSR
Laravel Cloud también permite servir paginas basadas en `Inertia` mediante renderizado del servidor (SSR), lo que puede mejorar el SEO de tu aplicación. Solo active la opción de `Use Inertia SSR`. A continuación guarde e implemente Laravel Cloud renderizará las plantillas de la aplicación utilizando el motor SSR.
Además, si está utilizando uno de los kits de Laravel, debe actualizar los comandos de compilación de su entorno para usar el comando `npm run build:ssr` en lugar de `npm run build`.

### Worker Clusters
Opcionales, se pueden agregar al entorno. Son una copia identica del `App Compute`, en el sentido de que utilizan el mismo repositorio, rama, versión de PHP, variables de entorno, comando de implementación, base de datos y otros recursos. Sin embargo, cada clúster puede dimensionarse de forma independiente y puede ejecutar un conjunto personalizado de `queue workers` o procesos en segundo plano.
Pueden ser útiles para aplicaciones más grande en las que se desea utilizar el `App Cluster` para gestionar el tráfico web, pero enviar las cargas de trabajo en segundo plano a recursos informáticos separados.

### Autoscaling
Laravel Cloud puede escalar automáticamente los recursos informáticos conectados a su entorno en función de la demanda, el uso de la CPU y el uso de la memoria. Para lograr la máxima flexibilidad, Laravel Cloud actualmente admite tres estrategias de autoescalado diferentes:
- `None`: ejecuta tu aplicación en una sola instancia sin escalado automático.
- `Custom`: escala automáticamente tus clústeres horizontalmente, tanto hacia arriba como hacia abajo, dentro de un rango mínimo y máximo de réplicas especificado.
- `Unlimited`: estala automáticamente hacia arriba y hacia abajo de forma indefinida en función de la demanda.
Tras ajustar la estrategia de autoescalado de su entorno, vuelva a implementarlo para que los cambios surtan efecto.

#### HTTP traffic autoscaling
Para los `app compute clusters`, Laravel Cloud escala automáticamente las aplicaciones HTTP en función de la presión del tráfico en tiempo real, además de los umbrales de CPU y memoria configurados, se escala en función del límite que se alcance primero.
La `Ingress Layer` de Laravel Cloud realiza un seguimiento continuo de las solicitudes activas que está procesando la aplicación y añade réplicas cuando la demanda supera la capacidad disponible.
La capacidad de cada réplica se basa en la cantidad de memoria asignada a la aplicación. Laravel Cloud estima que cada proceso PHP utiliza aproximadamente 30MB de memoria y puede procesar una solicitud a la vez.

Por ejemplo, una aplicación de 512MB admite aproximadamente 17 solicitudes simultáneas por réplica. Si las solicitudes HTTP activas superan esa capacidad, Laravel Cloud lanza réplicas adicionales; cuando el tráfico disminuye, se eliminan las réplicas sobrantes.

Dado que esta señal de escalado se basa en la solicitudes activas en lugar de en métricas residuales de CPU o memoria, los tiempo de respuesta más rápidos reducen la cantidad de réplicas necesarias para atender la misma cantidad de tráfico.

#### Scheduled autoscaling
Disponible en los planes `Business` y `Enterprise`. Permite definir configuraciones predefinidas basadas en el tiempo para clústeres que escalan entre un número mínimo y máximo de réplicas. Útil para patrones de tráfico predecibles, como lanzamientos de productos, campañas de marketing u horarios comerciales.
Cada anulación programada cambia temporalmente la configuración de su clúster:
- Réplicas mínimas
- Réplicas máximas
- Umbral de CPU
- Umbral de memoria
Las anulaciones se pueden configurar como programaciones únicas o recurrentes. Admiten patrones diarios, semanales, fines de semana, etc. cada horario utiliza su propia zona horaria para que estas coincidan con las horas importantes para el equipo.
Si varios horarios se superpones, Laravel Cloud aplica el `override` que coincida más específicamente. Los horarios puntuales tienes prioridad sobre los horarios recurrentes. Entre los horarios recurrentes, los horarios para días específicos de la semana tienen prioridad sobre los horarios de días laborable o fines de semana, y los horarios de días laborables o fines de semana tienen prioridad sobre los horarios diarios.

#### Scheduled autoscalling limitations
- El escalado automático programado solo esta disponible para clústeres que escalan entre un número mínimo y máximo definido.
- Los horarios recurrentes deben comenzar y terminar el mismo día. Si necesita un intervalo nocturno, cree dos horarios.
- Las programaciones de un solo uso se ejecutan una sola vez, y finalizado el período programado, dejan de aplicarse automáticamente.

### Compute classes
Las aplicaciones de Laravel Cloud se ejecutan en la última generación de servidores `AWS Graviton EC2` con una relación coste-rendimiento optimizada. Laravel Cloud ofrece dos clases de computación:
- `Flex`: Ligero y económico, para entornos de desarrollo o aplicaciones que necesitan flexibilidad para soportar picos de tráfico periódicos.
- `Pro`: Capacidades de procesamiento mayores, diseñádas para un uso intensivo y sostenido. Ideales para aplicaciones de producción a gran escala y misión crítica.
Dentro de cada categoría, hay disponibles diferentes tamaños de CPU y RAM. Los precios varían según la región.

### Compute metrics
Las métricas de computación del entorno se pueden consultar accediendo a la pestaña `Metrics`. Al graficar los datos, el uso de CPU y memoria de los clústeres de aplicaciones y `workers` del entorno se promediará en todas las réplicas.

### Scale to Zero
Su entorno puede reducirse a cero si no recibe una solicitud de red entrante a su aplicación dentro de un límite de tiempo especificado, conocido como tiempo de espera de suspensión. Mientras un entorno esté en modo de suspensión, no se le cobrará por su procesamiento hasta que se vuelva a activar.
Los nuevos tamaños de cómputo Flex se activan en menos de 500 milisegundos cuando el entorno recibe una nueva solicitud entrante, lo cual es lo suficientemente rápido como para que los usuarios de la aplicación no lo noten.

### Scheduled tasks and queues
Un entorno en reposo se activa automáticamente para ejecutar las tareas programadas y procesar los trabajos en cola, de modo que nada se quede atrás mientras el entorno está inactivo.
Cuando su entorno se activa para una tarea programada o un intervalo de activación manual, permanece activo durante el tiempo de espera de suspensión. Cualquier solicitud entrante reinicia este temporizador. Asegúrese que las tareas programadas y procesos en segundo plano puedan completarse dentro de ese plazo.

#### Manual wake interval
Puede configurar un intervalo de activación manual para activar un entorno según un horario fijo, lo cual resulta útil para procesos en segundo plano, como los procesos `queue worker` autogestionados que deben ejecutarse periódicamente. Al igual que con las tareas programadas, un intervalo de activación más corto que el tiempo de espera para dormir puede mantener el entorno activo de forma continua.

#### Worker clusters
Cada `worker clusters` puede permanecer activo en todo momento o seguir al clúster de la aplicación y entrar en modo de suspensión junto con él. Puede configurar este comportamiento en los ajustes del `worker clusters`.

#### Limitations
- Solo los tamaños de computo `Flex` pueden escalarse a cero.
- Los tiempos de activación inferiores a 500 ms se aplican únicamente a los nuevos tamaños `Flex`. Los tamaños anteriores conservan su comportamiento de activación previo.

### Maintenance mode
Laravel utiliza el controlador de archivos para el modo de mantenimiento. Si se ejecuta `php artisan down` desde la pestaña de `Commands`, es estado del modo mantenimiento no se mantendrá entre implementaciones ni se aplicará a varias réplicas.
Para garantizar un estado de modo de mantenimiento consistente en toda la aplicación y sus réplicas, configure el `driver` del modo mantenimiento para que utilice caché y use una base de datos o Laravel Valkey como backend del estado del modo de mantenimiento en sus variable de entorno.

```
APP_MAINTENANCE_DRIVER=cache
APP_MAINTENANCE_STORE=database
```
