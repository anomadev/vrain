### Introduction
- **Propósito**: Laravel Cloud permite la ejecución de tarea programadas (`Schedule tasks`) para aplicaciones PHP sin necesidad de que el desarrollador gestione o configure la infraestructura tradicional de `cron` en los servidores.
- **Mecanismos**: El `framework` Laravel cuenta con un sistema nativo para definir la frecuencia de estas tareas (cada minuto, hora, día, etc.). Laravel Cloud se integra con este sistema y permite activar su soporte directamente desde el panel de control.

### Enabling the task scheduler
- **Configuración Básica (`App Cluster`)**: La manera más sencilla de activarlo es ir al clúster de cómputo de la aplicación (`App compute cluster`) en el lienzo de infraestructura, activar el interruptor llamada `Scheduler` y guardar/re-desplegar.
- **Comportamiento por defecto**: Bajo este esquema, las tareas programadas se ejecutan en la mismas instancia de cómputo que atienden el tráfico web entrante (HTTP). Tras el despliegue, el comando `schedule:run` de `Artisan` se invocará automáticamente en el clúster cada minuto.
- **Aislamiento de Cómputo (Análisis Crítico)**: Si no deseas que las tareas pesadas consuman los recursos destinados a atender a los usuario web, la documentación permite activar el `Scheduler` de forma aislada dentro de un clúster de trabajadores dedicado (`Worker cluster`).

### Scheduled tasks and multiple replicas
- **Riesgo de Concurrencia**: Si la infraestructura está configurada para escalar horizontalmente usando múltiples réplicas (instancias), la tareas programadas se dispararán en todas las réplicas simultáneamente por defecto.
- **Solución a nivel de Software**: Para evitar comportamientos inesperados o ejecuciones duplicadas de la misma tarea (como enviar dos veces un reporte), es mandatorio utilizar el método `onOneServer()` en la definición del `schedule` dentro del código de Laravel. Si la intención explícita es que corra en todas las instancias en paralelo, dicho método puede omitirse.

### Scheduled tasks and Scale to Zero
Esta es la sección más densa y crítica a nivel de arquitectura elástica si se utiliza `Scale to Zero` (Escalado a cero):
- **Activación/Despertar Automático**: Cuando el entorno está "dormido" (en cero instancias), Laravel Cloud **despierta automáticamente la aplicación** para ejecutar las tareas en el momento exacto en que están programadas.
- **Mapeo en Tiempo de Despliegue**: Para saber cuándo debe despertar el entorno, Laravel Cloud ejecuta internamente el comando `php artisan schedule:list` únicamente durante el proceso de despliegue (`deploy time`) y almacena el resultado. El sistema calcula las próximas ejecuciones basándose en esa captura fija.
	- **Impacto técnico**: Cualquier cambio que realices en la frecuencia o los horarios de tus tareas en el código fuente no tendrá efecto en la nube hasta que realices un nuevo `deployment`.
- **El Dilema del `Sleep Timeout` y Frecuencias Cortas**: Cuando la app despierta para una tarea, permanece encendida durante todo el tiempo configurado en el `sleep timeout` del entorno.
	- **Regla de optimización**: No debes programar tareas con intervalos de tiempo más cortos que el `sleep timeout`. Por ejemplo, si tu entorno tiene un `sleep timeout` de 5 minutos y programas una tarea que corre cada 5 minutos, la infraestructura se despertará constantemente y nunca alcanzará el estado de reposo (nunca escalará a cero), incrementando los costos innecesariamente.
- **Impacto en el `Driver` de `Caché`**: Laravel Cloud vincula el controlador de caché del planificador directamente al almacén de caché del entorno. El planificador solo consulta este almacén si usas métodos como `withoutOverlapping()`, `onOneServer()` o tareas de sub-minuto.
	- *Si abusas de estos métodos en tareas muy frecuentes, las constantes consultas e interacciones de lectura/escritura en la caché pueden provocar que la base de datos o la instancia de caché adjunta se mantengan despiertas, bloqueando también su capacidad de escalar a cero*.
