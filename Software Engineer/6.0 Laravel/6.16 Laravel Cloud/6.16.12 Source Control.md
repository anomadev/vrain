### Introduction
- **Mecanismo Central**: Laravel Cloud no almacena código de forma directa: compila y despliega las aplicaciones conectándose nativamente a los proveedores de Git mediante `webhooks` y `APIs` de integración.
- **Proveedores Soportados**: Ofrece compatibilidad de fábrica con `GitHub`, `GitLab` y `Bitbucket`.

### Connecting a source control provider
- **Ruta de Configuración**: Se gestiona globalmente desde el perfil de usuario en `Account settings > Source controls`.
- **Diferencia de Permisos por Proveedor (Análisis Crítico)**: Para `GitHub`, la plataforma exige una doble autorización: el flujo estándar de `OAuth` y la instalación obligatorio de la `Laravel Cloud App` a nivel de cuenta personal u organización de GitHub. Esto actúa como un guardián de seguridad perimetral que decide exactamente a qué repositorios tiene acceso la nube.
	- Para `GitLab` y `Bitbucket`, el acceso se rige de forma simplificada por los permisos del `token` de usuario `OAuth`, omitiendo el paso de instalar una aplicación dedicada.

### Connecting the same Git to multiple Cloud accounts
- **Multi-inquilino/Cuentas**: Un mismo perfil de `Git` puede enlazarse a múltiples usuarios o cuentas distintas de Laravel Cloud de manera simultánea.
- **Aislamiento de Conexión**: Cada enlace es independiente. Si desvinculas `Git` de un usuario de Laravel Cloud, los demás usuarios que comparten ese mismo `Git` no se verán afectados. Para revocar el acceso de `Git` por completo, todos los usuarios de Laravel Cloud que lo usen deben desconectarlo explícitamente.

### Disconnecting a source control provider
- **Ruta**: Disponible en `Account settings > Source control`, haciendo clic en el menú contextual (`...`) junto al proveedor seleccionado y eligiendo `Disconnect`.
- **Impacto Crítico en la Infraestructura**: Desconectar un proveedor es una acción con efecto cascada. Afectará inmediatamente a todas las aplicaciones distribuidas en las diferentes organizaciones vinculadas a ese usuario de Cloud.
	- *Comportamiento de Resiliencia*: Las aplicaciones en producción seguirán corriendo con normalidad, pero los `pipelines` de despliegue quedarán completamente congelados. No se podrán realizar actualizaciones de código o configuraciones hasta que tú u otro usuario con permisos reestablezca el puente con el repositorio.

### Refreshing tokens
Si los `tokens` de acceso expiran, sufren una revocación externa o necesitas forzar una actualización de los alcances (`scopes`), se realiza mediante la opción `Reauthorize` en el menú contextual del proveedor `Git`.

### Troubleshooting
#### I can't find my repository
Si al intentar crear un proyecto la plataforma no localiza el repositorio `Git`, la documentación aísla el problema en dos causad raíz:
1. **`Your GitHub user doesn't have access to the repository`**: Laravel Cloud hereda en tiempo real los esquemas de permisos de `Git`. Si tu perfil autenticado no puede listar el repositorio de forma nativa en la interfaz web de `GitHub/GitLab/Bitbucket`, Laravel Cloud tampoco lo verá. Requiere solicitar acceso directo al propietario del repositorio.
2. **`The Laravel Cloud app doesn't have the permissions to access the repository`**: (Específico de `GitHub`). Ocurre cuando el usuario tiene permisos sobre el repositorio, pero la `Laravel Cloud GitHub App` está restringida para leer únicamente repositorios seleccionados.
	- Solución: Ir a la configuración de la `App` dentro de `GitHub` y cambiar la visibilidad a `"All repositories"` o añadir el nuevo repositorio manualmente a la lista permitida.

#### Missing GitHub installation
- **Causa Raíz**: Este error específico (*`Missing GitHub installation for...`*) se produce cuando ocurre una interrupción en la comunicación inicial: `GitHub` no logra enviar con éxito el `webhook` de confirmación a Laravel Cloud durante el proceso de instalación de la app.
- **Solución de Arquitectura / `Hard Reset`**:
	1. Ir a los ajustes de la app de Laravel Cloud en `GitHub` y, en la sección de peligro (`Danger zone`), hacer clic en `Uninstall`.
	2. Regresar al panel de Laravel Cloud, iniciar la creación de una nueva aplicación (`Overview > + New application`) y completar el flujo de autorización desde cero para re-vincular los `webhooks` correctamente.
