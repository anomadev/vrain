Laravel Private Cloud provee de un entorno completamente aislado operado de extremo a extremo por el equipo de Laravel. Ofrece la seguridad, el control y la postura  de cumplimiento (*compliance*) de una infraestructura empresarial sin requerir que la organización desvíe recursos de ingeniería para gestionar servidores o `Kubernetes`.

### Why Private Cloud 
Está diseñado para empresas con estrictos requisitos de rendimiento, seguridad o regulación bajo los siguientes pilares:
- **Infraestructura Dedicada**: Cómputo, redes y auto-escalado exclusivos. Elimina el problema del "vecino ruidoso" (*`noisy neighbor`*) garantizando un rendimiento predecible.
- **Redes Privadas**: Permite conectarse a servicios de AWS ya existentes de forma interna, reduciendo la latencia y eliminando la exposición a internet pública.
- **`IPs` de Salida Dedicadas**: Direcciones `IP` estáticas exclusivas por cliente para facilitar auditorías, trazabilidad y listas de distribución permitidas (*`allowed lists`*).

### How it works
Tu nube privada se despliega dentro de una cuenta de AWS y una VPC dedicadas, administradas por Laravel. Esto incluye:
- Un clúster de `Kubernetes` privado
- Nodos de cómputo dedicados
- `Gateways NAT` dedicados con `IPs` estáticas de salida
- Conectividad privada (`VPC Peering`, `PrivateLink`, etc.)
- Opcionalmente, bases de datos RDS gestionadas dentro del entorno privado.
- *Flujo de trabajo*: El desarrollador sigue desplegando mediante la interfaz estándar de Laravel Cloud, pero el código ejecuta en fierro 100% aislado y propio

### Dedicated edge network
Reemplaza la tarjeta de red estándar por la de `"Private network"` en el lienzo de infraestructura. Otorga una zona perimetral de `Cloudflare` aislada y añade capacidades avanzadas sobre el tráfico.

### Custom rules
- Permite crear reglas personalizadas basadas en condicionales (ruta, `hostname`, `IP`, país, `User-Agent`, `headers` o `query parameters`) y aplicar acciones inmediatas (`challenge`, `block` o `IP access` para permitir/denegar rangos).
- **Mecanismo**: Las reglas ejecutan en orden secuencial estricto de arriba hacia abajo (la primera que coincide, gana) y se pueden reordenar arrastrándolas en la interfaz.

### Cache rules
- Permite anular el comportamiento de caché por extensión de la red compartida y definir políticas personalizadas por rutas específicas (`per-path control`).
- **Ajustes avanzados**: Control de elegibilidad de caché, `Edge TTL`, `Browser TTL`, directivas `stale-while-revalidate` y manejo de `ETags` fuertes. Ejecutan en su propia sección independiente de las reglas de seguridad.

### Traffic visibility
- Muestra analíticas en tiempo real de las principales direcciones IP y rutas que golpean el perímetro (filtros de 24 horas, 7 días o 30 días).
- **Acción rápida (Análisis Crítico)**: Al hacer clic sobre cualquier fila del reporta de tráfico sospechoso, la interfaz precarga automáticamente una nueva `Custom Rule` con esa IP o ruta para mitigar ataques de forma inmediata.

### Rule event counts
Muestra un contador directo de eventos en las últimas 24 horas al lado de cada regla personalizada y de límite de tasa para verificar cuántas veces ha hecho `"match"`. *Las reglas de caché no muestran este conteo individual; requieren ver las tarjetas métricas generales.*

### Private connections
Permite que la infraestructura aislada se comunique con recursos de AWS del cliente de forma interna.

### Connection types
1. **`VPC Peering`**: Enlace directo entre dos `VPCs` usando `IPs` privadas. Es autoservicio y la opción ideal para redes simples.
2. **`VPC Lattice`**: Malla de servicios gestionada (*service mesh*) para conectar microservicios entre `VPCs` con alta seguridad y observabilidad.
3. **`PrivateLink`**: Expone un servicio o `API` individual mediante un `endpoint` privado (ideal para conectar BD gestionadas por terceros sin abrir la red entera).
4. **`Transit Gateway`**: Hub regional para conectar múltiples `VPCs` o redes complejas en estrella.

### Setting up a VPC peering connection
Describe el flujo de aprovisionamiento en 6 pasos desde el `dashboard`:
1. Ir a la tarjeta `Private Network` y seleccionar `Request private connection`
2. Elegir la `Private Cloud` de destino
3. Seleccionar `VPC Peering`
4. Proveer los datos de tu infraestructura: `AWS Account ID (12 dígitos)`, `VPC ID` y el rango `CIDR` de tu VPC (el cual no debe solaparse con el rango de tu `Private Cloud`)
5. Enviar solicitud
6. Acción en AWS: El administrador debe ingresar a su consola de AWS, aceptar la solicitud de `Peering` pendiente y actualizar las tablas de ruteo (`Route Tables`) de su VPC para dirigir el tráfico de las `IPs` hacia el nuevo enlace.
	- *Nota: El panel mostrará la sección `Ours` (`IPs` de Laravel Cloud) y `Target` (`IPs` de tu infraestructura) para mapear correctamente los `Firewalls/Security Groups`*.

### PrivateLink, VPC Lattice, and Transit Gateway
**Restricción operativa**: Estos tres esquemas avanzados de red no son autoservicio y requieren contactar al equipo de soporte de Laravel Cloud proporcionando los identificadores de recurso, nombres de servicio o `IDs` de `Transit Gateway` para su aprovisionamiento del lado del `staff`.

### Common uses cases
- Sectores altamente regulador (`Fintech`, `Healthcare`)
- Acceso directo a sistemas `legacy` corporativos o bases de datos internas privadas
- Cargas de trabajo de alta demanda con requerimiento de rendimiento predecible
- Adopción de Laravel Cloud dentro de un ecosistema AWS empresarial ya estructurado

### Requirements
- Se requiere entregas los `IDs` de cuentas de AWS, rangos `CIDR` planificados para evitar colisiones y detalles de conectividad.
- Tiempo de entrega: El aprovisionamiento e implementación de una `Private Cloud` toma típicamente de 4 a 6 semanas.

### Pricing
- Basado en cotizaciones personalizadas según el tamaño
- Análisis de Viabilidad Económica: La documentación estipula explícitamente que el esquema de Private Cloud se vuelve rentable y costo-eficiente para organizaciones con un gasto de infraestructura igual o superior a $2000 USD mensuales.
