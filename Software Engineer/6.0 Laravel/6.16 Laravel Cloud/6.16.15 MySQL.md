### Introducción
- Laravel MySQL provee una base de datos MySQL completamente gestionada para tus aplicaciones.
- Cada instancia tiene asignada una capacidad de cómputo y un tamaño de almacenamiento estáticos, con la posibilidad de reescalar en cualquier momento.
- Actualmente, las instancia de Laravel MySQL operan bajo un modelo *`always-on`* (siempre encendidas), pero la documentación oficial estipula que el soporte para *`Scale to Zero`* en la clase de cómputo `Flex` está catalogado como próximo a lanzarse.

### Creating MySQL databases
El aprovisionamiento de bases de datos se realiza de manera simplificada:
- **Rutas de Creación**: Se puede aprovisionar y enlazar directamente en un entorno desde el lienzo de infraestructura (`Add Database`) o centralizadamente en `Org > Resources > Databases`.
- **Reglas de Vinculación**: Las bases de datos están limitadas geográficamente; solo pueden adjuntarse a clústeres de cómputo que residan en su misma región de AWS. Tras conectar una base de datos en un entorno, es mandatorio realizar un nuevo despliegue (*`re-deploy`*) para que los cambios surtan efecto en el software.
- **Configuración del Clúster**: Al crear un nuevo clúster, se parametriza:
	- *`Cluster name`*: Identificador único
	- *`Instance size`*: Opciones de cómputo divididas en arquitecturas `Flex` o `Pro`.
	- *`Storage`*: Almacenamiento asignable desde un mínimo de 5GB hasta un máximo de 1000 GB.

### Database clusters vs databases
Un `Database Cluster` representa la instancia o el servidor físico aprovisionado (el `hardware` y su almacenamiento). Dentro de un mismo clúster, puedes crear e instanciar tantas bases de datos lógicas (a menudo llamadas esquemas) como el negocio requiera. Al asociar el recurso al entorno de Laravel, se elige la DB lógica específica que se inyectará en la aplicación.

### Editing and resizing database clusters
- **Flexibilidad Operativa**: El tamaño de cómputo y de almacenamiento se puede modificar desde el panel de recursos seleccionado `Edit settings`.
- **Restricción de Infraestructura (Análisis Crítico)**: El almacenamiento asignado a Laravel MySQL solo se puede incrementar y la plataforma prohíbe disminuir el tamaño del disco por limitaciones físicas de volumen de bloques. Adicionalmente, impone una ventana de bloqueo temporal: solo se puede aumentar el almacenamiento una vez cada 6 horas.

### Monitoring database cluster metrics
- **Observabilidad**: Permite analizar el rendimiento histórico a través del panel métrico del recurso.
- **Métricas Principales**: Monitorea el uso de CPU, el rendimiento de escritura (*`write throughput`*) y el consumo de almacenamiento en disco.

### Deleting databases clusters
*Permite la remoción total del hardware y los datos desde el panel de recursos lógicos de la organización.*

### Connecting to databases clusters
#### From your application
**Inyección de Dependencias Automatizada**: Al vincular la base de datos a un entorno, Laravel Cloud inyecta de forma nativa y transparente las variables de entorno estándar del `framework` (`DB_HOST`, `DB_USERNAME`, `DB_PASSWORD`, `DB_DATABASE`), eliminando la necesidad de quemar credenciales en el código fuente.

#### From your local machine
**Acceso Externo**: Para conectar herramientas de administración locales (como TablePlus, DBeaver, etc.), se debe ingresar al clúster en la sección de recursos y activar explícitamente el interruptor `"Enable public endpoint"`. El panel proporciona las credenciales y un enlace de apertura directa (*`Deeplink`*).

#### MySQL public endpoints & SSL connections
*Usa conexiones cifradas seguras a través de estos `endpoints` públicos validados para la administración remota de datos*.

#### Importing data
Optimización de Memoria (Criterio Técnico): Al migrar o importar bases de datos SQL grandes (especialmente hacia instancias pequeñas de tipo `Flex`), la documentación advierte que se deben generar `dumps` de `SQL` utilizando la opción `--skip-extended-insert`. Esto evita que MySQL procese inserciones masivas en una sola línea de comando, reduciendo drásticamente los picos de consumo de memoria RAM del clúster que podrían colapsar la base de datos durante la importación.

#### Connections and proxy layer
- **Capa Proxy Multiplexada**: Todas las bases de datos en Laravel Cloud están protegidas y gestionadas por una capa `proxy` inteligente que realiza multiplexación de conexiones.
- **Ventaja**: Permite escalar y tolerar hasta 10,000 conexiones simultaneas bajo escenarios específicos de consultas de lectura (`read-only queries`).
- **Advertencia Crítica de Arquitectura**: El `proxy` no puede multiplexar operaciones de escritura ni transacciones. Cada consulta de inserción, actualización o bloque transaccional (`BEGIN`/`COMMIT`) consume directamente una conexión física real del límite nativo de la instancia de MySQL, por lo que un software con alta transaccionalidad de escritura requerirá clústeres de cómputo `Pro` para mitigar el agotamiento de conexiones.

### Database backups
- **Esquema de Respaldos**: Soporta respaldos tanto automatizados diarios como `snapshots` manuales.
- **Retención**: Configurable desde el panel. Para entornos de desarrollo o desechables, se puede fijar en 0 días. Para entorno productivos, la plataforma recomienda un mínimo de 2 días de retención como política básica de recuperación de desastres.
- **Mecanismo de Restauración**: Al restaurar un respaldo, Laravel Cloud no sobrescribe el clúster actual. En su lugar, aprovisiona un clúster de base de datos completamente nuevo a partir del `snapshot`, garantizando la integridad de los datos actuales en caso de errores en el proceso de recuperación.
- **Facturación Desacoplada**: El almacenamiento de los respaldos se factura de forma independiente al almacenamiento en disco del clúster operativo (a una tasa aproximada de $0.10/GB-mes en regiones de EE.UU y $0.12/GB-mes en el resto de regiones).

### Troubleshooting & Performance monitoring
Al monitorear el rendimiento a través de los gráficos de analíticas, se debe prestar especial atención a tres indicadores de riesgo:
- **`High CPU usage`**: Típicamente delata la existencia de consultas ineficientes que carecen de índices apropiados o un clúster de cómputo subdimensión para la carga.
- **`High memory usage`**: Puede desencadenar fallos de ejecución en consultas complejas o cierres inesperados de conexiones por falta de recursos.
- **`Low disk space`**: Alerta crítica de infraestructura. Quedarse sin espacio en disco provocará de inmediato fallos catastróficos de escritura, tiempos de espera excedidos (`connections timeouts`) y la corrupción potencial de tablas operativas.
