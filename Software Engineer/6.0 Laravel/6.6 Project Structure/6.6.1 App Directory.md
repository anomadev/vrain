## Definición
El directorio `/app` es el corazón de tu aplicación. Contiene casi todo el código fuente personalizado de tu proyecto. Es donde reside la `Business Logic`, los modelos de datos y los controladores. Laravel utiliza es estándar `PSR-4` para cargar automáticamente las clases dentro de este directorio bajo el `namespace` `App`.

## Conceptos Clave
- **`Models`**: Ubicados en `app/Models`. Representan tus tablas de la base de datos y contiene la lógica de relaciones de `Eloquent`.
- **`Http`**: Contiene los `Controllers` (lógica de rutas), `Middleware` (filtros de peticiones) y `Requests` (validación de datos).
- **`Providers`**: Los "ensambladores" de la aplicación. Aquí se registran servicios en el `Service Container`.
- **`Broadcasting / Events / Listeners`**: (Opcionales) Se crean automáticamente cuando ejecutas comando `make` para manejar lógica asíncrona o eventos de sistema.
- **`Service / Actions`**: (`Custom`) Carpetas que nosotros, como arquitectos, solemos añadir para extraer lógica de los controladores y cumplir con `SOLID`.

## Ejemplos Prácticos
##### Estructura típica de una clase en `app/`
```php
// app/Models/Project.php
namespace App\Models;

use Illuminate\Database\Eloquent\Model;

class Project extends Model
{
	// Business logic and Relationships live here
}
```
##### Inyección de dependencias en un `Controller`
```php
// app/Http/Controllers/ProjectController.php
namespace App\Http\Controllers;

use App\Models\Project;
use Illuminate\Http\Request;

class ProjectController extends Controller
{
	public function show(Project $project)
	{
		return view('projects.show', compact('project'));
	}
}
```

## Analogía
Imagina que tu aplicación es un **Cuerpo Humano**. El directorio `app/` es el **Sistema Nervioso Central y los Órganos**.
- Los `Models` son los huesos y órganos (la estructura y los datos)
- Los `Controllers` son los reflejos (reciben un estímulo/petición y deciden qué mover).
- Los `Providers` son el sistema circulatorio que lleva la energía (servicios) a donde se necesitan antes de empezar a moverse.

## Buenas Prácticas
- **`Fat Models, Skinny Controllers? No`**: En `Clean Architecture`, preferimos `Skinny Controllers` y `Service Classes`. No satures tus modelos con lógica de negocio compleja; usa los modelos solo para datos y relaciones.
- **`Custom Namespaces`**: Si tu proyecto crece, no tengas miedo de crear carpetas como `app/Services`, `app/Actions` o `app/Repositories`. Laravel las reconocerá automáticamente gracias a `PSR-4`.
- **`Lean Providers`**: Mantén los `ServiceProviders` limpios. Si tienes mucha lógica de registro, crea un nuevo `Provider` especializado (ej. `app/Providers/ViewServiceProvider.php`).

## Escenarios y Aplicaciones
- **Refactorización**: Si ves que un controlador en `app/Http/Controllers` tiene más de 50 líneas, es el escenario perfecto para mover esa lógica a una nueva carpeta `app/Actions`.
- **`Domain Driven Design (DDD)`**: En aplicaciones masivas, a veces reestructuramos `app/` para que esté organizado por "Dominios" (ej. `app/Billing`, `app/Inventory`) en lugar de por tipos de archivos.

## Notas Personales
- En versiones anteriores existía `app/Http/Kernel.php`. Si buscas dónde registrar un `Middleware` global en Laravel moderno, ahora se hace en `bootstrap/app.php`.
- Para el `Senior`, el directorio `app/` debe estar impecable. El uso de `Actions` (clases con un solo método `handle()` o `execute()`) es la mejor forma de mantener este directorio escalable.
