## Definición
El directorio `tests/` contiene la suite de prueba automatizadas de tu aplicación. Su propósito es verificar que el código se comporte exactamente como se espera. Se organiza en dos grandes categorías: **`Unit Tests`** (pruebas aisladas de lógica pura) y **`Feature Tests`** (pruebas de integración que involucran la base de datos, rutas y respuestas `HTTP`).

## Conceptos Clave
- **`Feature Tests`**: Pruebas que verifican una "característica" completa. Simulan peticiones `HTTP`, revisan respuestas, verifican cambios en la base de datos y aseguran que los componentes (como `Livewire` o `Inertia`) funcionen juntos.
- **`Unit Tests`**: Pruebas de "unidad" que se enfocan en una sola clase o método en total aislamiento. No tocan la base de datos ni cargan el framework de Laravel (son extremadamente rápidas).
- **`Pest / PHPUnit`**: Los motores de ejecución. `Pest` es el estándar moderno por su sintaxis expresiva y minimalista.
- **`Assertions`**: Los métodos que usamos para validar resultados (ej. `assertStatus`, `assertSee`, `espect()`).
- **`TestCase.php`**: La clase base de la que heredan todos tus `tests`; aquí se configura el entorno de pruebas de pruebas antes de cada ejecución.

## Ejemplos Prácticos
##### `Feature Test (Testing a Route)`
```php
// tests/Feature/UserRegistrationTest.php

it('allows a user to register', function () {
    $response = $this->post('/register', [
        'name' => 'G Developer',
        'email' => 'g@example.com',
        'password' => 'password123',
        'password_confirmation' => 'password123',
    ]);

    $response->assertRedirect('/dashboard');
    $this->assertDatabaseHas('users', [
        'email' => 'g@example.com',
    ]);
});
```
##### `Unit Test (Pure Logic)`
```php
// tests/Unit/SalaryCalculatorTest.php

test('it calculates the correct tax amount', function () {
    $calculator = new SalaryCalculator();
    $tax = $calculator->calculateTax(1000, 0.15); // 15%

    expect($tax)->toBe(150.0);
});
```

## Analogía
Imagina que estás construyendo un Puente.
- Los **`Unit Tests`** son las pruebas de laboratorio que le haces a un solo tornillo o a una viga de acero para ver si aguanta la presión antes de ponerla.
- Los **`Feature Tests`** son las pruebas de carga donde pasas camiones reales por el puente terminado para ver si la estructura completa se mantiene firma y los semáforos funcionan.
- El directorio **`tests/`** es tu Departamento de Control de Calidad que asegura que el puente no se caiga cuando decidas cambiar una viga el próximo año.

## Buenas Prácticas
- **`TDD (Test Driven Development)`**: Intenta escribir el test antes del código. Te obliga a pensar en la arquitectura y en los requisitos antes de "picar" código.
- **`Don't over-unit test`**: En Laravel, las `Features Tests` suelen dar más valor por tu tiempo (Pareto), ya que cubren más terreno y aseguran que la integración de componentes sea correcta.
- **`RefreshDatabase`**: Usa este `Trait` en tus `Features Tests` para que la base de datos se limpie automáticamente despues de cada `tests`, garantizando un entorno puro.
- **`Keep Unit Tests Pure`**: Si tu `test` en la carpeta `Unit` necesita tocar la base de datos, entonces no es un `Unit Test`; muévelo a `Feature`.

## Escenarios y Aplicaciones
- **`Continuous Integration (CI)`**: Configurar `GitHub Actions` para que ejecute tus `tests` cada vez que hagas un `push`. Si un `test` falla, el código no se despliega.
- **`Documentation`**: Los `tests` bien escritos sirven como documentación viva de cómo se supone que debe funcionar cada parte del sistema.

## Notas Personales
- Un error común es olvidar que los `tests` usan un archivo `.env.testing`. Si tus `tests` fallan por conexión a DB, revisa ese archivo o tu `phpunit.xml`.
