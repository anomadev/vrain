## Definición
Los `Route Parameters` son segmentos variables dentro de una `URI` que permiten capturar valores dinámicos para ser procesados por la aplicación. Son esenciales para identificar recursos específicos (como un usuario o un post) sin necesidad de crear una ruta única para cada registro en la base de datos.

## Conceptos Clave
- **`Required Parameters`**: Segmentos obligatorios definidos entre llaves `{id}`. Si el parámetro falta en la `URL`, Laravel devolverá un error `404`.
- **`Optional Parameters`**: Segmentos que pueden o no estar presentes, indicados con un signo de interrogación `{name?}`. Requieren un valor por defecto en la firma de la función.
- **`Regular Expression Constraints`**: Filtros aplicados con el método `->where()` que aseguran que el parámetro cumpla con un formato específico (ej. solo números o solo letras).
- **`Global Constraints`**: Definición de patrones de validación en el `AppServiceProvider` que se aplican a todas las rutas que usen un nombre de parámetro específico.
- **`Dependency Injection Priority`**: Laravel siempre inyecta primero las dependencias de clase (como `Request`) y luego los parámetros de la ruta en el orden en que aparecen en la `URI`.

## Ejemplos Prácticos
```php
// routes/web.php
use Illuminate\Support\Facades\Route;
use Illuminate\Http\Request;

// 1. Required and Multiple Parameters
Route::get('/posts/{post}/comments/{comment}', function (string $postId, string $commentId) {
    return "Post: $postId, Comment: $commentId";
});

// 2. Optional Parameter with Default Value
Route::get('/user/{name?}', function (?string $name = 'Guest') {
    return "Hello, $name";
});

// 3. Regular Expression Constraints (Local)
Route::get('/user/{id}', function (string $id) {
    return "User ID: $id";
})->where('id', '[0-9]+');

Route::get('/category/{slug}', function (string $slug) {
    return "Category: $slug";
})->where('slug', '[A-Za-z0-9-]+');

// 4. Global Constraints (Inside AppServiceProvider::boot)
// public function boot(): void {
//     Route::pattern('id', '[0-9]+');
// }
```

## Analogía
Imagina que las rutas son buzones de correo. Una ruta estática es un buzón con un nombre grabado en piedra. Un `Route Parameter` es como un buzón con una ranura ajustable donde puedes realizar cualquier etiqueta de nombre. El cartero (Laravel) lee lo que dice la etiqueta y te entrega el contenido personalmente para que tú, el destinatario (`Controller`), decidas qué hacer con ese paquete específico.

## Buenas Prácticas
- `Type Hinting`: Usa siempre tipos de datos en los argumentos de tus funciones (`string $id`, `int $count`) para aprovechar el chequeo de tipos de PHP.
- `Semantic Naming`: Dale nombres descriptivos a los parámetros. Evita `{data}` o `{val}`; prefiere `{user_slug}` o `{order_number}`.
- `Regex Validation`: Si un parámetro debe ser un número, usa siempre `->where('id', '[0-9]+')`. Esto evita que peticiones mal formadas lleguen a tu lógica de base de datos, mejorando la seguridad y el rendimiento.

## Escenarios y Aplicaciones
- `SEO Friendly URLs`: en lugar de `/observation/123`, usa `/observation/{slug}` para que las `URLs` sean legibles por humanos y buscadores.
- `Versioning`: En `APIs`, puedes capturar la versión: `/api/{version}/users`.
- `Multi-step Forms`: Pasar un identificador temporal entre diferentes pasos de una configuración.

## Notas Personales
- El orden de las rutas es crítico. Si defines `/user/{id}` antes de `/user/settings`, Laravel capturará "`settings`" como si fuera un `ID`. Pon siempre las rutas fijas antes que las dinámicas.
- En `Laravel 11/12`, el uso de `Global Constraints` en el `AppServiceProvider` es la forma más limpia de manejar `IDs` consistentes en toda la arquitectura de tu aplicación.
