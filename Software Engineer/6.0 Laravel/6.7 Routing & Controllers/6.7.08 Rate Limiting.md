## Definición
Es el proceso de controla la cantidad de peticiones que un usuario o una dirección `IP` puede realizar a una ruta o `API` en un periodo de tiempo determinado. Su objetivo principal es prevenir el abuso del sistema, mitigar ataques `DoS (Denial of Service)` y gestionar el consumo de recursos en aplicaciones escalables.

## Conceptos Clave
- **`RateLimiter Facade`**: El punto de entrada para definir configuraciones de limitación personalizadas.
- **`Throttle Middleware`**: El encargado de interceptar las peticiones y aplicar las reglas de límite antes de que lleguen al controlador.
- **`HTTP 429 (Too Many Requests)`**: El código de estado estándar que Laravel devuelve automáticamente cuando un usuario excede su límite.
- **`Decay Minutes`**: El tiempo que debe transcurrir para que el contador de peticiones se reinicie.
- **`Dynamic Limiting`**: Capacidad de ajustar los límites basados en atributos del usuario (ej. usuarios autenticados vs invitados o planes "Premium").

## Ejemplos Prácticos
##### `Defining Limiters (AppServiceProvider)`
Laravel 11+ centraliza estas definiciones en el `boot()` del `AppServiceProvider`.
```php
// app/Providers/AppServiceProvider.php
use Illuminate\Cache\RateLimiting\Limit;
use Illuminate\Http\Request;
use Illuminate\Support\Facades\RateLimiter;

public function boot(): void
{
    RateLimiter::for('api', function (Request $request) {
        return $request->user()?->isPremium()
            ? Limit::perMinute(1000)->by($request->user()->id)
            : Limit::perMinute(60)->by($request->ip());
    });
}
```
##### `Applying to Routes`
```php
// routes/api.php

// Applying the named limiter 'api' defined above
Route::middleware('throttle:api')->group(function () {
    Route::get('/observations', [ObservationController::class, 'index']);
});

// Inline limiting (Max 10 requests per minute)
Route::middleware('throttle:10,1')->post('/login', [AuthController::class, 'login']);
```

## Analogía
Imagina el `Rate Limiting` como el Cadenero de un Club Nocturno.
El club tiene una capacidad limitada. El cadenero (`Middleware`) lleva un contador para cada persona en la fila. Si intentas entrar 10 veces en un minuto, el cadenero te detiene y te dice: "Espera un momento, estás viniendo demasiado seguido; vuelve en una hora". No importa si eres un cliente VIP o un desconocido, el contador se aplica para asegurar que el club no colapse y todos adentro tengan una buena experiencia.

## Buenas Prácticas
- `Segment by IP/User`: Siempre usa `by($request->ip())` para invitados y `by($request->user()->id)` para usuarios `logueados` para evitar que un usuario afecte a toda una red (como una oficina).
- `Separate Sensitive Routes`: Las rutas de `login`, `register` y recuperación de contraseña deben tener límites mucho más estrictos (ej. 5 intentos por minuto) que las rutas de lectura de datos.
- `Inform the User`: Laravel envía automáticamente cabeceras como `X-RateLimit-Remainig`. Asegúrate de que tu `frontend` (`Vue`/`Livewire`) maneje el error `429` para mostrar un mensaje amigable al usuario.

## Escenarios y Aplicaciones
- `Cost Control`: Si tu aplicación consume `APIs` de terceros, aplica `Rate Limit` estricto para evitar que un usuario malintencionado agote tu presupuesto.
- `Data Integrity`: Limitar las peticiones de sincronización masiva para evitar picos de carga innecesarios en el servidor durante las horas pico.
- `Public APIs`: Si expones datos para terceros, usa `Rate Limiting` para monetizar o simplemente para asegurar el `Uptime` del servicio.

## Notas Personales
- El `Rate Limiting` depende del `Cache Driver`. Si estás usando el driver `array`, los límites se reiniciarán en cada petición o reinicio del servidor. Usa `redis` o `database` en producción.
