## Definición
Un `Single-action Controller` es un controlador que tiene una única responsabilidad y expone un solo método público: `__invoke`. En lugar de agrupar múltiples métodos `CRUD` (`index`, `store`, `show`, etc.) en una sola clase, se crea una clase dedicada para una acción específica de la aplicación.

## Conceptos Clave
- El método `__invoke`: Es un método mágico de PHP que permite que una instancia de clase sea llamada como si fuera una función. Laravel lo utiliza para ejecutar la lógica del controlador automáticamente.
- Desacoplamiento: Permite extraer lógica compleja que no encaja naturalmente en un recurso `CRUD` (ej. `ProcessPayment`, `GenerateReport`, `PublishPost`).
- Inyección de Dependencias: Al igual que los controladores estándar, el método `__invoke` admite la inyección automática de dependencias (como  `Request` o servicios personalizados).
- Organización por Dominios: Facilita implementación de `Domain Driven Design (DDD)` al nombrar los controladores según la acción de negocio que realizan.

## Ejemplos Prácticos
##### Creación del Controlador
Se utiliza el comando `Artisan` con el `flag --invokable`:
```bash
php artisan make:controller ProvisionServerController --invokable
```
##### Estructura del Controlador
```php
namespace App\Http\Controllers;

use App\Services\ServerService;
use Illuminate\Http\Request;

class ProvisionServerController extends Controller
{
    /**
     * Maneja la acción de provisionar un servidor.
     */
    public function __invoke(Request $request, ServerService $service)
    {
        // Lógica específica de la acción
        $service->provision($request->input('server_id'));

        return redirect()->back()->with('status', 'Server provisioning started!');
    }
}
```
##### Registro en el archivo de rutas
Al ser invocable, no es necesario especificar el nombre del método en el `array` de la ruta; Laravel sane que debe buscar `__invoke`:
```php
use App\Http\Controllers\ProvisionServerController;
use Illuminate\Support\Facades\Route;

Route::post('/servers/provision', ProvisionServerController::class);
```

## Analogía
Imagina una Navaja Suiza frente a una Llave de Torsión Especializada.
Un `Resource Controller` es la navaja suiza: tiene muchas herramientas (métodos) en un solo mango. Es útil para tareas generales, pero a veces las herramientas se estorban entre sí. Un `Single-action Controller` es la herramienta especializada: Solo hace una cosa, pero la hace perfectamente y es mucho más fácil de encontrar y utilizar en situaciones críticas sin riesgo de usar la herramienta equivocada.
### Cuando usar `Single-action` vs. `Resource Controllers`
| Características | `Resource Controller` | `Single-action Controller`
| :--- | :--- | :--- |
| Uso Ideal | Operaciones `CRUD` estándar | Acciones complejas o procesos de negocio únicos |
| Organización | Agrupa lógica por entidad de datos | Agrupa lógica por acción/proceso |
| Mantenimiento | Puede volverse un `God Object` si es muy grande | Muy fácil de testear y refactorizar de forma aislada |
| Legibilidad | Estándar y predecible | Narrativo (el nombre de la case dice qué hace) |

## Buenas Prácticas
- `Naming is Key`: Usa verbos para nombrar estas clases (ej. `UpdateUserPassword`, `SyncExternalData`). El nombre debe describir la acción.
- `Move to Subdirectories`: Si usas muchos controladores de este tipo, organízalos en carpetas por dominio o recurso dentro de `app/Http/Controllers/`.
- `Extract to Actions`: Si la lógica dentro del `__invoke` supera las 20-30 líneas, considera mover la lógica pesada a una clase de `Action` o un `Service`, dejando el controlador solo para el manejo del `Request`/`Response`.

## Notas
El uso de controladores invocables es una señal de madurez en el desarrollo con Laravel. Evita que los controladores se conviertan en archivos masivos y facilita enormemente las **Pruebas Unitarias**, ya que solo hay un cambio de ejecución que verificar por clase.
