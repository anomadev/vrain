## Definición
El `middleware` es una capa que se sitúa entre la petición `HTTP` entrante y el destino final (normalmente un controlador). Actúa como un mecanismo de filtrado que puede inspeccionar, modificar o incluso rechazar la petición antes de que llegue a la lógica de tu aplicación, así como manipular la respuesta antes de enviarla de vuelta al cliente.

## Conceptos Clave
- **`The handle method`**: El corazón de cualquier `middleware`. Recibe el `$request` y un cierre `$next`. Si el `middleware` permite que la petición continúe, llama a `$next($request)`.
- **`Before vs After Middleware`**: Un `middleware` puede ejecutar lógica antes de que la petición sea procesada (ej. verificar un `token`) o después de que el controlador genere una respuesta (ej. añadir cabeceras de seguridad).
- **`Global Middleware`**: Se ejecuta en cada una de las peticiones `HTTP` que recibe la aplicación.
- **`Route Middleware`**: Se asigna a rutas específicas o grupos de rutas.
- **`Middlware Groups`**: Permiten agrupar varios `middlewares` bajo una sola etiqueta (como `web` o `api`).
- **`Terminable Middleware`**: Un tipo especial de `middleware` que ejecuta lógica después de que la respuesta `HTTP` haya sido enviada al navegador (útil para logs pesados).

## Ejemplos Prácticos
##### Creación de un `Middleware` personalizado
```bash
php artisan make:middleware EnsureUserIsExpert
```
##### Estructura de un `Middleware` (Antes)
```php
// app/Http/Middleware/EnsureUserIsExpert.php
public function handle(Request $request, Closure $next): Response
{
    if (! $request->user()?->is_expert) {
        return redirect('home')->with('error', 'Solo para expertos.');
    }

    return $next($request);
}
```
##### Configuración en Laravel 11/12 (`bootstrap/app.php`)
Ya no usamos `Kernel.php`. Ahora registramos y configuramos todo en `app.php`
```php
// bootstrap/app.php
return Application::configure(basePath: dirname(__DIR__))
    ->withMiddleware(function (Middleware $middleware) {
        // Añadir middleware global
        $middleware->append(GlobalLoggingMiddleware::class);

        // Alias para usar en rutas
        $middleware->alias([
            'expert' => \App\Http\Middleware\EnsureUserIsExpert::class,
        ]);
    })->create();
```
##### Aplicación en Rutas
```php
Route::get('/advanced-settings', function () {
    // Lógica
})->middleware('expert');
```

## Analogía
Imagina que tu aplicación es un **Club Exclusivo**.
- El `Request` es una persona intentando entrar.
- Los `Middlewares` son los distintos puntos de control:
	1. Seguridad (`Auth`): Tienes tu identificación?
	2. Código de Vestimenta (Validación): Vienes vestido adecuadamente para este club?
	3. Detector de Metales (`Sanitize`): Traes algo peligroso en tus bolsillos?
	Si pasas todos los puntos, llegas a la Mesa VIP (`Controller`). Si fallas en cualquiera, te escoltan a la salida antes de que siquiera veas la pista de baile.

## Buenas Prácticas
- **`Single Responsability`**: Cada `middleware` debe hacer una sola cosa. No mezcles lógica de `logs` con lógica de permisos.
- **`Avoid Heavy Logic`**: El `middleware` se ejecuta en el camino crítico de la petición. Evita consultas pesadas a la base de datos si es posible; usa el caché.
- **`Leverage Built-in Middleware`**: Laravel ya incluye `middlewares` potentes para `CSRF`, autenticación y `rate limiting`. No los reinventes.
- **`Use parameters`**: Puedes pasar datos extra a tus a tus `middlewares` (ej. `middleware('role:admin')`) para hacerlos reutilizables.

## Escenarios y Aplicaciones
- **`Localization`**: Detectar el idioma del navegador del usuario y configurar el `locale` de la app.
- **`Maintenance Mode`**: Bloquear el acceso a ciertas rutas mientras realizas el `refactor` de tu SaaS.
- **`API Logging`**: Guardar un registro de todas las peticiones entrantes para auditoría de seguridad.

## Notas Personales
- Si olvidas retornar `$next($request)` en tu `middleware`, la petición se detendrá y el usuario recibirá una pantalla en blanco. Es el error más común al empezar.
- El orden de los `middleware` en el `pipeline` importa. Por ejemplos, el `middleware` de sesión debe ejecutarse antes que cualquier `middleware` que intente acceder al usuario autenticado.
