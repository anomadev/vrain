### `Migrations`: El Control de Versiones de la Base de Datos
Las `Migrations` son archivos PHP que definen la estructura de las tablas de la base de datos. Permiten que el equipo de desarrollo comparta y modifique el esquema de forma sincronizada y reversible.
##### Conceptos Clave
- **`Scheme Builder`**: La interfaz de Laravel para crear y modificar tablas independientemente del motor de base de datos (`MySQL`, `PostgreSQL`, etc.).
- **Métodos `up` y `down`**: 
	- **`up`**: Define los cambios que se aplicarán (crear tablas, añadir columnas)
	- **`down`**: Define cómo revertir esos cambios (borrar tablas, quitar columnas)
- **`Migration Table`**: Laravel mantiene una tabla llamada `migrations` en la DB para saber qué archivos ya han sido ejecutados.
##### Ejemplo de Migración
```php
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Support\Facades\Schema;

return new class extends Migration {
    public function up(): void {
        Schema::create('posts', function (Blueprint $table) {
            $table->id();
            $table->string('title');
            $table->text('content');
            $table->foreignId('user_id')->constrained()->onDelete('cascade');
            $table->timestamps(); // Crea created_at y updated_at
        });
    }

    public function down(): void {
        Schema::dropIfExists('posts');
    }
};
```
##### Comandos de `Artisan`
| Comando                             | Propósito                                                        |
| :---------------------------------- | ---------------------------------------------------------------- |
| `make:migration create_posts_table` | Crea un nuevo archivo de migración                               |
| `migrate`                           | Ejecuta todas las migraciones pendientes                         |
| `migrate:rollback`                  | Revierte la última operación de migración                        |
| `migrate:fresh`                     | Borra todas las tablas y ejecuta todas la migraciones desde cero |
### `Seeders`: Poblando de Datos
Los `Seeders` son clase diseñadas para insertar datos de prueba o datos iniciales (como roles, países o configuraciones) en la base de datos de forma automática.
##### Relación con las `Factories`
Mientas que el `Seeder` es la "acción" de insertar, las `Factories` son las "plantillas" que generan datos falsos realistas (usando la librería `Faker`). Un `Seeder` profesional suele llamar a una `Factory` en lugar de definir los datos manualmente.
##### Ejemplos de `Seeder`
```php
// database/seeders/UserSeeder.php
public function run(): void {
    // Crear 10 usuarios con datos aleatorios usando una Factory
    \App\Models\User::factory(10)->create();

    // Crear un usuario específico para pruebas
    \App\Models\User::factory()->create([
        'name' => 'Admin User',
        'email' => 'admin@example.com',
    ]);
}
```

## Analogía
Imagina que estás construyendo una **Casa**.
- Las `Migrations` son los Planos Azules. Definen dónde vas la paredes, cuántas habitaciones hay y dónde están las tuberías. Si decides añadir un baño, actualizas el plano para que todos los constructores sepan qué hacer.
- Los `Seeders` son los Muebles y Habitantes. Una vez que la casa está construida, el `Seeder` "muda" los muebles a la sala y trae a la familia para que la casa sea funcional y puedas probar si la cocina realmente funciona.

## Buenas Prácticas
1. **Nunca edita una migración ejecutada**: Si necesitas cambiar una tabla en producción, crea una nueva migración que altere la tabla (`table` en lugar de `create`). Editar archivos antiguos romperá la sincronización de tus compañeros.
2. **Uso de `migrate:fresh --seed`**: Durante el desarrollo, este comando es tu mejor amigo para limpiar la base de datos y tener un entorno fresco con datos de prueba listos para usar.
3. **Atomicidad**: Cada migración debería realizar una sola tarea lógica (ej. crear una tabla específica o añadir una columna de configuración).
4. **No uses `Seeders` para Lógica de Negocio**: Los `seeders` son para datos de "arranque" o de "prueba". La lógica para crear datos reales durante el uso de la app debe vivir en Controladores o Servicios.
