## Definición
Es el proceso de indicarle a `Vue` dónde encontrar la definición de un componente para que pueda ser renderizado dentro de un `template`. Determina el `scope` (alcance) de visibilidad del componente y tiene un impacto directo en la mantenibilidad del código y el rendimiento final del `bundle` (`Tree-shaking`).

## Conceptos Clave
- **`Global Registration`**: Componentes registrados en la instancia de la aplicación (`app.component()`). Son accesibles en cualquier parte sin necesidad de importarlos.
- **`Local Registration`**: Componentes importados explícitamente dentro de otro componente. Solo son visibles para quien los importa.
- **`PascalCase`**: Convención recomendada para el nombre de los componentes tanto en el registro como en los `templates`: (ej. `<MyComponent />`).
- **`Tree-shaking`**: Proceso por el cual el `build tool` (`Vite`) elimina el código que no se utiliza. El registro global impide que esto ocurra de forma eficiente.

## Ejemplos Prácticos
##### Global Registration (Limitado a casos base)
```typescript
// main.ts
import { createApp } from 'vue';
import App from './App.vue';
import BaseIcon from './components/BaseIcon.vue';

const app = createApp(App);

// Registered globally
app.component('BaseIcon', BaseIcon);
app.mount('#app');
```
##### Local Registration (Stardard for Senior Devs)
```typescript
<script setup lang="ts">
// Explicit import makes dependencies clear
import UserProfile from './UserProfile.vue';
import UserSettings from './UserSettings.vue';
</script>

<template>
	<div class="dashboard">
		<UserProfile />
		<UserSettings />
	</div>
</template>
```

## Analogía
El `Global Registration` es como tener un martillo colgado en la plaza pública del pueblo: cualquiera puede usarlo en cualquier momento sin pedirlo, pero si el martillo se rompe o se cambia, todo el pueblo podría verse afectado y siempre ocupa un espacio físico aunque nadie lo use. El `Local Registration` es como tener el martillo dentro de tu propia caja de herramientas: solo tú lo usas cuando lo necesitas, y si decides no llevarlo contigo, no ocupa espacio innecesario en tu mochila (`bundle`).

## Buenas Prácticas
- **`Avoid Global Pollution`**: No registres componentes globalmente a menos que sean verdaderamente ubicuos (ej. componente de una UI Library como `BaseButton` o `BaseInput`).
- **`PascalCase in Templates`**: Aunque el HTML no es `case-sensitive`, `Vue` compila los `templates` y el uso de `PascalCase` ayuda a diferenciar componentes de `Vue` de elementos HTML nativos.
- **`Explicit Dependency Graph`**: Prefiere el registro local para que herramientas como `VS Code` puedan realizar búsquedas de referencias (*`Find All References`*) y `Refactors` de forma precisa.

## Escenarios y Aplicaciones
- **Global**: Ideal para componentes de librerías externas (como un set de iconos) que usaras en el 90% de tus archivos.
- **Local**: El estándar para componentes de negocio en proyectos. Si un componente pertenece a un módulo de "Configuración", solo debe vivir y ser registrado dentro de ese módulo.
- **`Identifying Anti-patterns`**: Si notas que tu `main.ts` tiene una lista gigante de `app.component()`, estás ante un problema de arquitectura que afectará el tiempo de carga inicial de la aplicación.

## Notas Personales
- Al registrar globalmente, podrías usar colisiones de nombres. Si dos librerías registran un componente llamado `Button`, el último en registrarse sobrescribirá al anterior sin previo aviso claro.
- En mis proyectos, trato el registro global como un `Singleton`. Si no es una utilidad transversal absoluta, se queda como registro local. Esto facilita enormemente la migración de componentes entre `micro-frontends` o módulos independientes.
