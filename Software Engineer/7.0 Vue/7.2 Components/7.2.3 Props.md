## Definición
Las `Props` son atributos personalizados que registramos en un componente para permitir que el componente padre le pase datos. Representan el canal de comunicación oficial "hacia abajo" (*`Top-Down`*) y son la base para crear componentes reutilizables y configurables. En `Vue 3`, se definen mediante la macro `defineProps`.

## Conceptos Clave
- **`One-Way Data Flow`**: Las `props` fluyen de forma unidireccional (de padre a hijo). El hijo **nunca** debe modificar una `prop` directamente.
- **`Prop Validation`**: Mecanismo para asegurar que el componente reciba el tipo de dato correcto (`String`, `Number`, `Boolean`, etc.), mejorando la robustez.
- **`Static vs. Dynamic Props`**: Las `props` estáticas se pasan como `string` simples, mientras que las dinámicas usan `v-bind` (o  `:`) para pasar variables o expresiones.
- **`Boolean Casting`**: Comportamiento especial donde la presencia de un atributo sin valor se interpreta como `true`.
- **`Props Destructuring`**: En versiones recientes de `Vue 3`, se puede desestructurar `defineProps` manteniendo la reactividad bajo ciertas condiciones, pero tradicionalmente requiere precaución para no perder el vínculo reactivo.

## Ejemplos Prácticos
##### Declaración con `TypeScript` (Recomendado)
```html
<script setup lang="ts">
// Defining props using Type-based declaration
interface UserProps {
	id: number;
	username: string;
	role?: 'admin' | 'editor' | 'viewer'; // Optional prop
	isActive: boolean;
}

// Macro available in <script setup> without import
const props = defineProps<UserProps>();

// Accessing props in logic
console.log(`Loading user: ${props.username}`);
</script>

<template>
	<div :class="{ 'active-user': isActive }">
		<h3>{{ username }}</h3>
		<p>Role: {{ role || 'Guest' }}</p>
	</div>
</template>
```
##### Uso en el componente Padre
```html
<UserCard username="John Doe" />

<UserCard
	:id="userData.id"
	:username="userData.name"
	:is-active="true"
/>
```

## Analogía
Imagina que un componente es una maquina expendedora. Las `props` son las ranuras de entrada: una para monedas, otra para el código del producto. La máquina no puede cambiar el valor de la moneda que insertaste; solo puede leerla y reaccionar ante ella. Si quieres tu dinero de vuelta, la máquina emite una señal (evento), pero no altera la moneda en tu bolsillo directamente.

## Buenas Prácticas 
- **`Kebab-case en Templates`**: Aunque en `JS` uses `camelCase` (`userRole`), en el `HTML` del padre usa kebab-case (`:user-role="role"`). Es el estándar de legibilidad.
- **`Validation is Safety`**: Siempre define tipos y valores por defecto (`withDefaults`). Esto actúa como documentación viva para otros desarrolladores.
- **`Props are Read Only`**: Si necesitas modificar el valor de una `prop`, crea una propiedad computada (`computed`) basada en ella o utiliza un estado local inicializado con la `prop`.
- **`Avoid Prop Drilling`**: Si pasas la misma `prop` a través de 5 niveles de componentes, considera usar `Pinia` o `Provide/Inject` para mantener la arquitectura limpia.

## Escenarios y Aplicaciones
- **`UI Components`**: Un componente `AppButton` que recibe `props` como `color`, `size` o `isDisabled`.
- **`Data Visualization`**: Componentes de gráficas que reciben un `array` de objetos `data` para renderizar.
- **`List Rendering`**: Pasar el objeto del `item` actual a un componente `ListItem` dentro de un `v-for`.

## Notas Personales
- Al desestructurar `const {title} = defineProps()`, podrías perder la reactividad si no usas una versión de `Vue` que soporte *`Reactive Props Destructure`* (habilitado por defecto en 3.4+). Si estás en versiones anteriores, usa `toRefs`.
- Pasar objetos grandes como `props` es eficiente porque se pasan por referencia, pero asegúrate de que el componente hijo solo use lo que necesita para evitar re-renders innecesarios.
- Usa interfaces `TS` compartidas para las `props`. Así, si el modelo de datos cambia, el compilador te avisará en todos los componentes que lo consumen.
