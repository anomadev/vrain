## Definición
Los `Events` son el mecanismo oficial de comunicación "hacia arriba" (*`Bottom-Up`*) en `Vue`. Permiten que un componente hijo notifique a su padre sobre interacciones o cambios de estado internos sin acoplarse directamente a la lógica del padre. En `Vue 3`, se gestionan mediante la macro `defineEmits`.

## Conceptos Clave
- **`defineEmits`**: Macro utilizada en `<script setup>` para declarar los eventos que el componente puede disparar.
- **`Event Bubbling (No existe)`**: A diferencia de los eventos del `DOM` nativos, los eventos de componentes de `Vue` no escalan (`bubble`) automáticamente. Si un abuelo necesita escuchar a un nieto, el padre debe re-emitir el evento o usar un `bus/store`.
- **`Validation`**: Al igual que las `props` ([[7.2.3 Props]]), los eventos pueden ser validados para asegurar que los datos emitidos (`payload`) cumplan con ciertos requisitos.
- **`Kebab-case vs. camelCase`**: Se recomienda emitir en `camelCase` en el código (`updateValue`), pero escucharlos en `kebab-case` en el `template (@update-value)`.

## Ejemplos prácticos
##### Declaración y Emisión con `TypeScript`
```html
<script setup lang="ts">
// 1. Define events with their expected payload types
const emit = defineEmits<{
	(e: 'close'): void;
	(e: 'submit', payload: { id: number, data: string }): void;
}>();

const handleAction = () => {
	// 2. Trigger the event
	emit('submit', { id: 1, data: 'New Item' });
};
</script>

<template>
	<div class="modal">
		<button @click="$emit('close')">Cancel</button>
		<button @click="handleAction">Save Changes</button>
	</div>
</template>
```
##### Escuchando en el Padre
```html
<template>
	<CustomModal
		@close="isModalOpen = false"
		@submit="handleFormSubmission"
	/>
</template>

<script setup lang="ts">
const handleFormSubmission = (payload: { id: number; data: string }) => {
	console.log('Received form child:', payload);
}
</script>
```

## Analogía
Imagina que el componente hijo es un **empleado** y el padre es el **gerente**. El empleado no puede entrar a la oficina del gerente y cambiar los archivos directamente (No mutar `props`). En su lugar, el empleado envía un **correo electrónico** (`Event`) diciendo: "*Oye, he terminado el reporte*". El gerente recibe el correo y él decide qué hacer con la información.

## Buenas Prácticas
- **Documentación mediante Tipos**: Usa `TypeScript` para definir exactamente qué estructura tiene el `payload` del evento. Esto evita que el padre intente leer propiedades inexistentes.
- **Evita el `Event Drilling`**: Si un evento debe pensar por muchos niveles de componentes intermedios que no lo usan, es un `code smell`. Usa `Pinia` o una arquitectura basada en servicios.
- **`Namespacing`**: Para componentes complejos, usa nombres de eventos claros como `user:update` o `modal:confirm` para evitar ambigüedades.
- **`Validation Logic`**: Si el `payload` es crítico, usa la sintaxis de objeto en `defineEmits` para retornar `true` o `false` y validar los datos antes de que lleguen al padre.

## Escenarios y Aplicaciones
- **Formularios**: Un componente de `input` personalizado que emite `update:modelValue` cada vez que el usuario escribe.
- **Modales y Overlays**: Notificar al padre que el usuario hizo clic fuera o presionó "Cerrar".
- **Listas Dinámicas**: Un `item` de una lista que emite su `ID` para ser eliminado de un `array` que reside en el estado del padre

## Notas Personales
- Si accidentalmente nombras un evento igual que un evento nativo (ej. `click`), `Vue` escuchará ambos. Usa nombres específicos para evitar comportamientos extraños.
- Si un componente está emitiendo demasiados eventos (ej. 10+), probablemente ese componente está haciendo demasiado y debería dividirse.
- `v-model` es azúcar sintáctica para una `prop` (`modelValue`) y un evento (`update:modelValue`). Es la forma más limpia de lograr comunicación bidireccional controlada.
