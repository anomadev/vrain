## Definición
`v-model` es una directiva de azúcar sintáctica que simplifica la comunicación bidireccional entre un componente padre y un hijo. En lugar de pasar manualmente una `prop` ([[7.2.3 Props]]) y escuchar un evento de actualización, `v-model` crea una sincronización automática. Desde `Vue 3.4+`, la forma estándar y recomendada de implementarlo es mediante la macro `defineModel()`,  que abstrae la complejidad de los `refs` y los `emits`.

## Conceptos Clave
- `Syntactic Sugar`: Es la combinación de una `prop` (entrada) y un `emit`  (salida).
- `defineModel()`: La macro moderna que simplifica la declaración de modelos reactivos en el componente hijo.
- `Named v-model`: Capacidad de tener múltiples `v-model` en un mismo componente (ej. `:v-model:first-name` y `:v-model:last-name`).
- `Modifiers`: Funcionalidad para transformar el valor antes de sincronizarlo (ej. `trim`, `.number` o modificadores personalizados como `.uppercase`).
- `One Way Data Flow` (Interno): Aunque parece bidireccional, por debajo sigue respetando que el padre es el dueño del estado; el hijo simplemente solicita el cambio.

## Ejemplos Prácticos
##### Implementación Moderna (`CustomInput.vue - Vue 3.4+`)
```html
<script setup lang="ts">
// Modern way: defineModel handles the prop and the emit automatically
const model = defineModel<string>({ default: '' });

// Optional: defineModel with custom name and modifiers
const [firstName, modifiers] = defineModel<string>('firstName', {
	set(value) {
		if (modifiers.uppercase) {
			return value.toUpperCase();
		}
		return value;
	}
});
</script>

<template>
	<input v-model="model" type="text" placeholder="Default model" />
	<input v-model="firstName" type="text" placeholder="Named model with uppercase" />
</template>
```
##### Uso en el Padre
```html
<script setup lang="ts">
import { ref } from 'vue';
import CustomImput from './CustomInput.vue';

const searchText = ref('');
const userName = ref('gemini');
</script>

<template>
	<CustomInput v-model="searchText" />
	<CustomInput v-model:firstName.uppercase="userName" />
	
	<p>Search: {{ searchText }}</p>
	<p>User: {{ userName }}</p>
</template>
```

## Analogía
Imagina un **Walkie-Talkie**. Cuando tú hablas (Padre), el otro escucha (`Prop`). Cuando el otro responde (Evento), tú escuchas. `v-model` es como tener un manos libres inteligente: no tienes que presionar el botón de "hablar" y "soltar" manualmente cada vez; la comunicación fluye de manera natural en ambos sentidos tan pronto como hay actividad, manteniendo a ambos dispositivos sincronizados.

## Buenas Prácticas
- **`Prefer defineModel`**: Si estás en `Vue 3.4+`, deja de usar la combinación manual de `props` y `emits` para modelos. `defineModel` es más limpio y reduce el código repetitivo (*`boilerplate`*).
- **`Use Named Models for Clarity`**: Si tu componente tiene más de una entrada de datos principal (ej. un formulario de dirección), usa `v-model:street`, `v-model:city`, etc., en lugar de un solo objeto gigante.
- **`Validation`**: Utiliza las opciones de `defineModel` para incluir validaciones o transformaciones (usando `get` y `set`) para asegurar que el componente padre reciba datos limpios.

## Escenarios y Aplicaciones
- **`Form Wrappers`**: Componentes que envuelven input de librerías de tercero (como un `DatePicker`) para integrarlos con el estado de la app.
- **`Complex UI State`**: Un componente de "Filtros de búsqueda" que necesita sincronizar varios campos (precio, categoría, fecha) con la URL o el estado global.
- **`Custom Toggles/Checkboxes`**: Crear interruptores de UI personalizados que se comporten como `inputs` nativos.

## Notas Personales 
- Si trabajas en proyectos antiguos de `Vue 2`, recuerda que el evento era `input` y la `prop value`. En `Vue 3` cambió a `update: modelValue` y `modelValue`.
- Si ves componentes con `props` llamadas `data` y eventos `@update-data`, refactorízalos a `v-model`. Es más idiomático y facilita el uso al consumidor del componente.
- `defineModel` devuelve un `Ref`, lo que significa que puedes usarlo directamente con `v-model` en elementos nativos dentro de tu componente hijo, manteniendo la reactividad en cadena.
