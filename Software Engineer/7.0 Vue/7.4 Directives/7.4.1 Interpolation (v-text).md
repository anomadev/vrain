## Definición
La `Interpolation` (comúnmente llamada "`Mustache Syntax {{}}`") y la directiva `v-text` son los mecanismos de `Vue` para renderizar datos como texto plano dentro del `DOM`. Ambas técnicas aseguran que el contenido sea tratado como texto y no como `HTML`, protegiendo la aplicación contra ataques de `Cross-Site Scripting (XSS)` mediante el escape automático de caracteres.

## Conceptos Clave
- `Mustache Syntax {{ }}`: Permite realizar una interpolación parcial. Puedes mezclar texto estático con variables dinámicas en un mismo nodo de texto.
- `v-text Directive`: Reemplaza todo el contenido de texto del elemento con valor de la expresión vinculada. Es el equivalente funcional de establecer la propiedad `textContent` nativa.
- `XSS Protection`: Ambos métodos escapan el contenido, convirtiendo caracteres como `<` y `>` en sus entidades `HTML` equivalentes.
- `FOUC (Flash of Uncompiled Content)`: Problema visual donde el usuario ve las llaves `{{...}}}` por un instante antes de que `Vue` tome el control.

## Ejemplos Prácticos
##### `Mustache vs v-text Comparison`
```html
<script setup lang="ts">
const username = ref<string>('G');
const rawContent = ref<string>('This is <b>bold</b>');
</script>

<template>
  <p>Welcome back, {{ username }}!</p> 
  <p v-text="username">This text will be overwritten</p>
  <div>{{ rawContent }}</div> 
  <div v-text="rawContent"></div>
</template>
```

## Analogía
Imagina que tiene una pizarra en una oficina. Usar `{{ }}` es como escribir un mensaje con tiza dejando un espacio en blanco para rellenar el nombre del empleado del mes: "Felicidades a Nombre". Usar `v-text` es como si llegaras con un **letrero digital** que tapa toda la pizarra por completo; no importa qué había escrito antes, el letrero ahora es lo único que se ve.

## Buenas Prácticas
- **`Prefer {{ }} for Readability`**: En el 99% de los casos, las "`mustaches`" son preferibles porque son más naturales de leer dentro de la estructura `HTML`.
- **`v-clock for FOUC`**: Si te preocupa que el usuario vea las llaves `{{ }}` antes de cargar la app (especialmente en entornos sin pre-renderizado como `SRR`), usa la directiva `v-cloak` con un estilo `CSS [v-cloak] { display: none; }` en lugar de cambiar todo a `v-text`.
- **`Avoid unnecessary v-text`**: No uses `v-text` solo por "estilo"; añade un atributo extra al tag `HTML` que lo hace más denso visualmente sin ganar beneficios reales de rendimiento.

## Escenarios y Aplicaciones
- **`Dynamic Labels`**: Mostrar nombres de usuario, contadores o estados de carga.
- **`SSR/SSG`**: En aplicaciones modernas (`Nuxt`/`VitePress`), las `mustaches` son procesadas en el servidor, eliminando casi por completo la necesidad de preocuparse por `v-text`.
- **`Performance`**: `v-text` es ligeramente más rápido en el primer renderizado ya que no tiene que "pasear" el contenido del nodo de texto para buscar llaves, pero la diferencia es insignificante en apps modernas.

## Notas Personales
- Muchos desarrolladores junior confunden `v-text` con una mejora de rendimiento. La realidad es que en una arquitectura `Clean Code`, la legibilidad gana. Solo usaría `v-text` si estuviera construyendo una herramienta que genere `templates` dinámicos donde las llaves `{{ }}` pudieran causar conflictos de `parsing`.
- Si usas `v-text` en un elemento que ya tiene hijos, esos hijos desaparecerán. Es un error común intentar anidar elementos dentro de un tag con `v-text`.
